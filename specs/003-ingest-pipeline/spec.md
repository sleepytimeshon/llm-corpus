# Feature Specification: Inbox Watcher + Ingest Pipeline (Validation, Normalization, Content-Hash Idempotency)

**Feature Branch**: `003-ingest-pipeline`
**Created**: 2026-05-12
**Status**: Draft
**Input**: SP-003 from `.product/SPRINT-PLAN.yaml` / `.product/ROADMAP.yaml` RM-004 ("Inbox watcher + ingest normalization + content-hash idempotency"). Source FR text: `.product/REQUIREMENTS.yaml` FR-010 (filesystem inbox watcher + validation gate), FR-011 (Markdown-with-YAML-frontmatter normalization), FR-017 (content-hash idempotency), NFR-016 (≥6 named telemetry event classes). Source acceptance scenarios: `.product/ACCEPTANCE-CRITERIA.feature`. Governing ADRs: `ADR-002` (full-file SHA-256 for FR-017) and `ADR-007` (FR-010 v1 MIME allowlist — PDF, Markdown, plain text, HTML). Constitutional context: Principles III (Substrate, Not Surface), V (Schema-Enforced Structured Output), VII (Cancellable IO), VIII (Atomic Writes), IX (Concurrency-Safe Shared State), X (Idempotent Pipeline Transitions; Three-Folder Routing), XI (Library/CLI Boundary), XII (Subprocess Hygiene), XIII (Telemetry-or-Die), XIV (XDG Paths via single resolver), XVI (Validation Honesty).

This is feature 003: the **producer side** of the corpus. SP-001 made the corpus *reachable* over MCP. SP-002 made the corpus *legible* (manifest/taxonomy/recent/per-document resources) against an empty index. SP-003 makes the corpus *populated* — a user dropping a file into the configured inbox results in a `documents` row, a normalized Markdown body on disk, and ≥6 named telemetry event classes recording what happened, without further user action and without any LLM in the loop. The consumer (agent) reads the produced state via SP-002's auto-loaded resources and (once SP-005 ships ranking) via `corpus.find`. The principal user of the corpus is the agent (FR-005, UR-003, WHITEPAPER §line 159, ARCHITECTURE §5.3-5.4, ADR-011); SP-003 is the substrate that makes that principal-relationship populate-able.

SP-003 deliberately excludes the LLM classifier (SP-004), embedding/ranking (SP-005), retrieval prompt templates (RM-003), and idempotency/resumability/failure-lane infrastructure beyond what is structurally required by the three-folder routing primitive (SP-006 for SIGKILL-recovery, resumable checkpoints, replayability, and `corpus://failures`). Documents that SP-003 produces enter the `documents` table with raw/sentinel classification fields that SP-004 will overwrite once classification lands; until then, the rows are present and dereferenceable via `corpus://docs/{id}` (FR-008, shipped in SP-002) but `corpus.find` continues to return an empty SearchHit list (its ranking lives in SP-005).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — User drops a file into the inbox and it becomes a queryable document without further action (Priority: P1)

The user (Maya/Shon persona) drops a file (PDF, Markdown, plain text, or HTML) into the configured inbox directory using ordinary filesystem operations (`cp`, `mv`, file-manager drag, `scp`). The system MUST detect the file's arrival, validate it against the v1 ADR-007 MIME allowlist, compute its full-file SHA-256 content hash, normalize the body to Markdown with structured YAML frontmatter, write the normalized body to the canonical store under `Paths.data()`, and insert a `documents` row with a stable `doc-[0-9a-f]{8}` id and the FR-008 minimum frontmatter fields (`id`, `source_path`, `ingest_timestamp`, `mime_type`, `hash`). All of this MUST happen without any additional user action.

**Why this priority**: This is UR-001 — the load-bearing stakeholder requirement for SP-003 and the entire ingest sprint. Without it, the corpus is a read surface over a perpetually empty index. P1 because every other ingest behavior (idempotency, validation rejection, mime mismatch, failure routing) is a refinement on or a guard around this primary happy path. The agent (principal user) cannot consult the corpus on the user's documents until this happens at least once.

**Independent Test**: Initialize a clean corpus on the user's primary machine. Drop a 100-page PDF, a 50-KB Markdown note, a 5-KB plain-text file, and a single-page HTML article into the inbox directory by user-initiated `cp` operations. Without issuing any further command, observe that within a bounded wall-clock budget (NFR-014's 90-second first-run envelope is a related but distinct budget; SP-003's own per-ingest budget is the SP-003-Plan-resolved value, see Edge Cases) each file produces exactly one `documents` row with `status='success'`, a normalized Markdown body file on disk under the canonical store path, and a successful read against `corpus://docs/{id}` returning the contracted frontmatter + body. Verified end-to-end on the SP-001 + SP-002 baseline.

**Acceptance Scenarios**:

1. **Given** a clean corpus and an empty inbox, **When** the user drops one PDF, one Markdown, one plain-text, and one HTML file into the inbox, **Then** within the per-document ingest budget each file produces exactly one `documents` row with `status='success'` AND exactly one normalized Markdown body file written under the canonical store AND the body file's content matches the row's `body_path`.
2. **Given** a single accepted file in the inbox, **When** ingest completes, **Then** the row's `id` matches `doc-[0-9a-f]{8}` AND the row's `hash` equals the full-file SHA-256 of the dropped file's original bytes AND the row's `mime_type` is one of the four allowlisted types AND the row's `source_path` records the original inbox path AND the row's `ingest_timestamp` is a valid ISO-8601 UTC string.
3. **Given** a freshly produced ingest, **When** an MCP client reads `corpus://docs/{id}` for the produced id, **Then** the response is the normalized Markdown body AND the YAML frontmatter contains `id`, `source_path`, `ingest_timestamp`, `mime_type`, `hash` (FR-008 minimum surface) AND the dereferenced content matches what SP-003 wrote.
4. **Given** an accepted file in the inbox, **When** ingest is in progress, **Then** the file moves through the three-folder model — present in `pending/` between validation and normalization, then moved to `processed/` on success — AND no file remains in `pending/` after the drain step completes (Constitution Principle X).

### User Story 2 — The same file dropped twice does not produce duplicate documents (Priority: P1)

The user drops a file into the inbox; ingest succeeds. Some time later, the user re-drops the same file (identical bytes, possibly under a different filename, possibly into a different inbox subdirectory). The system MUST detect that this content has already been ingested (via the FR-017 / ADR-002 full-file SHA-256 content hash) and short-circuit to a no-op rather than producing a second `documents` row. The dedup decision MUST be reported via telemetry; the user MUST be able to observe (via CLI / agent) that the second drop was deduplicated rather than silently dropped.

**Why this priority**: P1 because content-hash idempotency is the load-bearing correctness property of the ingest pipeline. Without it, the user's natural workflow (drop a paper, rename a folder, re-sync from backup, re-import from a USB drive) silently doubles the corpus and pollutes retrieval. FR-017 is `must`. ADR-002 binds the algorithm choice to full-file SHA-256 specifically because partial-prefix hashes would false-dedup at the first 4 KB, and ADR-002's compliance scenarios call out the 60-MB-identical-prefix-different-tail adversary case explicitly.

**Independent Test**: Drop a file into the inbox, wait for ingest to complete, verify one `documents` row exists. Drop the *same bytes* under a *different filename* (`cp file.pdf same-content-different-name.pdf`) into the inbox. Verify that the second drop produces *no* new `documents` row, that telemetry records a `ingest.dedup_hit` event tagged with the existing document's id, and that the second filename is either removed from `pending/` (no-op routing) or moved into `processed/` adjacent to the original (Plan-resolved routing detail). Then construct two 60-MB files identical in the first 1 MB but differing in the last byte; drop both. Verify both ingest as separate documents (ADR-002's F-10 adversary scenario).

**Acceptance Scenarios**:

1. **Given** a file has been successfully ingested into the corpus, **When** the same file (identical bytes) is dropped into the inbox a second time, **Then** no new `documents` row is created AND a `ingest.dedup_hit` telemetry event is emitted referencing the existing document's `id` AND the duplicate file is removed from `pending/` (no orphaned pending entry).
2. **Given** an ingested file at byte-identical content, **When** that content is re-dropped under a different filename (e.g., `notes.md` then `notes-copy.md`), **Then** the dedup decision fires on content hash, not filename AND a single `documents` row remains AND the original `source_path` is preserved.
3. **Given** a 60-MB file `A.pdf` already ingested, **When** a second 60-MB file `B.pdf` identical to `A.pdf` in the first 1 MB but differing in the last byte is dropped, **Then** both files produce separate `documents` rows with different hashes AND neither is treated as a duplicate of the other (ADR-002 F-10 adversary).
4. **Given** the same content is dropped while a separate ingest is in-flight on a different file, **When** the dedup-hit decision fires, **Then** the dedup decision does NOT corrupt or block the unrelated in-flight ingest (Constitution Principle IX, concurrency-safe shared state).

### User Story 3 — A file with a disallowed MIME type is rejected at the validation gate with structured diagnostics (Priority: P1)

The user drops a file into the inbox whose detected MIME type is outside the v1 ADR-007 allowlist (e.g., `.docx`, `.zip`, a code file, an RTF, an image). The system MUST reject the file at the validation gate, route it into the three-folder routing primitive's `failed/` lane with a structured `.error.json` sidecar (`error_code: mime_not_allowlisted`, retriable flag, source pointer, stage), emit a `inbox.allowlist_miss` telemetry event, and MUST NOT enter the file into the `documents` table. The rejection MUST happen after the extension check but using MIME-sniff verification (per ADR-007 / D-011 / C-018 F-5) so that a `.md`-extension file containing PDF bytes is also rejected.

**Why this priority**: P1 because validation honesty is a Constitution Principle XVI commitment: the system MUST NOT silently accept inputs it cannot classify or normalize, and MUST NOT silently drop them either. The rejection path is the v1 contract for the failure-lane surface — once SP-006 ships `corpus://failures`, the user inspects and acts on these entries. SP-003 ships the structural rejection (file moves to `failed/`, sidecar written, telemetry emitted); the inspection surface and replay arrive with SP-006.

**Independent Test**: Drop a `.docx` file, a `.zip` file, a `.png` file, and a `.md` file whose body is binary PDF content (extension/content mismatch) into the inbox. Verify each file moves into the `failed/` lane within the per-document budget, each receives an `.error.json` sidecar with `error_code`, `message`, `retriable`, `source_path`, `stage` fields, no `documents` row is created for any of them, and a `inbox.allowlist_miss` (or, for the mismatch case, `inbox.mime_mismatch`) telemetry event is appended for each.

**Acceptance Scenarios**:

1. **Given** a file with a disallowed file extension (e.g., `.docx`, `.zip`, `.png`, `.rtf`), **When** the file is dropped into the inbox, **Then** the file is moved to the `failed/` lane AND an `.error.json` sidecar is written next to the file with `error_code='mime_not_allowlisted'`, `retriable=false`, `source_path`, and `stage='validate'` fields AND no `documents` row is created AND a `inbox.allowlist_miss` telemetry event is emitted with the file path and detected MIME type.
2. **Given** a file with an allowed extension (`.md`) but a body whose magic bytes are `%PDF` (extension/content mismatch), **When** the file is dropped, **Then** the file is moved to the `failed/` lane AND the `.error.json` sidecar records `error_code='mime_mismatch'`, `retriable=false`, with both the claimed extension and the detected MIME type AND a `inbox.mime_mismatch` telemetry event is emitted (ADR-007 / C-018 F-5).
3. **Given** a file exactly at the configured maximum allowed size, **When** the file is dropped, **Then** the file passes validation AND proceeds to normalization (boundary inclusion: ≤ max size is accepted).
4. **Given** a file one byte larger than the configured maximum allowed size, **When** the file is dropped, **Then** the file is moved to the `failed/` lane AND the `.error.json` sidecar records `error_code='size_exceeded'`, `retriable=false` AND a `inbox.size_exceeded` telemetry event is emitted AND the content read is bounded — the system MUST NOT read past the max-size cutoff before rejecting (Constitution Principle VII, bounded IO).
5. **Given** a file whose filename fails the sanity check (e.g., null bytes, path-traversal sequences, control characters, empty name), **When** the file is dropped, **Then** the file is moved to `failed/` AND the `.error.json` sidecar records `error_code='filename_sanity_failed'`, `retriable=false` AND the rejection occurs BEFORE the file's content is read (Principle VII bounded-IO discipline — do not consume bytes from a structurally invalid input).

### User Story 4 — Pipeline emits ≥6 named telemetry event classes covering the SP-003 surface (Priority: P2)

Every state transition the SP-003 pipeline performs MUST emit a structured event to `Paths.telemetry()` (Constitution Principle XIII, NFR-016). The named event classes covered by SP-003 are: `inbox.allowlist_hit`, `inbox.allowlist_miss`, `inbox.mime_mismatch`, `inbox.size_exceeded`, `ingest.dedup_hit`, `ingest.dedup_miss`, `ingest.normalized`, `ingest.completed`, and (cross-cutting) `tool_invoked` when subprocess extractors are invoked (Principle XII). The NFR-016 floor is 6 event classes for SP-003 inclusive of related SP-002 read events; SP-003 alone ships ≥6 distinct ingest-side classes. Each event MUST validate against the contracted Zod schema and MUST be observably appended on both success and failure paths uniformly.

**Why this priority**: P2 because telemetry coverage is a substrate-quality requirement, not a stakeholder-visible workflow. The user does not consume telemetry directly; the operator (also Shon, in single-user posture) inspects telemetry when triaging ingest failures or measuring pipeline behavior. Without telemetry-or-die, silent failure becomes the default failure mode, which is constitutionally forbidden. P2 alongside US3 because US3 already produces three of the named classes; US4 makes the coverage and schema-uniformity contract explicit.

**Independent Test**: Run a fixture-driven mixed-workload script that drops 10 valid files, 5 disallowed-MIME files, 5 mime-mismatch files, 3 oversize files, 2 filename-invalid files, and 5 duplicates of already-ingested content. Inspect `Paths.telemetry()`. Verify at least 6 distinct event class names appear AND every event validates against the Zod schema AND every recorded outcome (success and each failure class) appears at least once AND no event is emitted with a missing required field. Then run the test harness with the JSONL writer's filesystem mounted read-only mid-run; verify the pipeline observes the telemetry write failure and routes the in-flight document to `failed/` with `error_code='telemetry_write_failed'` rather than silently completing the ingest (Principle XIII: no swallowed exceptions; Principle XVI: honest failure).

**Acceptance Scenarios**:

1. **Given** a mixed-workload run of 25 files spanning valid/disallowed/mismatched/oversize/invalid-name/duplicate cases, **When** the run completes, **Then** ≥6 distinct telemetry event class names appear in `Paths.telemetry()` AND each event includes at minimum `event_class`, `timestamp`, `severity`, `outcome`, and a path/id reference appropriate to the event class AND every event validates against the canonical Zod telemetry schema (Principle V).
2. **Given** the canonical SP-003 telemetry event classes (`inbox.allowlist_hit`, `inbox.allowlist_miss`, `inbox.mime_mismatch`, `inbox.size_exceeded`, `ingest.dedup_hit`, `ingest.dedup_miss`, `ingest.normalized`, `ingest.completed`), **When** the corresponding pipeline transition fires, **Then** the matching event class is emitted exactly once per transition per document (Principle XIII: one event per state transition).
3. **Given** a subprocess extractor (PDF or HTML normalizer) is invoked via `runTool`, **When** the subprocess completes (success or failure), **Then** a `tool_invoked` telemetry event is emitted with the binary name and exit code per Principle XII — the args themselves MUST NOT appear in the event payload.
4. **Given** the telemetry JSONL append fails mid-ingest (filesystem read-only, disk full, ENOSPC), **When** the pipeline observes the failure, **Then** the in-flight document is routed to `failed/` with `error_code='telemetry_write_failed'`, `retriable=true` AND the system does NOT silently complete the ingest AND the exception is observable to the caller (Principle XIII: no silent swallow; Principle XVI: honest failure).
5. **Given** any caught exception inside an SP-003 pipeline stage, **When** the catch block executes, **Then** a structured telemetry event is appended BEFORE the exception is re-thrown or converted into a `Result<T, E>` failure return AND the event severity matches the actual error severity (no downgrading to `info` to silence alerts — Principle XIII).

### Edge Cases

- **Watcher misses a file due to fast write**: A user atomically renames a 1-byte file into the inbox; the OS-level watcher MAY deliver the create event before the rename settles, before the file's full content is on disk, or with the wrong size. The system MUST stabilize on file content before reading (e.g., stat-and-recheck or use rename-into-inbox atomically as the only sanctioned drop pattern documented in the install README). The decision between watcher-driven and poll-driven detection is a Plan-stage resolution (`/speckit-plan` ADR scope); SP-003 commits to the behavioral guarantee — no file dropped via `mv tmpfile inbox/` or atomic rename MUST be missed.
- **File modified during hash**: A user begins writing into the inbox using a streaming editor (vim, `echo … >> inbox/file.md`). The hash computation MUST be performed on a stable snapshot of the file's bytes. The implementation pattern (open + read full file with a single stream, OR move file to `pending/` before hashing so the source path is stable) is a Plan-stage decision; SP-003 commits to: the hash recorded in `documents.hash` MUST be the SHA-256 of the bytes that were normalized into `body_path`. A file whose content changes during hash computation MUST either ingest as the snapshot bytes (deterministic) or be routed to `failed/` with `error_code='file_unstable'`, retriable.
- **Torn write during atomic move from `pending/` to `processed/`**: Constitution Principle VIII requires `tmp + fsync + rename + dirsync` for every disk write, with PID-and-entropy temp suffixes. A SIGTERM/SIGKILL between the body-file write and the `documents` row insertion is SP-006 territory (resumable checkpoints + idempotency for kill-9 cases); SP-003 ships the atomic-write primitive but defers the cross-stage SIGKILL-recovery scenario to SP-006. Within SP-003, the requirement is: a single-stage write MUST either fully succeed or fully fail; partial writes MUST NOT leave a `documents` row pointing at a non-existent body file.
- **Hash collision on full-file SHA-256**: Cryptographically negligible per ADR-002, but if two distinct files produce the same SHA-256 hash the system MUST treat the second as a `ingest.dedup_hit` against the first (the algorithm's correctness depends on the hash function's collision resistance). The system MUST NOT attempt a content-equality fallback for ADR-002's stated reason (it would defeat the streaming-hash performance contract).
- **mtime changes during hash**: Stable hash computation MUST NOT depend on file mtime; only the file's byte content matters. mtime is recorded for observability but is not part of the hash input or the dedup decision.
- **Per-document ingest wall-clock budget**: NFR-014 mandates ≤ 90 seconds from `npx <pkg> init` through the first successful `corpus.find` query end-to-end. NFR-009 mandates classifier wall-clock per-document budgets (a SP-004 concern). SP-003's own per-document ingest budget — the wall-clock from inbox detection through `documents` row insertion, excluding classification and embedding — is a Plan-stage resolution informed by the ADR-002 ~250 ms-on-60-MB hash measurement and the extractor subprocess invocation costs. SP-003 commits to: the SP-003 portion of the pipeline must not consume more than its allocated share of the NFR-014 90-second envelope.
- **Empty file**: A 0-byte file in the inbox passes filename sanity, passes extension check, has a detectable MIME (`text/plain` or `application/x-empty` per the file-type implementation), and MUST normalize to an empty Markdown body. The `documents` row is written with `hash` = SHA-256 of empty bytes (a known constant); the body file exists but is zero bytes. Duplicate empty files dedup against each other normally.
- **Filename collision in `processed/`**: Two files with the same basename ingested from the same or different sources MUST NOT overwrite each other in `processed/`. The atomic-rename target MUST be uniquified (e.g., via `processed/{doc_id}-{basename}` or a deterministic per-id subdirectory) — Plan-stage decision. SP-003 commits to: the rename MUST NEVER overwrite a pre-existing file in `processed/`.
- **Filesystem watcher resource limits**: On Linux, `inotify` has per-user watch limits (`fs.inotify.max_user_watches`). Hitting the limit MUST surface as a structured `inbox.watcher_resource_exhausted` telemetry event (an additional event class beyond the NFR-016 floor) AND the ingest daemon MUST exit with a non-zero status rather than silently degrading to a partial-coverage watch (Principle XIII + Principle XVI).
- **Inbox subdirectory traversal**: SP-003's v1 watcher scope is the top-level inbox directory only; subdirectories under the inbox are NOT watched and files in them MUST NOT be ingested. The user's documented drop pattern is `cp file.pdf <Paths.inbox()>/` (or atomic rename into that path). Recursive watching is out of scope for SP-003 and is not deferred to any specific SP — it is a future enhancement.
- **Drop-during-init**: Files dropped between `corpus init` and the first watcher activation MUST be picked up by an initial-scan pass on watcher startup — no file MUST be silently skipped because the watcher was not yet running.
- **Three-folder routing invariants (cross-stage)**: At the end of any drain run, `pending/` MUST be empty (Constitution Principle X). Files in `processed/` MUST have a corresponding `documents` row with `status='success'`. Files in `failed/` MUST have an `.error.json` sidecar AND MUST NOT have a `documents` row (or, if a row exists from a prior partial state, the row's `status` MUST NOT be `'success'`). Reconciliation between filesystem state and `documents` rows is a structural invariant SP-003 maintains and SP-006 strengthens with kill-9-recovery.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-INGEST-001 — Inbox watcher detects new files in the configured inbox directory.** The system MUST run a filesystem watcher monitoring `Paths.inbox()` for newly-arrived files. The watcher MUST detect files arriving via user-initiated `cp`, `mv` (rename into inbox), or `scp` operations. The watcher MUST run in the same local process tree as the MCP server (FR-010). The watcher MUST also perform an initial-scan pass on daemon startup so files dropped between `corpus init` and watcher activation are not silently skipped. Subdirectory recursion is out of scope for v1. *(Sources: FR-010; Principle XIV — Paths resolver; Principle IX — single-writer drain.)*

- **FR-INGEST-002 — Validation gate runs in fixed order: filename sanity → extension check → MIME sniff → size limit.** Before any content read, the filename MUST be sanity-checked (no null bytes, no path-traversal sequences, no control characters, non-empty). Then the file extension MUST be checked against the ADR-007 allowlist (`.pdf`, `.md`, `.txt`, `.html`). Then the file's MIME type MUST be sniffed from its magic bytes; the sniffed MIME MUST be one of the four allowlisted types AND MUST match the extension family (per ADR-007 / C-018 F-5: `.md` extension with `%PDF` magic bytes is rejected as `mime_mismatch`). Then the file's size MUST be ≤ the configured maximum (Plan-stage default; user-configurable via `corpus config`). Rejection at any gate produces an `.error.json` sidecar and routes the file to `failed/` per FR-INGEST-007. *(Sources: FR-010; ADR-007; D-011; C-018 F-5; Principle VII — bounded IO.)*

- **FR-INGEST-003 — Validated files pass through the three-folder routing primitive.** Files that pass validation MUST be atomically moved into `Paths.pending()` before content extraction begins. On successful ingest, files MUST be atomically moved from `pending/` to `Paths.processed()`. On any pipeline failure, files MUST be atomically moved from `pending/` to `Paths.failed()` with a sibling `.error.json` sidecar. At the end of any drain run, `pending/` MUST be empty (Constitution Principle X). Atomic moves MUST use `tmp + fsync + rename + dirsync` semantics for both the body-file write and the move (Principle VIII). The rename target MUST NEVER overwrite a pre-existing file in `processed/` — filename collisions MUST be uniquified deterministically (Plan-stage detail). *(Sources: FR-010; FR-011; Constitution Principles VIII, X.)*

- **FR-INGEST-004 — The pipeline computes a full-file SHA-256 content hash on accepted files (ADR-002).** Each validated file's content MUST be hashed with streaming full-file SHA-256 via Node's built-in `crypto.createHash('sha256').update(stream)` API. The hash MUST be the bytes-as-of-snapshot SHA-256 — the implementation MUST NOT compute the hash over a partial prefix, MUST NOT use a non-cryptographic algorithm (xxHash, MurmurHash), MUST NOT depend on file mtime, and MUST NOT depend on filename. The hash value MUST be stored in `documents.hash` as a lowercase hex string. The `documents.hash` column carries a UNIQUE constraint enforced at the SQLite level so duplicate-hash insertion is structurally rejected by the database (defense-in-depth alongside FR-INGEST-005's application-level dedup). *(Sources: FR-017; ADR-002; C-020 F-10; Principle V — schema-enforced.)*

- **FR-INGEST-005 — Content-hash idempotency: same content dropped twice does not produce a second document.** Before normalization or `documents` row insertion, the pipeline MUST query `documents` for an existing row with the computed hash. If a row exists, the pipeline MUST emit a `ingest.dedup_hit` telemetry event tagged with the existing document's `id`, MUST NOT create a second `documents` row, MUST NOT write a second body file, and MUST remove the duplicate file from `pending/` (no orphaned pending entry). If no row exists, the pipeline MUST emit a `ingest.dedup_miss` event and proceed to normalization. The dedup check MUST fire on content hash alone, NOT on filename or `source_path`. Two distinct files producing the same hash (cryptographically negligible) MUST be treated as the same content per ADR-002. *(Sources: FR-017; ADR-002 F-10 adversary scenario.)*

- **FR-INGEST-006 — Pipeline normalizes accepted documents to Markdown body + structured YAML frontmatter (FR-011).** For each non-duplicate, validated file the pipeline MUST produce a normalized Markdown body and structured YAML frontmatter. PDF input MUST be extracted via a subprocess extractor invoked through the `runTool` helper (Principle XII); HTML input MUST be normalized via a deterministic HTML-to-Markdown converter (Plan-stage selection — `turndown` or equivalent); Markdown input MUST pass through unchanged; plain-text input MUST be wrapped in a minimal Markdown structure that preserves the text body verbatim. The frontmatter MUST contain at minimum the FR-008 frontmatter contract: `id`, `source_path`, `ingest_timestamp` (ISO-8601 UTC), `mime_type` (one of the four allowlisted), `hash` (lowercase hex SHA-256). The normalized body MUST be written to disk at a path under the canonical store (Plan-stage layout, anchored at `Paths.data()`); the `documents.body_path` column MUST point at that file. The body-file write MUST be atomic per Principle VIII. *(Sources: FR-011; Principles V, VIII, XII; ADR-007 normalizer selections.)*

- **FR-INGEST-007 — Failed validation or pipeline failure routes the file to `failed/` with a structured `.error.json` sidecar.** Any rejection at the validation gate or any unrecoverable pipeline error within SP-003's scope MUST move the offending file to `Paths.failed()` and write a sibling `.error.json` sidecar with fields: `error_code` (stable enum string), `message` (human-readable), `retriable` (boolean), `source_path` (the original inbox path), `stage` (one of `validate`, `normalize`, `persist`), and `timestamp`. The sidecar write MUST be atomic per Principle VIII. The `error_code` enum MUST include at minimum: `filename_sanity_failed`, `mime_not_allowlisted`, `mime_mismatch`, `size_exceeded`, `file_unstable`, `extract_failed`, `normalize_failed`, `persist_failed`, `telemetry_write_failed`. The exposure of `failed/` through `corpus://failures` is SP-006 scope; SP-003 ships the on-disk structured-data primitive. *(Sources: FR-018 partial — SP-003 ships the structural failure-lane primitive, SP-006 ships the inspection / replay / `corpus://failures` resource; Principles VIII, XVI.)*

- **FR-INGEST-008 — Each successfully normalized document is inserted into `documents` with one row per content hash.** The pipeline MUST insert exactly one `documents` row per successfully normalized document. The row MUST satisfy the schema-migration column constraints from `packages/storage/src/schema-migration.ts`: `id GLOB 'doc-[0-9a-f]{8}'` (stable id format), `status IN ('success','failed','trashed')` (SP-003 writes `'success'` on the happy path), and all NOT NULL columns populated. Classifier-owned columns (`facet_domain`, `tags_json`, `facet_type`, `source_type`) are NOT yet populated with semantic content in SP-003; they MUST be populated with documented sentinel values (Plan-stage selection — e.g., `facet_domain=''`, `tags_json='[]'`, `facet_type='unclassified'`, `source_type='inbox-filesystem'`) that satisfy NOT NULL but are observably-unclassified. SP-004 (classifier) overwrites these columns; until then `corpus://taxonomy` returns empty established lists (no rows with `state='established'` in `taxonomy_terms`) and the rows are discoverable only by id, not by domain/tag filter. The row insertion and the body-file rename into `processed/` MUST commit in a single SQLite transaction or both fail (Principle VIII — transactional index). *(Sources: FR-011; FR-008 minimum frontmatter; schema-migration.ts canonical column list; Principles V, VIII, X.)*

- **FR-INGEST-009 — Pipeline emits ≥6 named telemetry event classes (NFR-016).** The SP-003 pipeline MUST emit structured telemetry events per Principle XIII via the canonical Zod-validated telemetry contract. The minimum SP-003 event classes are: `inbox.allowlist_hit`, `inbox.allowlist_miss`, `inbox.mime_mismatch`, `inbox.size_exceeded`, `ingest.dedup_hit`, `ingest.dedup_miss`, `ingest.normalized`, `ingest.completed`. The `tool_invoked` event (Principle XII, emitted by `runTool` whenever a subprocess extractor is invoked) is required and counts toward the NFR-016 floor. Additional event classes (`inbox.watcher_resource_exhausted`, `ingest.file_unstable`, `persist.failed`, etc.) MAY be emitted; the contract is the minimum, not the maximum. Every emitted event MUST validate against the Zod schema (Principle V), MUST include `event_class`, `timestamp`, `severity`, `outcome`, and a path/id reference appropriate to the class, and MUST NOT include sensitive content (Principle I — document body content MUST NOT appear in telemetry; hashes and ids and file paths are permitted). *(Sources: NFR-016; Principles XIII, V, I, XII.)*

- **FR-INGEST-010 — Cancellable and bounded IO throughout SP-003 (Principle VII).** Every external IO call in the SP-003 pipeline (filesystem watcher, file read for hash, file read for extract, subprocess extractor invocation, body-file write, `documents` row insertion, telemetry append, atomic rename) MUST accept an `AbortSignal` and propagate it through. SIGTERM/SIGINT MUST trigger `controller.abort()`; in-flight ingests MUST mark themselves as `failed` with `error_code='aborted'` (retriable=true), persist the failed state, and exit within 2 seconds. Per-document, per-stage, and per-batch timeouts MUST be configurable (Plan-stage defaults). `Promise.race` against `setTimeout` is FORBIDDEN; `AbortController` is mandatory (Principle VII). `execSync` against a string-formed shell command is FORBIDDEN; all subprocess invocation goes through `runTool` (Principle XII). *(Sources: Principles VII, XII; NFR-005 partial — full kill-9-survival is SP-006.)*

- **FR-INGEST-011 — Drain is serialized via a single advisory lock at `Paths.drainLock()` (Principle IX).** The SP-003 drain process MUST acquire `Paths.drainLock()` with `flock(LOCK_EX | LOCK_NB)` before processing any files in `pending/`. A second invocation of the drain (whether triggered by a watcher event, by a systemd timer, by a cron job, or by manual CLI invocation) MUST observe the lock contention, emit a `lock_contention` telemetry event (additional class beyond the FR-INGEST-009 minimum), and exit cleanly with exit code 0. Watchers trigger drain; drain decides whether to run. The lock MUST be released on normal exit, on exception, and on SIGTERM (Principle VII coordination). *(Sources: Architecture §11.1 + §15.2 + §15.11; Principles IX, VII.)*

- **FR-INGEST-012 — Read-only-by-design surface; no SP-003 MCP write tool.** SP-003 MUST NOT register any MCP tool, resource, or prompt that mutates corpus state. Files arrive via the inbox filesystem operation; SP-003's role is to consume the inbox, not to expose a write surface. The `corpus` CLI binary's `corpus add <url>` (FR-020) is a should-priority next-horizon item, not SP-003 scope. The MCP server remains read-only per Constitution Principle III. *(Sources: Principle III; FR-020 deferred.)*

- **FR-INGEST-013 — Library/CLI boundary; no `process.exit` in pipeline code (Principle XI).** All SP-003 source files under `packages/pipeline/`, `packages/extract/`, `packages/storage/`, `packages/inference/` MUST NOT contain `process.exit`. Library functions MUST return `Result<T, E>` or throw typed errors. Only CLI wrappers (`packages/cli/`, `packages/transport/`) and the daemon entry point (`packages/daemon/`) MAY exit the process. *(Sources: Principle XI; Architecture §15.3.)*

- **FR-INGEST-014 — XDG-compliant paths only; no writes outside `Paths.*` (Principle XIV).** Every file SP-003 reads or writes MUST resolve through the `Paths.*` resolver: `Paths.inbox()`, `Paths.pending()`, `Paths.processed()`, `Paths.failed()`, `Paths.data()` (for the canonical body store), `Paths.telemetry()` (JSONL), `Paths.drainLock()`, `Paths.cache()` (for tmp writes via `withTempDir`). The system MUST NOT write to `/tmp/`, `os.tmpdir()`, system paths, or any location not derivable from `Paths.*`. The user MAY relocate the data root via `CORPUS_HOME`; the system MUST work transparently under that override. *(Sources: Principle XIV; Architecture §2.1 + §15.5.)*

### Key Entities *(include if feature involves data)*

- **Inbox file**: a file dropped by the user into `Paths.inbox()` via filesystem operation. Has filename, extension, bytes, sniffed MIME type, size, and mtime. Not yet a corpus document.
- **Pending file**: a validated inbox file atomically moved into `Paths.pending()` awaiting normalization. Visible to the drain process; invisible to the watcher's subsequent passes (the watcher tracks new arrivals in `Paths.inbox()`, not movement out of it).
- **Processed file**: an inbox file that has been successfully ingested, normalized, and persisted; lives in `Paths.processed()` with a deterministic uniquified filename. Corresponds 1:1 with a `documents` row whose `status='success'`.
- **Failed file**: an inbox file that failed validation or pipeline processing; lives in `Paths.failed()` with a sibling `.error.json` sidecar. Has no `documents` row (or, if a partial-state row exists, the row's `status` is not `'success'`).
- **`.error.json` sidecar**: a structured-data record adjacent to a `failed/` file capturing `error_code`, `message`, `retriable`, `source_path`, `stage`, `timestamp`. Inspectable from the filesystem; agent-inspectable once SP-006 ships `corpus://failures`.
- **Content hash**: the lowercase-hex SHA-256 of the file's full bytes (ADR-002). The idempotency key for the corpus. Stored in `documents.hash` with a UNIQUE constraint.
- **Normalized body**: the Markdown content extracted from the inbox file and written under the canonical store at `documents.body_path`. Atomic; per Principle VIII.
- **`documents` row**: one SQLite row per successfully ingested document, satisfying `id GLOB 'doc-[0-9a-f]{8}'` and `status IN ('success','failed','trashed')`. Columns owned by SP-003: `id`, `title`, `body_path`, `source_path`, `mime_type`, `hash`, `ingest_timestamp`, `status`. Columns NOT YET populated semantically by SP-003 (sentinel-only, overwritten by SP-004): `facet_domain`, `tags_json`, `facet_type`, `source_type`.
- **Telemetry event**: one structured record appended to `Paths.telemetry()` (JSONL, ≤ 4 KB per record per Principle IX). Validated against the canonical Zod telemetry schema. Includes `event_class`, `timestamp`, `severity`, `outcome`, and a class-appropriate path/id reference; MUST NOT include document body content.
- **Drain lock**: an advisory `flock(LOCK_EX | LOCK_NB)` file-lock held by the active drain process. Located at `Paths.drainLock()`. Held for the duration of one drain pass; released on exit (normal, exception, SIGTERM).

## Success Criteria *(mandatory)*

### Verification Strategy

For SP-003 verification, several criteria depend on infrastructure that SP-001 and SP-002 already shipped, and several criteria interact with infrastructure that SP-004 / SP-005 / SP-006 will land later. To prevent circular dependencies and to keep SP-003 honest about its scope:

- **End-to-end ingest of real files (US1, SC-INGEST-001 through SC-INGEST-004)**: Verified directly against the SP-001 + SP-002 baseline on the user's primary machine. This is the canonical SP-003 verification surface.
- **Content-hash idempotency including the ADR-002 F-10 adversary case (US2, SC-INGEST-005, SC-INGEST-006)**: Verified by deterministic test harness that constructs the 60-MB identical-prefix-different-tail file pair and the same-content-different-filename pair.
- **Validation rejection paths (US3, SC-INGEST-007 through SC-INGEST-011)**: Verified against deterministic test inputs covering each error class. The exposure of `failed/` via `corpus://failures` is SP-006 scope; SP-003's verification ends at the on-disk sidecar and the telemetry event.
- **Telemetry coverage including the failure-mode honesty path (US4, SC-INGEST-012, SC-INGEST-013)**: Verified by the mixed-workload test harness; the telemetry-write-failure case is verified by mounting the JSONL parent directory read-only mid-test.
- **Classifier-owned columns containing sentinel values (FR-INGEST-008)**: Verified at the SQL level — `documents` rows have `facet_domain=''`, `tags_json='[]'`, etc., as Plan-stage finalized. Once SP-004 ships, these rows MUST be updateable in place without breaking SP-003's invariants.
- **Concurrency invariants (FR-INGEST-011 lock contention, US2.4)**: Verified by launching two drain processes concurrently and asserting only one acquires the lock and processes files, while the other emits `lock_contention` and exits 0.
- **Per-document ingest budget**: SP-003 commits to a Plan-stage-resolved budget. The verification harness records p95 wall-clock per ingested file across a representative fixture set on the user's primary machine. The number itself is finalized in `/speckit-plan`; SP-003's spec-level commitment is "documented, measured, and within the SP-003 share of the NFR-014 envelope."

### Measurable Outcomes

- **SC-INGEST-001 — End-to-end happy-path ingest across all four allowlisted MIME types.** Dropping one PDF, one Markdown file, one plain-text file, and one HTML file into a clean inbox produces, without further user action, exactly four `documents` rows with `status='success'`, four normalized Markdown body files on disk under the canonical store, and four successful reads against `corpus://docs/{id}`. Verified on the user's primary machine.
- **SC-INGEST-002 — Three-folder routing invariant post-drain.** After every drain run completes, `Paths.pending()` is empty. Every file in `Paths.processed()` has a corresponding `documents` row with `status='success'`. Every file in `Paths.failed()` has a sibling `.error.json` sidecar AND has no `documents` row with `status='success'`. Verified by structural reconciliation after each test run.
- **SC-INGEST-003 — `corpus://docs/{id}` minimum frontmatter contract (FR-008).** For every `documents` row produced by SP-003, reading the SP-002 per-document resource `corpus://docs/{id}` returns a normalized Markdown body and YAML frontmatter containing at minimum `id`, `source_path`, `ingest_timestamp`, `mime_type`, `hash`. Zero ingested documents return frontmatter missing any contracted field.
- **SC-INGEST-004 — Body file atomicity.** Across a 50-ingest run with random SIGTERM delivered between watcher detect and `documents` row commit, zero `processed/` files exist without a corresponding `documents` row AND zero `documents` rows exist pointing at a missing body file. (Full kill-9-survival is SP-006; SP-003's atomicity claim is bounded to the single-stage write contract.)
- **SC-INGEST-005 — Content-hash idempotency: identical content dedups.** Dropping the same byte-identical file under three different filenames produces exactly one `documents` row AND emits one `ingest.dedup_miss` event plus two `ingest.dedup_hit` events referencing the original document's id.
- **SC-INGEST-006 — ADR-002 F-10 adversary scenario passes.** Two 60-MB files identical in the first 1 MB but differing in the last byte produce two distinct `documents` rows with distinct hashes, AND neither is dedup'd against the other.
- **SC-INGEST-007 — MIME allowlist correctness across happy + sad paths.** A test set of (a) one file per allowed extension (PDF, MD, TXT, HTML), (b) five files with disallowed extensions (`.docx`, `.zip`, `.png`, `.rtf`, `.exe`), and (c) one extension/content-mismatch file (`.md` body with `%PDF` magic) produces: 4 `documents` rows on the (a) path, 6 `failed/` entries with correct `error_code` values (`mime_not_allowlisted` x5, `mime_mismatch` x1) on (b) and (c), and zero false positives or false negatives.
- **SC-INGEST-008 — Size-limit boundary behavior.** A file exactly at the configured max size passes validation; a file one byte larger is rejected with `error_code='size_exceeded'`. The system reads at most max-size+1 bytes from the oversize file before rejecting (bounded IO).
- **SC-INGEST-009 — Filename sanity rejection.** Files with null-byte filenames, path-traversal sequences (`../../foo`), control characters, and zero-length names are rejected with `error_code='filename_sanity_failed'` BEFORE any content read; zero filename-invalid files produce a `documents` row or a body-file write.
- **SC-INGEST-010 — Structured `.error.json` sidecar contract.** Every entry in `failed/` has a sibling `.error.json` file with all required fields (`error_code`, `message`, `retriable`, `source_path`, `stage`, `timestamp`) AND the file is valid JSON AND the `error_code` value is a member of the documented enum (FR-INGEST-007). Verified across all rejection paths.
- **SC-INGEST-011 — No `documents` row created for any `failed/` entry.** For every file in `Paths.failed()` across the SP-003 test surface, the SQLite documents table contains zero rows with `status='success'` whose `source_path` matches the failed file's original path. (Rows with `status='failed'` and partial data MAY exist if a SP-006 partial-state recovery is in scope; SP-003 ships the clean "no success row" guarantee.)
- **SC-INGEST-012 — Telemetry event class coverage.** A mixed-workload run of 25+ files spanning success/dedup/rejection paths produces at least 6 distinct named telemetry event classes in `Paths.telemetry()` (the FR-INGEST-009 floor). Every emitted event validates against the canonical Zod telemetry schema with zero validation failures. (NFR-016 requirement.)
- **SC-INGEST-013 — Honest failure on telemetry-write failure.** When the JSONL telemetry append fails (filesystem mounted read-only, ENOSPC), the in-flight ingest is routed to `failed/` with `error_code='telemetry_write_failed'`. The system does NOT silently complete the ingest. (Principle XIII; Principle XVI.)
- **SC-INGEST-014 — Document body content never appears in telemetry.** Across every test run, zero telemetry event payloads contain substrings drawn from the body text of the ingested documents (verified by a deterministic post-hoc grep over the JSONL log against the fixture document bodies). Hashes, ids, paths, MIME types, sizes are permitted; body content is forbidden. (Principle I — local-first, no egress; Principle XIII — telemetry observability without content leak.)
- **SC-INGEST-015 — Drain lock serialization.** Two `corpus drain` processes launched concurrently produce exactly one effective drain run: one process acquires the lock and processes files; the other emits a `lock_contention` telemetry event and exits 0. Zero double-ingests occur. (Principle IX.)
- **SC-INGEST-016 — Cancellable IO under SIGTERM.** Sending SIGTERM during an in-flight ingest results in `controller.abort()` propagation, the in-flight document routed to `failed/` with `error_code='aborted'` and `retriable=true`, and process exit within 2 seconds. (Principle VII.)
- **SC-INGEST-017 — Library/CLI boundary lint passes.** A code-search lint over SP-003 source files under `packages/pipeline/`, `packages/extract/`, `packages/storage/`, `packages/inference/` detects zero `process.exit` invocations. (Principle XI.)
- **SC-INGEST-018 — XDG-paths-only lint passes.** A code-search lint over SP-003 source files detects zero writes to `/tmp/`, `/var/`, `os.tmpdir()`, system root paths, or any literal path not derivable from `Paths.*`. (Principle XIV.)
- **SC-INGEST-019 — Subprocess hygiene.** A code-search lint detects zero `execSync`, `exec`, or string-formed shell command invocations in SP-003 source. All subprocess invocations route through `runTool`. (Principle XII.)
- **SC-INGEST-020 — No agent-derived content lands in the canonical store.** Every body file under the canonical store is byte-identical to the deterministic normalization of the inbox source bytes; no LLM-generated, rewritten, or synthesized content appears in `documents.body_path` files. (Principle II; AG-002; SP-004 classifier writes frontmatter metadata, not body content.)

## Assumptions

- **Primary user**: shonrs on Fedora workstation (pai-node01) with Claude Code as the primary MCP-aware client.
- **Supported platforms for v1**: Linux (Fedora baseline) and macOS. Windows is out of scope for v1.
- **Prerequisite — SP-001 merged**: SP-001 (Local-Only Enforcement + MCP Server Foundation) is merged on `main`. SP-003 inherits the MCP server registration, the egress hook, the read-only stdio transport, and the empty `corpus.find` handler.
- **Prerequisite — SP-002 merged**: SP-002 (MCP Resources — Manifest, Taxonomy, Recent, Per-Document) is merged on `main`. SP-003 produces the data that the SP-002 resources surface; the SP-002 fixture-driven test paths re-verify with real SP-003 data once SP-003 lands.
- **Paths resolver keys exist**: `Paths.inbox()`, `Paths.pending()`, `Paths.processed()`, `Paths.failed()`, `Paths.data()`, `Paths.telemetry()`, `Paths.drainLock()`, `Paths.cache()` are present in `packages/contracts/src/paths.ts` (verified: lines 65-68 + cross-cutting Paths.* methods).
- **`documents` and `taxonomy_terms` schema baseline**: The schema in `packages/storage/src/schema-migration.ts` is the canonical column shape. SP-003 writes rows satisfying the existing constraints; the `hash` column is the FR-INGEST-005 idempotency key. The `documents.hash` UNIQUE constraint is a Plan-stage commitment (defense-in-depth alongside application-level dedup); if the existing schema lacks it, SP-003's Plan will add it as a forward-compatible migration.
- **Classifier-owned columns get sentinel values from SP-003**: `facet_domain`, `tags_json`, `facet_type`, `source_type` are NOT NULL in the current schema. SP-003 populates them with documented sentinel values (Plan-stage finalized) so the schema constraint passes; SP-004 (classifier) overwrites them with real classification output. Until SP-004 lands, no row in `taxonomy_terms` has `state='established'`, so `corpus://taxonomy` continues to return empty established lists per SP-002's contract.
- **`runTool` subprocess helper exists or will be scaffolded**: Per Principle XII, all subprocess invocation flows through a single `runTool(name, args, opts)` helper that uses `spawn` with an argument array. If the helper is not yet present, SP-003's Plan scaffolds it as a SP-003-shared utility (the principle is binding even if the helper's first user is SP-003).
- **PDF extractor and HTML-to-Markdown converter selection is Plan-stage**: ADR-007 names normalizer functions in the abstract (`pdf-parse` or equivalent for PDF, `turndown` or equivalent for HTML); concrete library selection is finalized in `/speckit-plan`. The selection MUST satisfy: no network egress (Principle I), bounded execution (Principle VII), error path → routes to `failed/` (FR-INGEST-007).
- **Watcher implementation choice is Plan-stage**: Linux `inotify` via Node's `chokidar` or `fs.watch`, or a deterministic poll loop, or some hybrid. The decision is Plan-stage; SP-003's spec-level commitment is "no dropped file is silently skipped under the user's documented drop pattern (`cp` / `mv` rename into `Paths.inbox()`)."
- **Per-document ingest budget**: NFR-014 (90-second first-run end-to-end) is a related but distinct envelope. SP-003's own per-document ingest budget is Plan-stage finalized.
- **Single-user, single-machine scope**: Per Constitution Principle IV, SP-003 designs for one user on one machine. No cross-user inbox, no shared inbox, no remote inbox. The watcher monitors one local filesystem path.
- **No `process.exit` in any SP-003 library code**: Per Principle XI, all SP-003 source under `packages/pipeline/`, `packages/extract/`, `packages/storage/`, `packages/inference/` MUST NOT exit the process. Only `packages/cli/`, `packages/transport/`, `packages/daemon/` may.

This feature delivers the producer side of the corpus: an inbox-watcher-driven, validation-gated, content-hash-idempotent, atomic-write ingest pipeline that lands normalized Markdown documents in the canonical store and emits ≥6 named telemetry event classes covering both success and failure paths. SP-003 is honest about what it produces (raw frontmatter with sentinel classifier columns) and what it defers (semantic classification to SP-004, embedding/ranking to SP-005, kill-9 survival and failure-lane MCP resource to SP-006). The agent, as principal user of the corpus per the project's framing, gains its first real data through SP-002's resources reading what SP-003 writes.
