# Feature Specification: Production Hardening — Kill-9 Cross-Stage Recovery, `corpus://failures` MCP Resource, Tier 1/2/3 Fallthrough

**Feature Branch**: `006-hardening`
**Created**: 2026-05-13
**Status**: Draft
**Input**: SP-006 from `.product/SPRINT-PLAN.yaml` / `.product/ROADMAP.yaml` (final substrate sprint — closes the design-ladder defer items per SP-003/004/005 plan citations and ARCHITECTURE-FINAL §10.6). Source FR text — SP-006 closes three explicitly-deferred items: (a) the SP-003 `failure-lane.ts` sidecar contract is consumed by `corpus://failures` as a read-only MCP resource (per SP-005 spec.md Out-of-Scope verbatim "SP-005 writes `<doc-id>.error.json` sidecars consistent with SP-003 / SP-004; the MCP resource exposing them is SP-006"); (b) kill-9 cross-stage survival (per SP-005 spec.md Out-of-Scope verbatim "SP-005 ships single-transaction atomicity for the embed → index → edges-build chain (Constitution VIII), but cross-stage recovery (resume an aborted edges-build from the partial-edges checkpoint) is SP-006 territory"); (c) ARCHITECTURE-FINAL §10.6 Tier 1/2/3 fallthrough (per SP-005 FR-RETRIEVAL-010 verbatim "Tier 1/2/3 fallthrough optimization is EXPLICITLY OUT OF SCOPE for SP-005, deferred to SP-006"). Source constitutional principles: I (Local-First, No Egress — Tier 3 fs-grep stays in `Paths.docs()`; recovery scanner reads only `Paths.telemetry()` JSONL), III (Substrate, Not Surface — `corpus://failures` is a fifth read-only MCP resource mirroring the SP-002 `corpus://{manifest,taxonomy,recent,docs/{id}}` pattern; ZERO mutation surfaces), V (Schema-Enforced Output — failures resource output Zod-validated; tier-fallthrough preserves SearchHit shape additively via new `tier_used` field enum), VII (Cancellable IO — recovery scanner takes AbortSignal; tier-fallthrough aggregate latency budget enforced via AbortController), IX (Concurrency-Safe Shared State — recovery scanner acquires `Paths.drainLock()` BEFORE re-queueing work; concurrent CLI invocations during recovery emit `pipeline.lock_contention` exit 0), X (Idempotent Pipeline Transitions — re-queued recovery work re-runs the same idempotent transitions; classify/embed/index on already-completed rows are no-ops), XI (Library/CLI Boundary — no `process.exit` in `packages/{contracts,inference,index,storage,pipeline,transport}`), XIII (Telemetry-or-Die — recovery and tier-fallthrough emit ≥ 6 new event classes), XIV (XDG Paths — all reads via `Paths.failed()`, `Paths.telemetry()`, `Paths.docs()`), XVI (Validation Honesty — §10.6 tier latency budgets are TARGETS, not guarantees; SP-006 measures empirically and reports honestly).

This is feature 006: the **production-hardening owner** for the post-SP-005 substrate. SP-001 shipped the MCP foundation + egress hook + `corpus.find` placeholder. SP-002 shipped the four read-only resources (`corpus://{manifest,taxonomy,recent,docs/{id}}`). SP-003 shipped the ingest pipeline + sentinel rows + drain-lock + the `failure-lane.ts` sidecar writer with verbatim schema `{doc_id, stage, error_code, message, timestamp, retriable}`. SP-004 shipped the classifier + body-file frontmatter mirror + proposed-state taxonomy. SP-005 shipped the four-signal hybrid retriever (BM25 + dense + graph + confidence) + RRF fusion + atomic index transaction + `corpus reindex` CLI; `tier_used` was hardcoded `'hybrid'` per FR-RETRIEVAL-010. SP-006 closes the three orthogonal production-hardening loops in one final substrate sprint: (1) when the daemon is killed mid-stage (SIGKILL — not handled by Constitution VII AbortSignal cleanup), restart-time recovery detects orphan work and either re-queues it (idempotent transitions per Constitution X) or fail-cleans with a `<doc-id>.recovery.error.json` sidecar; (2) the existing `<doc-id>.error.json` sidecar backlog at `Paths.failed()` is exposed to MCP-aware agents as a fifth read-only resource `corpus://failures` with `?stage=`, `?since=`, `?limit=`, `?offset=` query parameters (mirrors SP-002's four read-only resources structurally; no mutation surface); (3) the §10.6 Tier 1/2/3 cascade — Tier 0 (hybrid) returns fewer than `min_results` (default=3, config-driven) → fall to Tier 1 (BM25-only fast path, < 5 ms target) → Tier 2 (CATALOG.md body grep, < 50 ms target) → Tier 3 (filesystem `grep -rn` over `Paths.docs()`, < 500 ms target) — within an aggregate latency budget (default 600 ms = 20 + 5 + 50 + 500 + 25 ms slack). Each tier inherits the caller's master AbortSignal; the SearchHit returns the actually-matched tier in a new `tier_used` enum field (`hybrid`/`bm25-only`/`catalog-grep`/`fs-grep`). The SP-005 SearchHit shape is extended additively — no break. The SP-003 sidecar schema is consumed read-only — no break. SP-006 ships ZERO new MCP mutation surfaces (Constitution III); ZERO writes to `taxonomy_terms` (Principle XV); ZERO new XDG bases (Principle XIV); ZERO subprocesses except the Tier 3 `grep -rn` invocation via `runTool()` per Constitution XII subprocess hygiene. SP-006 is the FINAL substrate sprint: after merge, the corpus substrate is install-complete (SP-001..SP-006 on `main`), and product / app sprints (v1.5+) build on top.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — The daemon survives a SIGKILL mid-stage and recovers orphan work on restart (Priority: P1)

The user (Maya/Shon persona) is mid-ingest of a 50-file batch. Halfway through, the daemon is forcibly terminated — `kill -9 <pid>`, OOM-killer, machine power-loss, `systemctl kill -s KILL`, anything where Constitution VII's `AbortSignal` cleanup cannot run. Several documents are stuck in transient states: row exists with classify-started telemetry but no classify-completed; row exists classified but no embed-completed; row exists embedded but no index-completed; row exists indexed but no edges-completed. On daemon restart, BEFORE accepting any new ingest work, the recovery scanner scans `Paths.telemetry()` JSONL backwards from end-of-file to the most-recent `daemon.started` marker (the previous daemon's session boundary), builds a map of `*.started` events keyed by doc_id + stage, and for each checks whether a corresponding `*.completed` or `*.failed` event exists later in the same daemon-session log range. For each orphan (started without a matching completed/failed), the scanner consults the **resumability matrix** (Decision B in research.md): `ingest.*` orphans → resumable if the inbox file still exists (re-validate); `classify.*`/`embed.*`/`index.*`/`edges-build.*` orphans → resumable (the row already has the prior stage's output and the SP-005 idempotency contracts make re-running a no-op when already complete, or a clean redo when not). Non-resumable orphans get a `<doc-id>.recovery.error.json` sidecar at `Paths.failed()` with `error_code='unrecoverable_orphan'` and `stage=<stage>`. The recovery scanner acquires `Paths.drainLock()` BEFORE re-queueing any work (no race with concurrent CLI `corpus drain` / `corpus reenrich` / `corpus reindex`); a concurrent CLI invocation during recovery observes contention via the existing FR-INGEST-011 contract and emits `pipeline.lock_contention` + exits 0. After recovery completes the scanner releases the drain-lock and the daemon enters its normal accept-new-work state. Telemetry: `recovery.scan_started`, `recovery.orphan_found` (per orphan), `recovery.resumed` (per re-queued), `recovery.aborted` (per fail-clean), `recovery.scan_completed` are all emitted.

**Why this priority**: This is the load-bearing stakeholder requirement for SP-006 and the most-likely operational concern across the entire substrate lifecycle. Constitution VII handles soft-shutdown via `AbortSignal`, but the `kill -9` / power-loss / OOM-killer class of failures is NOT amenable to cooperative shutdown — those code paths skip every `finally` block, every transaction rollback, every signal handler. Without SP-006 US1, a user who kills the daemon during an embed (slow) or an edges-build (slow on large corpora) restarts to a corpus that quietly has classified-but-unindexed rows with no automatic recovery — they would have to remember to manually run `corpus reindex` after every hard kill. P1 because the user's stated trust model for the substrate ("I drop files and the corpus quietly does the right thing") requires that the daemon's lifecycle survive process kills as gracefully as it survives clean restarts.

**Independent Test**: Initialize a clean corpus on the user's primary machine with the SP-003 daemon running (post-SP-006 — daemon now runs the recovery scanner at startup before accepting new ingest work). Drop 20 mixed-MIME files into the inbox. Wait until ~10 have completed the four-sub-stage chain and ~10 are mid-pipeline (some classified-but-unindexed; some embedded-but-not-indexed; some indexed-but-not-edges-built). Send `kill -9 <pid>` to the daemon. Restart the daemon. Observe: stderr shows the recovery scan running before accept-new-work; recovery telemetry events fire (`recovery.scan_started` / `recovery.orphan_found` per orphan / `recovery.resumed` per re-queued / `recovery.aborted` per fail-clean / `recovery.scan_completed`); after recovery, the previously-orphaned documents complete their remaining sub-stages within the per-doc budget; no `recovery.error.json` sidecars exist for resumable orphans; any non-resumable orphans (e.g., inbox files manually deleted during the kill window) have `<doc-id>.recovery.error.json` sidecars with `error_code='unrecoverable_orphan'`.

**Acceptance Scenarios**:

1. **Given** a daemon running with N classified-but-unindexed rows whose `embed.started` events exist in `Paths.telemetry()` without matching `embed.completed`/`embed.failed`, **When** the daemon process receives SIGKILL and is then restarted, **Then** the new daemon process invokes the recovery scanner BEFORE accepting new ingest work AND the scanner acquires `Paths.drainLock()` AND scans backwards from end-of-file to the most-recent `daemon.started` marker AND classifies each orphan via the resumability matrix AND re-queues each resumable orphan into the appropriate sub-stage AND writes `<doc-id>.recovery.error.json` sidecars for non-resumable orphans AND emits `recovery.*` telemetry classes for each transition AND releases the drain-lock AND transitions the daemon into accept-new-work state.
2. **Given** a daemon killed mid-classify-stage where the row has sentinel state but no `classify.completed` event, **When** recovery runs, **Then** the orphan is classified as resumable (the SP-004 `classifyStage` is idempotent — re-running on an unclassified row produces the same classifier output OR routes the row to the failure lane with a documented `error_code`) AND the orphan is re-queued for classify AND a `recovery.resumed` event fires with `stage='classify'`.
3. **Given** a daemon killed mid-edges-build where the row is indexed but no `edges-build.completed` event exists, **When** recovery runs, **Then** the orphan is classified as resumable (the SP-005 edges-builder uses `INSERT OR IGNORE` per FR-RETRIEVAL-008; re-running is idempotent) AND the orphan is re-queued for edges-build AND a `recovery.resumed` event fires with `stage='edges-build'`.
4. **Given** a daemon killed mid-ingest where the inbox file was already moved but no `ingest.completed` event exists AND the inbox file was manually deleted during the kill window (operator interference, disk error, NFS unmount), **When** recovery runs, **Then** the orphan is classified as NON-resumable AND a `<doc-id>.recovery.error.json` sidecar is written at `Paths.failed()` with `error_code='unrecoverable_orphan'` and `stage='ingest'` AND a `recovery.aborted` event fires.
5. **Given** the daemon was killed during a previous recovery scan AND the recovery scan itself emitted `recovery.scan_started` without a matching `recovery.scan_completed`, **When** the new daemon's recovery scan runs, **Then** it detects the orphan recovery-scan marker AND skips re-recovering the same window (idempotent recovery; the recovery's own scan boundaries are the only operations not subject to recovery) AND emits a `recovery.scan_reentry` event AND proceeds to recover any orphans found from the current scan.
6. **Given** a concurrent `corpus drain` / `corpus reenrich` / `corpus reindex` CLI is invoked WHILE the recovery scanner holds the drain-lock, **When** the CLI attempts `flock(LOCK_EX | LOCK_NB)`, **Then** it observes contention AND emits `pipeline.lock_contention` telemetry AND exits 0 (FR-INGEST-011 / FR-CLASSIFY-015 / FR-RETRIEVAL-018 contract preserved across all writer surfaces).
7. **Given** the recovery scan completes with N resumed orphans + M fail-cleaned orphans, **When** telemetry is inspected, **Then** the `recovery.scan_completed` event payload carries `resumed_count: N`, `aborted_count: M`, `duration_ms`, `daemon_session_start_ts` (the `daemon.started` marker that bounded the scan range).

### User Story 2 — The agent reads the `corpus://failures` MCP resource to inspect ingest/classify/embed/index/edges failures (Priority: P1)

The agent (Pallas / Claude Code / any MCP-aware client) invokes `corpus://failures` to get the current backlog of `<doc-id>.error.json` sidecars at `Paths.failed()`. The resource returns a structured response `{entries: FailureEntry[], total_count: int, returned_count: int, schema_version: 1}` where each FailureEntry has the SP-003-written shape (`doc_id`, `stage`, `error_code`, `message`, `timestamp`, `retriable`) plus a SP-006-added `sidecar_path` (the absolute path under `Paths.failed()` so the operator can rm the file after triaging). The query supports the parameters `?stage=<stage>&since=<ISO date>&limit=<int>&offset=<int>` for paginated reads; defaults are `stage=*`, `since=<unbounded>`, `limit=50`, `offset=0`. The resource is READ-ONLY: the resource handler never writes a sidecar, never deletes a sidecar, never modifies a sidecar's contents; mutation surfaces (manual remediation: `rm <sidecar>` after fixing the underlying issue) are operator-driven CLI / fs-level concerns, not MCP-exposed. The agent uses this resource to answer questions like "what classification failures happened in the last 24 hours?" or "how many embed-stage failures are retriable vs. non-retriable?" — directly relevant to the substrate's debuggability story per WHITEPAPER-FINAL.md's "the corpus is itself queryable for what's broken" principle.

**Why this priority**: P1 because (a) the `<doc-id>.error.json` sidecars have been silently accumulating since SP-003 with no agent-facing surface to read them (operators have had to `ls Paths.failed()/*.error.json` and `cat` each one — not a substrate-grade workflow); (b) the resource is structurally trivial (it MIRRORS the SP-002 `corpus://recent` pattern with paginated reads); (c) SP-005's spec.md Out-of-Scope explicitly defers it to SP-006. Shipping it in SP-006 closes the last agent-facing read surface of the substrate, completing the four-corners-of-debuggability triple (manifest = "what exists", taxonomy = "what's classified", recent = "what just succeeded", failures = "what's broken"). P1 also because the alternative (defer to v1.5+) would leave SP-001..SP-005's failure-lane writes invisible to the agent for the entire v1.0 lifetime.

**Independent Test**: With the SP-003 + SP-004 + SP-005 baseline running and a seeded inbox of 10 mixed-MIME documents that include 2 deliberately-malformed files (corrupt MIME headers triggering SP-003 validation failures + LLM-classifier rejection triggering SP-004 schema failures), drive a normal drain. Confirm 2 sidecars exist at `Paths.failed()/*.error.json` with the SP-003-verbatim schema. Then invoke `corpus://failures` from an MCP client (no query parameters). Verify: the response is a structured JSON object with an `entries` array of length 2 + `total_count: 2` + `returned_count: 2` + `schema_version: 1`; each entry has the SP-003 fields plus the SP-006 `sidecar_path` field. Invoke `corpus://failures?stage=classify&limit=1` — verify response has `entries.length === 1`, `total_count: 1` (the one classify-stage sidecar), `returned_count: 1`. Invoke `corpus://failures?since=<future-date>` — verify `entries.length === 0`, `total_count: 0`. Invoke `corpus://failures?stage=invalid_stage_name` — verify a `validation_error` envelope is returned as a successful resource response (NOT a transport error).

**Acceptance Scenarios**:

1. **Given** N `<doc-id>.error.json` sidecars exist at `Paths.failed()` with the SP-003-verbatim shape, **When** the agent invokes `corpus://failures` with no query parameters, **Then** the response is `{entries: FailureEntry[], total_count: N, returned_count: min(N, 50), schema_version: 1}` AND each entry carries the SP-003 fields (`doc_id`, `stage`, `error_code`, `message`, `timestamp`, `retriable`) PLUS the SP-006-added `sidecar_path` field AND entries are ordered descending by `timestamp` (most-recent failure first) AND a `resource.read` telemetry event fires (existing SP-002 telemetry class).
2. **Given** the agent invokes `corpus://failures?stage=embed`, **When** the handler runs, **Then** the response includes only entries whose `stage === 'embed'` AND `total_count` reflects the post-filter count AND `returned_count` reflects min(total_count, limit).
3. **Given** the agent invokes `corpus://failures?since=2026-05-10T00:00:00Z&limit=10`, **When** the handler runs, **Then** the response includes only entries whose `timestamp >= '2026-05-10T00:00:00Z'` AND `entries.length <= 10` AND `total_count` reflects the post-filter total AND offset defaults to 0.
4. **Given** the agent invokes `corpus://failures?limit=5&offset=10`, **When** the handler runs against a backlog of 20 sidecars, **Then** the response includes entries 11-15 (zero-indexed offset=10, limit=5) AND `total_count: 20` AND `returned_count: 5`.
5. **Given** the agent invokes `corpus://failures?stage=nonexistent_stage`, **When** the Zod input validation runs against the closed stage-enum, **Then** the response is a `{error_code: 'validation_error', message, hint}` envelope returned as a successful MCP resource response (NOT a transport error) AND a `resource.read` event fires with `outcome='failure'` and `error_code='validation_error'`.
6. **Given** a sidecar's JSON is malformed (manual edit, partial write from a SP-003 bug), **When** the handler parses it, **Then** the malformed sidecar is SKIPPED from the response AND a `failures.sidecar_parse_failed` event is emitted with the sidecar path AND the remaining well-formed sidecars are still returned. (The resource degrades gracefully on a per-sidecar basis; one bad sidecar doesn't fail the whole read.)
7. **Given** `Paths.failed()` directory does not exist (clean install, no failures yet), **When** the agent invokes `corpus://failures`, **Then** the response is `{entries: [], total_count: 0, returned_count: 0, schema_version: 1}` as a successful read AND no error envelope is returned AND a `resource.read` event fires.

### User Story 3 — When Tier 0 (hybrid) returns fewer than `min_results`, search falls through to Tiers 1/2/3 within an aggregate latency budget (Priority: P2)

The agent invokes `corpus.find({query: <natural-language>})`. The Tier 0 (hybrid) search runs as in SP-005; if it returns fewer than `[search].min_results` hits (default=3, configurable in `config.toml`), the search orchestrator falls through to Tier 1 (BM25-only fast path against the existing FTS5 table, < 5 ms target per §10.6) — which may surface hits that Tier 0's dense-vector signal dropped. If Tier 1 still falls short, the orchestrator falls through to Tier 2 (body grep over an auto-generated `CATALOG.md` at `Paths.data() + '/CATALOG.md'` — written at SP-005 index-stage time as part of the index transaction; this is additive to the SP-005 index-writer and falls through gracefully if the file is absent on legacy DBs). If Tier 2 still falls short, the orchestrator falls through to Tier 3 (`grep -rn` over `Paths.docs()` via `runTool()` subprocess invocation per Constitution XII subprocess hygiene; < 500 ms target). The aggregate latency budget across all four tiers is configurable in `config.toml [search].tier_total_budget_ms` (default 600 ms = 20 ms Tier 0 + 5 ms Tier 1 + 50 ms Tier 2 + 500 ms Tier 3 + 25 ms slack), enforced via an AbortController firing across the whole cascade. Each SearchHit carries a new `tier_used` field (enum: `hybrid` / `bm25-only` / `catalog-grep` / `fs-grep`) indicating which tier produced it; the `search.completed` event additionally carries `tier_used` reflecting the lowest tier that produced any hit. If all four tiers exhaust the budget without producing min_results, the response is the partial set with `tier_used: 'fs-grep'` (the lowest-numbered tier that ran to completion); telemetry `search.tier_budget_exceeded` fires.

**Why this priority**: P2 because the SP-005 four-signal hybrid retriever is already production-quality at Tier 0 — most queries are answered within Tier 0 with confidence. Tier fallthrough is a graceful-degradation guard for the long tail of queries where Tier 0 underperforms (rare-term queries, queries against an under-classified corpus, queries during partial-signal degradation). Shipping it in SP-006 closes the §10.6 architectural commitment but the operational impact is small in the typical case. P2 also because Tier 3 (`grep -rn`) is the only new subprocess invocation in the substrate and warrants the standalone scope to verify the Constitution XII `runTool()` pattern, the AbortController-driven kill, and the latency budget enforcement.

**Independent Test**: With the SP-005 baseline (US1+US2 from SP-005 verified end-to-end), construct a query that's known to return ≥ 3 hits at Tier 0 (the SP-005 happy path); invoke `corpus.find` and verify `tier_used: 'hybrid'` in every SearchHit and the `search.completed` event. Then construct a query that's known to return < 3 hits at Tier 0 but ≥ 3 hits at Tier 1 (a literal rare term not embedded but indexed in FTS5); invoke `corpus.find` and verify some SearchHits have `tier_used: 'hybrid'` (the few that Tier 0 found) and others have `tier_used: 'bm25-only'` (the ones Tier 1 produced); `search.tier_fallthrough` event fires with `from_tier='hybrid'`, `to_tier='bm25-only'`, `reason='below_min_results'`. Then construct a query against a corpus where `CATALOG.md` exists but FTS5 is empty (forced via manual `DELETE FROM documents_fts`); invoke `corpus.find` and verify `tier_used: 'catalog-grep'` on the hits. Finally construct a query that requires Tier 3 (delete `documents_fts` AND delete `CATALOG.md`); invoke `corpus.find` and verify `tier_used: 'fs-grep'` on the hits AND the `runTool('grep', ['-rn', ...])` invocation fired AND the SQLite write path was never touched.

**Acceptance Scenarios**:

1. **Given** Tier 0 (hybrid) returns ≥ `min_results` hits (default=3), **When** the search orchestrator runs, **Then** NO fallthrough occurs AND every SearchHit has `tier_used: 'hybrid'` AND the `search.completed` event has `tier_used: 'hybrid'`.
2. **Given** Tier 0 returns < `min_results` but Tier 1 produces enough additional hits to reach `min_results`, **When** the orchestrator runs, **Then** a `search.tier_fallthrough` event fires with `from_tier='hybrid'`, `to_tier='bm25-only'`, `reason='below_min_results'`, `hits_before_fallthrough: <int>` AND the final SearchHit list mixes `tier_used: 'hybrid'` (the Tier 0 hits) with `tier_used: 'bm25-only'` (the Tier 1 hits) AND the response carries `tier_used: 'bm25-only'` (the deepest tier that ran).
3. **Given** Tier 0 + Tier 1 + Tier 2 all return < `min_results` AND Tier 3 (`grep -rn` over `Paths.docs()`) produces enough hits to reach `min_results`, **When** the orchestrator runs, **Then** the cascade emits three `search.tier_fallthrough` events AND `runTool('grep', ['-rn', '--include=*.md', '-l', <pattern>, Paths.docs()])` is invoked (with AbortSignal propagation per Constitution VII) AND the hits carry `tier_used: 'fs-grep'`.
4. **Given** the cascade exhausts the aggregate latency budget (default 600 ms) before any tier produces `min_results`, **When** the AbortController fires, **Then** all in-flight tier work is cancelled AND the response is the partial set with `tier_used` set to the deepest tier that completed AND a `search.tier_budget_exceeded` event fires with `budget_ms`, `actual_ms`, `tiers_attempted`, `final_hit_count`.
5. **Given** `CATALOG.md` does not exist (legacy DB pre-SP-006-migration), **When** Tier 2 is invoked, **Then** the orchestrator emits a `search.tier_skipped` event with `tier='catalog-grep'`, `reason='catalog_missing'` AND falls through to Tier 3 AND does NOT raise an error.
6. **Given** the user invokes `corpus.find` with no special configuration, **When** the orchestrator runs, **Then** Tier 0 is ALWAYS attempted first (Tier 1/2/3 are only fallthroughs, not standalone entry points) AND `tier_used` in every successful response reflects the actual matched tier (not a default).
7. **Given** the search response includes any `tier_used: 'fs-grep'` hits, **When** telemetry is inspected, **Then** the `grep` subprocess invocation appears in the `runTool` telemetry stream (existing Constitution XII surface) AND the invocation's `args` array contains the `Paths.docs()` literal path (no string-formed shell command).

### Edge Cases

- **Daemon killed mid-classify**: SP-005's classify-stage commits transactionally; the row is either fully classified (`facet_type != 'unclassified'`) or fully sentinel. If killed AFTER `classify.started` and BEFORE `classify.completed`, the recovery scanner detects the orphan, classifies as resumable (the SP-004 `classifyStage` is idempotent — re-running on an unclassified row produces the same output), re-queues for classify. The SP-005 atomic index transaction has not yet started; nothing to roll back.
- **Daemon killed mid-embed**: Row is classified but `embed.started` exists without `embed.completed`. Recovery classifies as resumable; the embed-stage on the same body content + same configured model produces a deterministic vector. The vector is re-embedded; the document re-enters the SP-005 index transaction.
- **Daemon killed during recovery**: The recovery scan itself can be interrupted. Scenario 5 of US1 handles this: the new daemon detects a `recovery.scan_started` without `recovery.scan_completed` and skips re-recovering the same window (idempotent recovery — `recovery.scan_reentry` event); the scanner re-runs against the current end-of-log to detect any orphans from the current scan boundary.
- **Telemetry log corrupted (last line malformed)**: The recovery scanner reads JSONL line-by-line; a malformed last line (partial write from the SIGKILL itself) is gracefully skipped — the scanner discards the partial line and proceeds with the previous well-formed line as the scan boundary. The scanner emits `recovery.telemetry_parse_failed` for the bad line. The scan continues.
- **Sidecar JSON malformed**: The `corpus://failures` resource handler reads each `*.error.json` and parses it; a malformed sidecar is skipped from the response AND a `failures.sidecar_parse_failed` event is emitted with the sidecar path. The remaining well-formed sidecars are returned unaffected. This is per-sidecar graceful degradation.
- **`CATALOG.md` absent at Tier 2**: The Tier 2 retriever shells out to `runTool('grep', ['-l', <pattern>, <CATALOG.md path>])` only if the file exists. If absent (legacy DB, manual rm, never-indexed corpus), the orchestrator emits `search.tier_skipped` with `tier='catalog-grep'`, `reason='catalog_missing'` and immediately falls through to Tier 3. No error.
- **All four tiers return empty**: The cascade fully exhausts; response is `{hits: [], result_count: 0, tier_used: 'fs-grep', signals_used: ['bm25','dense','graph','confidence']}`. A `search.tier_budget_exceeded` event MAY fire if the budget was exhausted; otherwise a normal `search.completed` event fires with `result_count: 0`. NO error envelope.
- **Tier 3 `grep` subprocess fails to start**: `runTool` returns an error (e.g., `ENOENT` on `grep`); the cascade emits `search.tier_failed` with `tier='fs-grep'`, `errno=<errno>` and returns the partial set from previous tiers. No error envelope unless the partial set is also empty (then `search.error` with `all_tiers_failed`).
- **Recovery scan runs before the daemon's `started` event has been emitted**: The recovery scanner is ordered BEFORE the daemon's `daemon.started` event emission. Scenario: the scanner finds no prior daemon-session marker (clean install — empty telemetry log) and skips the scan entirely; emits `recovery.scan_skipped` with `reason='no_prior_session'` and the daemon proceeds to its normal startup.
- **Query during recovery**: A `corpus.find` invocation during the recovery scan READS the existing index tables (read-only operation; not gated by drain-lock). The query returns whatever state the index is in mid-recovery (potentially incomplete). This is acceptable per Constitution III (substrate, not surface — queries are non-blocking) — the user gets honest results that reflect actual indexed state. The recovery scan does NOT block reads. (Cross-reference: Tier 1 BM25-only is the fastest read path during recovery if Tier 0 hybrid is stale.)
- **Concurrent recovery across two daemons (process-level race)**: A second daemon process could in principle be started while the first is still recovering. The drain-lock at `Paths.drainLock()` is the serialization point (Constitution IX); the second daemon's recovery scanner acquires the lock via `flock(LOCK_EX | LOCK_NB)` → on contention, emits `pipeline.lock_contention` and exits 0 cleanly. Only ONE daemon's recovery runs at a time.
- **Large failure backlog (1000+ sidecars)**: The `corpus://failures` resource is bounded by limit/offset; default `limit=50`. Without pagination the read would be O(N) on the sidecar count; a 1000-sidecar backlog at default limit=50 returns the most-recent 50 with `total_count: 1000`. Parse-per-sidecar latency is bounded; expect < 100 ms on a 1000-sidecar backlog with default limit (measured empirically; documented in plan.md).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-HARDEN-001 — On daemon startup, the recovery scanner runs BEFORE the daemon accepts new ingest work.** The daemon's `startDaemon()` entry point invokes `runRecoveryScan(deps)` synchronously (awaited) before the watcher / classify-hook / embed-hook chains are activated. The recovery scan acquires `Paths.drainLock()` via `flock(LOCK_EX | LOCK_NB)` (Constitution IX single serialization point); on contention the scanner emits `pipeline.lock_contention` and exits cleanly without re-recovering. On successful lock acquisition, the scanner reads `Paths.telemetry()` JSONL from end-of-file backwards until it encounters a `daemon.started` event (the previous daemon-session boundary); within that window, it builds a map of `*.started` events keyed by `(doc_id, stage)` and checks for matching `*.completed` or `*.failed` events; orphans (started without completed/failed) are routed through the resumability matrix (Decision B). After processing, the lock is released. *(Sources: Constitution IX; Decision A in research.md.)*

- **FR-HARDEN-002 — The resumability matrix routes orphans to the correct recovery action.** Per Decision B in research.md: `ingest.*` orphans → resumable if the inbox file still exists at `Paths.inbox() + '/' + filename` (re-run SP-003 validation + hash + normalize + persist; idempotent because SP-003's persister checks for duplicate hashes); → non-resumable if the inbox file is absent (the operator removed it during the kill window; emit `unrecoverable_orphan` sidecar). `classify.*` orphans → ALWAYS resumable (the SP-004 classifier is idempotent — re-classifying an `facet_type='unclassified'` row produces the same output OR routes to failure lane with a documented `error_code`). `embed.*` orphans → ALWAYS resumable (the SP-005 embed-stage is idempotent — re-running on a row that has a `documents_vec` entry is a no-op per FR-RETRIEVAL-012; re-running on a row without one re-embeds the same body content with the same model). `index.*` orphans → ALWAYS resumable (the SP-005 index-stage uses `INSERT OR IGNORE` per FR-RETRIEVAL-008; `documents_fts` insert is idempotent because the index transaction is atomic — a kill DURING the transaction rolls back to before any inserts). `edges-build.*` orphans → ALWAYS resumable (the SP-005 edges-builder uses `INSERT OR IGNORE` per FR-RETRIEVAL-008). All non-resumable cases produce a `<doc-id>.recovery.error.json` sidecar at `Paths.failed()` with `error_code='unrecoverable_orphan'`, `stage=<stage>`, `message=<diagnostic>`, `timestamp=<scan-time>`, `retriable=false`. *(Sources: Constitution X — idempotent transitions; Decision B in research.md.)*

- **FR-HARDEN-003 — The recovery scanner is itself idempotent — re-running it produces the same set of orphans + same set of resumed re-queues.** Per Constitution X: the scan is deterministic over `(telemetry log, inbox state, db state)`. Running the scanner twice in succession (e.g., daemon restart loop during testing) re-detects the same orphans, re-classifies them via the same resumability matrix, and re-queues already-resumed work — the re-queue itself is a no-op because the resumed work's underlying transitions (classify, embed, index, edges-build) are all idempotent per the SP-003/004/005 atomic contracts. The scanner records its own session boundaries via `recovery.scan_started` + `recovery.scan_completed` events; a `recovery.scan_started` without a matching `recovery.scan_completed` indicates a recovery-during-recovery scenario (Scenario 5 of US1) and emits a `recovery.scan_reentry` event. *(Sources: Constitution X; Decision C in research.md.)*

- **FR-HARDEN-004 — The recovery scanner reads `Paths.telemetry()` JSONL backwards from end-of-file to the most-recent `daemon.started` marker.** The scan window is bounded by the previous daemon session's lifetime; events before the most-recent `daemon.started` belong to even-older sessions whose orphans were either resumed in their own recovery cycle OR fail-cleaned with a sidecar already on disk. The reverse-line-iterator implementation reads the file in chunks, splitting on newlines, parsing JSON per line; malformed lines are skipped with a `recovery.telemetry_parse_failed` event. If no `daemon.started` marker is found (clean install, log truncated), the scanner emits `recovery.scan_skipped` with `reason='no_prior_session'` and exits cleanly without scanning. *(Sources: Constitution XIII, XIV; Decision A in research.md.)*

- **FR-HARDEN-005 — The recovery scanner emits ≥ 6 new SP-006 telemetry event classes for full observability.** Per Constitution XIII: `recovery.scan_started`, `recovery.scan_completed`, `recovery.scan_skipped` (when no prior daemon-session marker), `recovery.scan_reentry` (when prior scan was itself orphaned), `recovery.orphan_found` (per orphan), `recovery.resumed` (per resumable re-queue), `recovery.aborted` (per non-resumable fail-clean), `recovery.telemetry_parse_failed` (per malformed line). Each event Zod-validates against the existing `TelemetryEvent` discriminated union (additive variants). *(Sources: Constitution XIII; Decision D in research.md.)*

- **FR-HARDEN-006 — The recovery scanner runs inside `Paths.drainLock()` to prevent races with concurrent CLI writer paths.** Per Constitution IX: the scanner acquires `flock(LOCK_EX | LOCK_NB)` BEFORE reading the telemetry log AND before re-queueing work; on contention the scanner emits `pipeline.lock_contention` + exits cleanly (the SP-003 / SP-004 / SP-005 contract preserved across all writer surfaces); on successful acquisition the lock is held for the entire scan + re-queue duration; the lock is released after the daemon enters accept-new-work state. The re-queueing path uses the existing SP-003 inbox enqueue / SP-004 classify-hook / SP-005 retrieval-orchestrator surfaces — these are the same code paths that handle new ingests, NOT a separate recovery write path. *(Sources: Constitution IX; FR-INGEST-011, FR-CLASSIFY-015, FR-RETRIEVAL-018.)*

Inside the scan, `signal.throwIfAborted()` is called between each orphan resolution and between each telemetry-log read chunk. - **SC-HARDEN-005 — Recovery during recovery (Scenario 5 of US1) is detected.** With the daemon mid-recovery scan (`recovery.scan_started` emitted, but the scan was itself killed before `recovery.scan_completed`), the new daemon's recovery scan detects the prior-orphaned scan AND emits a `recovery.scan_reentry` event AND proceeds with the current-session scan (the prior scan's orphans are either resumed or re-detected based on the inbox/db state at scan-time). ## Assumptions

- **Primary user**: shonrs on Fedora workstation (pai-node01) with Claude Code as the primary MCP-aware client; same scope as SP-001..SP-005.
- **Supported platforms for v1**: Linux (Fedora baseline) and macOS. Windows out of scope for v1. The Tier 3 `grep -rn` invocation assumes a POSIX-compatible `grep` binary on `$PATH` (vanishingly safe on Linux/macOS).
- **Prerequisite — SP-001..SP-005 merged**: SP-001 (egress hook + MCP server foundation) + SP-002 (resource layer) + SP-003 (ingest pipeline + failure-lane.ts sidecar writer) + SP-004 (classifier) + SP-005 (hybrid retrieval + atomic index transaction) all merged on `main`. SP-006 consumes the SP-003 sidecar schema verbatim AND extends the SP-005 SearchHit shape additively.
- **`grep` binary is available**: Tier 3 fs-grep invokes `grep -rn -l --include=*.md <pattern> <path>` via `runTool()`. Assumed present on PATH on Linux and macOS. If absent (`ENOENT`), Tier 3 emits `search.tier_failed` with `errno='ENOENT'` and the cascade returns whatever prior tiers produced.
- **`Paths.*` resolver keys**: SP-006 reuses existing getters only — `Paths.failed()`, `Paths.telemetry()`, `Paths.docs()`, `Paths.data()`, `Paths.drainLock()`, `Paths.inbox()`. NO new XDG base.
- **Schema baselines**: SP-006 does NOT modify any SQL tables. The recovery scanner consumes the existing `documents` columns + the SP-005 `documents_fts`/`documents_vec`/`edges` tables read-only; the `corpus://failures` resource consumes the SP-003 `Paths.failed()/*.error.json` sidecars read-only; the tier-fallthrough cascade consumes the SP-005 `documents_fts` read-only. The CATALOG.md generator (FR-HARDEN-018) is a new flat-file write target, not a new SQL table.
- **Failure-lane sidecar schema is stable**: SP-003's `<doc-id>.error.json` shape `{doc_id, stage, error_code, message, timestamp, retriable}` is consumed READ-ONLY by SP-006. The schema is NOT extended (the SP-006-added `sidecar_path` field lives in the RESOURCE-RESPONSE shape, not in the on-disk sidecar shape).
- **SP-005 `SearchHit` shape is extended additively**: SP-006 adds a new `tier_used: enum` field to `SearchHitZodSchema`. The existing fields (`uri`, `score`, `title`, `facet_domain`, `facet_type`, `tags`, `snippet`) are unchanged. The SP-005 `search.completed` event's `tier_used: 'hybrid'` literal is updated to the enum (additive in the enum-domain sense).
- **No worker-pool parallelism for tier-fallthrough**: SP-006 ships single-threaded sequential cascade (Tier 0 → Tier 1 → Tier 2 → Tier 3), mirroring SP-005 FR-RETRIEVAL-002 (Tier 0's four retrievers run in parallel; the four tiers themselves run sequentially because each is a gated fallthrough).
- **`corpus://failures` is the only new MCP surface**: No new MCP tools, no new prompts. The four existing read-only resources (`corpus://{manifest, taxonomy, recent, docs/{id}}`) are unchanged.
- **No `process.exit` in any SP-006 library code**: Per Principle XI, all SP-006 source under `packages/{contracts,inference,index,storage,pipeline,transport,daemon}/` MUST NOT exit the process. Only `packages/cli/` may.
- **The recovery scanner's telemetry-log read is bounded by the previous daemon-session size**: For typical sessions of 1000-10000 events, the reverse-line iteration completes sub-second on pai-node01. For pathological sessions (millions of events from a very-long-running daemon), the scanner is still bounded by the per-scan timeout (30s default).
- **CATALOG.md is operator-readable**: The file lives at `Paths.data() + '/CATALOG.md'` — a known XDG-compliant path. The operator can `cat` it for a quick textual overview of the corpus. This is a low-stakes nice-to-have; the primary substrate read-paths are the SP-002 MCP resources and SP-005 `corpus.find`.
- **No automatic sidecar cleanup**: SP-006 does NOT delete sidecars after the underlying issue is fixed. Operator manually `rm Paths.failed()/<doc-id>.error.json` after triaging; the `corpus://failures` resource then shows a smaller backlog. A future sprint may add a `corpus failures clear --stage=<stage>` CLI surface; SP-006 ships only the read surface.

## Out of Scope (deferred to other features or explicitly excluded)

- **`corpus failures clear` / `corpus failures retry` CLI commands** — out of scope for SP-006. The `corpus://failures` resource is READ-ONLY. Operator-driven remediation (rm the sidecar after fixing; manually re-ingest the file) is the v1 workflow. A future sprint may add CLI surfaces.
- **MCP mutation surfaces of any kind** — FORBIDDEN by Principle III. SP-006 ships ZERO new mutation surfaces — the new `corpus://failures` is read-only by construction.
- **Automated kill-9 testing in CI** — out of scope. The kill-9 recovery tests run on the user's primary machine with controlled fixture injection; CI tests cover the recovery-scanner logic against simulated telemetry-log fixtures, NOT actual SIGKILL injection (CI environments don't reliably support process-kill semantics).
- **Recovery from corrupted SQLite file** — out of scope. SP-006 recovery scope is process kills, not data corruption. SQLite-file corruption is a v1.5+ concern (would need WAL replay, page-level recovery, etc.); SP-006 ships the kill-9 cross-stage recovery only.
- **Retrieval-evaluation harness** — explicitly out of scope per Constitution XVI / NFR-009 (deferred to v1.5+ per SP-005 spec.md Out-of-Scope).
- **CATALOG.md backfill for legacy corpora** — partial scope. The new CATALOG.md is auto-generated for SP-006+ ingests; `corpus reindex` is extended to regenerate it for already-classified corpora. Pre-SP-006 corpora that don't run reindex still get graceful Tier 2 skip + Tier 3 fallthrough.
- **Tier 4+ retrievers** — out of scope. The §10.6 four-tier model is the architectural ceiling. Future-horizon if the cascade proves insufficient.
- **Recovery for `corpus reindex` mid-run kill** — partial scope. If the CLI command is killed mid-flight, the row is left classified-but-partially-indexed (the SP-005 transaction either rolled back or committed atomically); the SP-006 recovery scanner detects orphans from CLI invocations the same way it detects daemon orphans (CLI sub-stage events are written to the same `Paths.telemetry()` log). Treated as in-scope for the resumability matrix.
- **Cross-corpus / federated recovery** — out of scope per Constitution IV.
- **Worker-pool parallelism for tier cascade** — out of scope. v1 ships sequential cascade.
- **Chunked embeddings or HNSW or cross-encoder re-ranking** — out of scope (inherited from SP-005 deferrals).
- **Auto-detection of embedding-model change for recovery** — out of scope. Embedding-model mismatch is a manual-reindex concern (SP-005 documented this).
- **User-facing review UI for recovery decisions** — out of scope. The recovery scanner is fully autonomous (the resumability matrix is deterministic; no human-in-the-loop).
- **`corpus://recovery` MCP resource** — out of scope. Recovery state is observable via telemetry events + the existing `corpus://failures` resource (which surfaces the `*.recovery.error.json` sidecars). A separate resource would be redundant.
- **Recovery sidecar auto-deletion** — out of scope. `<doc-id>.recovery.error.json` sidecars accumulate forever; the operator manually rm's them. The `corpus://failures` resource's paginated read handles large backlogs gracefully.
- **PRD-defined CATALOG.md format extensions** — out of scope. The v1 format is `<doc-id> | <title> | <facet_domain> | <facet_type> | <summary-first-200-chars>`. Future ADRs may extend fields if Tier 2 quality data demands.
- **Cross-agent compatibility claims** — FR-022 / FR-023 / NFR-007 / NFR-008; should-priority next-horizon; non-goal for v1 per Principle XVI.

This feature delivers the production-hardening owner for the SP-001..SP-005 substrate: a kill-9 cross-stage recovery scanner that detects orphans from `Paths.telemetry()` JSONL and re-queues them through the existing idempotent transitions (or fail-cleans with a recovery sidecar); a fifth read-only MCP resource `corpus://failures` exposing the SP-003-written sidecar backlog to agents with `?stage=`, `?since=`, `?limit=`, `?offset=` paginated reads; and the §10.6 Tier 1/2/3 fallthrough cascade (BM25-only / CATALOG.md grep / fs-grep) honoring an aggregate latency budget with per-tier targets reported honestly per Constitution XVI. SP-006 is honest about what it produces (recovery + failures-read + tier-fallthrough, all read-only / idempotent / drain-lock-serialized) and what it defers (sidecar mutation surfaces to future sprints, eval-harness to v1.5+, Tier 4+ to future-horizon, cross-corpus federation to never). SP-006 is the FINAL substrate sprint: after merge, the substrate is install-complete and product sprints build on top.
