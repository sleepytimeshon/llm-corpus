## User Scenarios & Testing *(mandatory)*

### User Story 1 — UR-001: drop a file into the inbox and have it become queryable without further action (Priority: P1)

The user (Maya / Shon-as-Maya during the dogfood window) has a working SP-007 install (US1 verified) with the daemon running. The user drops a single PDF, Markdown, or plain-text document into `Paths.inbox()` via any file-manager mechanism (drag-and-drop, `cp`, IDE save-as, file-system sync). The inbox watcher (SP-003 FR-010) enqueues the file; the ingest pipeline normalizes the document (SP-003 FR-011); the SP-004 classifier produces schema-valid metadata via grammar-constrained generation against the established + curated-seed taxonomy; the SP-005 embed + index stages persist the document; the SP-006 edges-build stage completes the per-document graph. From a Claude Code session, the user asks the agent a question whose answer is in the document. The agent invokes `corpus.find`; the SP-005 hybrid retriever returns a ranked SearchHit list with the dropped document in the top-K. The user (or the agent on the user's behalf) issues `corpus accept <request-id>` for the returned `corpus.find` query, recording an explicit acceptance event. The user has NOT performed any classification, embedding, indexing, or vocabulary configuration. The document went from "dropped into inbox" to "returnable from `corpus.find` on the next matching query" with no further action.

**Why this priority**: P1 because UR-001 is the AI-native operator value proposition — "drop files and trust the pipeline" is the substrate's first promise. The SP-007 install + 90-second first-run UX delivers the entry path; SP-008 UR-001 verifies the entry path produces a queryable document end-to-end. P1 also because without UR-001 the Maya engagement-proxy is unmeasurable (the operator cannot run `corpus.find` queries against documents that didn't make it into the index).

**Independent Test**: On a fresh SP-007 install with the daemon running, the operator drops `~/Downloads/some-research.pdf` into `Paths.inbox()`. The operator does NOT run any classification, indexing, or configuration commands. The operator waits for the SP-006-recorded `edges-build.completed` telemetry event (typical wall-clock: 5-15 seconds depending on Ollama latency for the SP-004 classifier + SP-005 embedder). The operator then opens a Claude Code session and asks "what does my corpus say about <topic-from-the-PDF>?". The agent invokes `corpus.find({query: '<topic>'})`; the returned response contains ≥ 1 SearchHit whose URI dereferences (via `corpus://docs/{id}` per SP-002 FR-008) to a normalized version of the dropped PDF. The operator runs `corpus accept <request-id>` (the SP-008 acceptance-event CLI, see FR-ACCEPT-001); the acceptance event is recorded in `Paths.telemetry()` as an `engagement.acceptance_event` linked to the `engagement.corpus_find_invoked` event by `request_id`. Verified by `corpus engagement-proxy report --since=<5min-ago>` showing ≥ 1 `corpus.find` query AND ≥ 1 acceptance event in the window.

**Acceptance Scenarios**:

1. **Given** a working SP-007 install with the daemon running AND the inbox empty AND the index empty (or containing only the SP-007 smoke seed), **When** the operator drops a single PDF, Markdown, or plain-text document into `Paths.inbox()` and performs NO further commands, **Then** the document traverses validate → normalize → classify → embed → index → edges-build within the SP-003/SP-004/SP-005/SP-006 per-stage budgets AND on a subsequent `corpus.find({query: '<topic-from-the-doc>'})` invocation the document appears in the returned SearchHit list with `result_count ≥ 1` AND the SearchHit's URI dereferences to a normalized Markdown body of the dropped document AND the `corpus.find` call emits an `engagement.corpus_find_invoked` telemetry event with `query`, `result_count ≥ 1`, `tier_used`, `request_id`, and `timestamp` fields.
2. **Given** a document whose body normalizes to text the classifier proposes a brand-new domain or tag for (proposed term not in the established + seeded taxonomy), **When** the classifier processes the document, **Then** the document is NOT silently queryable under a never-seen-established-term AND the proposed term is recorded in `taxonomy_terms` with `state='proposed'` per SP-004 FR-014 AND the document is either (a) routed to `facet_type='unclassified'` AND surfaceable via `corpus://failures` per SP-006 OR (b) classified under the established axes only and indexed (operator can later run `corpus taxonomy promote` from SP-007 to elevate the proposed term and re-enrich). The exact stuck/indexed disposition is the SP-004 + SP-007 contract; UR-001 acceptance for this scenario is that the proposed term is captured (not silently dropped) AND the operator has a documented path to make the document queryable under the proposed facet (`corpus taxonomy promote` + `corpus reenrich`).
3. **Given** a dropped document whose SP-003 validation gate rejects it (disallowed extension, MIME mismatch, oversized, filename sanity failure), **When** the watcher enqueues the file, **Then** the document is routed to `Paths.failed()` with a structured sidecar per SP-006 AND is NOT silently queryable AND is NOT returned by any subsequent `corpus.find` invocation.
4. **Given** a dropped document successfully classified, embedded, and indexed, **When** the operator invokes `corpus.find` from a Claude Code session AND the agent surfaces ≥ 1 returned SearchHit in its response, **Then** the operator can issue `corpus accept <request-id>` AND an `engagement.acceptance_event` is appended to `Paths.telemetry()` with `request_id`, `accepted_at`, and an optional `acceptance_note` (operator-supplied free text) AND `corpus engagement-proxy report --since=<window>` counts this as +1 acceptance event for the window.
5. **Given** the dogfood window is in progress, **When** the operator drops 10 distinct documents across PDF / Markdown / plain-text into `Paths.inbox()` over 2 days AND makes ≥ 5 `corpus.find` queries against the resulting corpus over those 2 days AND issues ≥ 1 `corpus accept` for a useful result, **Then** `corpus engagement-proxy report --since=<2d-ago>` returns PASS (≥ 5 queries, ≥ 1 acceptance event).

### User Story 2 — UR-002: ask an agent a knowledge-grounded question and receive a corpus-grounded answer with traceable references (Priority: P1)

The user (Maya / Shon-as-Maya / David in tax-attorney mode) has a working SP-007 install with documents already ingested (US1 verified for ≥ 5 representative documents). The user opens a Claude Code session and asks a question whose answer is in one or more of the ingested documents. The agent's MCP-client connects to the corpus MCP server via stdio (SP-001 FR-001), the SP-002 auto-loaded resources (corpus://manifest, corpus://taxonomy, corpus://recent) are surfaced at session start (UR-003 territory), and the SP-002 FR-009 retrieval-prompt template instructs the agent to invoke `corpus.find` when answering knowledge-grounded questions. The agent invokes `corpus.find({query: '<user-question-as-search-terms>'})`. The SP-005 hybrid retriever (BM25 + dense + graph + confidence per FR-003) returns a ranked SearchHit list within the SP-005 latency budget (NFR-003: p95 ≤ 250ms on a 5k-doc corpus). The agent reads the per-SearchHit excerpt and grounds its answer with traceable references back to the source documents (the SearchHit `uri` field dereferences via `corpus://docs/{id}` per SP-002 FR-008 to the normalized Markdown body of each source document). The user reads the answer; the answer cites the corpus documents (NOT fabricated citations). The user issues `corpus accept <request-id>` for the `corpus.find` invocation; the acceptance event is recorded.

**Why this priority**: P1 because UR-002 is the load-bearing user-value proposition that turns the corpus into a retrieval substrate the agent uses. Without UR-002, the corpus is a passive knowledge store nobody queries. P1 also because UR-002 is gated by the SP-002 FR-009 retrieval-prompt template + the SP-002 auto-loaded resources; if either is broken (per SP-006 retrospective F-1 "transport cutover gap" pattern), the agent never invokes `corpus.find` and the Maya engagement proxy ≤ 0 by definition.

**Independent Test**: On a SP-007 install with ≥ 5 ingested documents containing a known fact (e.g., a specific date, technical term, or quote that appears in exactly one document and is unique enough that the agent could not answer from training-data alone), the operator opens a fresh Claude Code session AND asks the agent "What does my corpus say about <known-fact>?". Verified by (a) the SP-005 `search.query` telemetry event firing within milliseconds of the agent's question (the agent invoked `corpus.find` per the FR-009 prompt template), (b) the returned SearchHit list contains the expected source document in the top-3, (c) the agent's natural-language response cites the document by title or by a snippet from the SearchHit excerpt, (d) the operator runs `corpus accept <request-id>` and the acceptance event records. Verified by the adversary case: same operator asks a question whose answer is NOT in the corpus (e.g., a brand-new fact post-dating every ingested document); the agent invokes `corpus.find`, the returned SearchHit list is empty (`result_count == 0`, no citations in the response surface per SP-005 + SP-006 contract), the agent answers honestly without fabricating a corpus citation, and the operator does NOT issue `corpus accept` (no false positive in the engagement metric).

**Acceptance Scenarios**:

1. **Given** a working SP-007 install with ≥ 5 ingested documents AND a Claude Code session opened with the corpus MCP server registered, **When** the operator asks the agent a question whose answer is in exactly one ingested document, **Then** within the session the agent invokes `corpus.find({query: '<terms>'})` AND a `search.query` telemetry event records the invocation AND an `engagement.corpus_find_invoked` event records the engagement-grade metadata (`query` or query_hash, `result_count`, `tier_used`, `request_id`, `timestamp`) AND the response contains ≥ 1 SearchHit whose URI dereferences to the source document AND the agent's natural-language answer cites the source by title or by an excerpt from the SearchHit (per UR-002 ACCEPTANCE-CRITERIA scenario "agent invokes corpus.find and grounds answer with traceable references").
2. **Given** a working install AND a Claude Code session, **When** the operator asks a question whose answer is NOT in the corpus, **Then** the agent invokes `corpus.find` AND the returned `result_count == 0` AND the response does NOT include any fabricated `corpus://docs/...` URIs (per UR-002 ACCEPTANCE-CRITERIA scenario "agent does not fabricate corpus citations when corpus.find returns empty") AND no `engagement.acceptance_event` is recorded (the operator does NOT run `corpus accept`).
3. **Given** an empty corpus (the daemon is running but no documents have been ingested yet, OR the operator manually drained the index), **When** the operator opens a Claude Code session AND asks the agent any knowledge-grounded question, **Then** the agent's `corpus.find` invocation returns `result_count == 0` AND the MCP response's `hits` array is empty AND the response surface contains NO citation fields, no `corpus://docs/...` URIs, no fabricated SearchHit entries (per SP-008 exit criterion `corpus_find_empty_corpus_adversary_scenario_in_execution_journal_shows_mcp_response_hits_array_empty_and_no_citation_fields`).
4. **Given** a question whose answer requires synthesis across multiple ingested documents (e.g., "compare what document A says about X with what document B says about X"), **When** the agent invokes `corpus.find({query: '<terms>'})`, **Then** the returned SearchHit list spans both source documents AND the agent's answer cites both source documents by URI or by SearchHit-excerpt (per UR-002 ACCEPTANCE-CRITERIA scenario "cross-document grounding works when SearchHits span multiple documents").
5. **Given** the SP-005 corpus contains 5,000+ documents AND the operator asks a question, **When** `corpus.find` runs, **Then** the response returns within the SP-005 NFR-003 p95 budget (≤ 250 ms) AND the `engagement.corpus_find_invoked` event records `duration_ms ≤ 250` (the engagement-proxy report optionally reports a p95 latency aggregate for the dogfood window).
6. **Given** the operator finds a returned SearchHit useful (the agent's grounded answer was accurate AND the operator confirms by inspecting the cited document), **When** the operator runs `corpus accept <request-id>` (optionally with `--note "<free-text>"`), **Then** an `engagement.acceptance_event` is appended to `Paths.telemetry()` with `request_id`, `accepted_at`, `acceptance_note?` AND `corpus engagement-proxy report` counts this as +1 acceptance event for the window.

### User Story 3 — UR-003: install once and have the corpus available across new agent sessions, with auto-loaded session state and no per-session re-feed (Priority: P1)

The user has a SP-007 install with documents ingested (US1 + US2 verified). The user CLOSES the Claude Code session (exits the IDE, kills the terminal, restarts the machine). The user opens a NEW Claude Code session — possibly hours or days later, possibly from a different terminal or a different IDE window. At session-start, Claude Code re-reads its MCP-client config (the `~/.claude.json` entry the SP-007 install registered), starts the corpus MCP server via stdio (SP-001 FR-001), and reads the auto-loaded resources at `corpus://manifest` (SP-002 FR-005 — doc count, established domains, established tags, last ingest timestamp), `corpus://taxonomy` (SP-002 FR-006 — established domains and tags with counts), `corpus://recent` (SP-002 FR-007 — recently ingested documents). The agent has the corpus context surfaced via the auto-loaded resources WITHOUT the user re-feeding any documents. The user immediately asks a knowledge-grounded question; the agent invokes `corpus.find` against the SAME corpus the previous session used. The corpus state is preserved across the session boundary. CRITICAL: opening a new session does NOT re-ingest the existing inbox contents (idempotency invariant per SP-008 exit criterion `session_start_idempotency_test_in_execution_journal_shows_sqlite_corpus_table_row_count_unchanged_after_second_session_start_with_identical_inbox`); the SQLite documents-table row count is identical before and after the second session-start.

**Why this priority**: P1 because UR-003 is the "no librarian for a model that forgets me at midnight" promise (Maya's verbatim key_quote). It is the entire raison d'être of the corpus substrate over per-session document re-feed. P1 also because UR-003 is the deepest invariant of the Maya engagement-proxy: a 7-day dogfood window REQUIRES the operator to span ≥ multiple sessions (Claude Code sessions naturally end and restart; the operator's workflow involves session boundaries). If UR-003 fails (corpus state invisible to new sessions OR new sessions re-ingest the inbox), the operator's engagement-proxy queries become per-session bootstrapping work, not retrieval. P1 also because the SP-008 exit criterion `session_start_idempotency_test_in_execution_journal_shows_sqlite_corpus_table_row_count_unchanged_after_second_session_start_with_identical_inbox` is a hard guardrail against a regression class that would silently corrupt the corpus during dogfooding.

**Independent Test**: On a SP-007 install with N documents ingested (where N is a known count, e.g., N=20), the operator records the documents-table row count: `sqlite3 ~/.local/share/llm-corpus/index.db "SELECT count(*) FROM documents"` returns N. The operator opens a Claude Code session, asks a query whose answer is in the corpus, observes the agent invoke `corpus.find` against the auto-loaded manifest/taxonomy/recent context. The operator closes the session, waits arbitrary time, opens a fresh session. The operator asks a different query whose answer is in the same corpus. The agent invokes `corpus.find` against the SAME corpus (no re-ingest). The operator re-checks the documents row count: same N. The operator verifies that the SP-007 install-receipt is unchanged (the install side-effects persist), the `~/.claude.json` MCP-server registration is unchanged, and the corpus MCP server's `corpus://manifest` response reports the same N documents from the new session. Additionally: the operator drops the SAME 20 documents into `Paths.inbox()` again (the SP-003 FR-017 content-hash idempotency invariant); each is dedup-hit'd and the documents row count stays at N. The SP-008 session-start idempotency adversary scenario is: spawn a fresh `corpus daemon` start with the SAME inbox contents present from a prior run; the `documents` row count is unchanged after the second session-start.

**Acceptance Scenarios**:

1. **Given** a working install with N documents ingested, **When** the operator closes one Claude Code session AND opens a fresh session, **Then** at the new session's start the corpus MCP server starts via stdio AND the `corpus://manifest` resource returns `doc_count == N` AND the `corpus://taxonomy` resource returns the existing established domains and tags AND the `corpus://recent` resource returns the N most-recent successful ingests AND no documents are re-ingested AND the SQLite `documents` row count is identical to before the session close (per UR-003 ACCEPTANCE-CRITERIA scenario "install once, available across new agent sessions via auto-loaded resources").
2. **Given** an install with N documents AND an inbox containing the SAME files that produced those N documents (already content-hash-idempotency-deduped per SP-003 FR-017), **When** the operator restarts the daemon (kills `corpus daemon` and starts it fresh), **Then** the SP-003 watcher re-enqueues the inbox files BUT the SP-003 content-hash idempotency check short-circuits each as a `ingest.dedup_hit` AND the documents row count remains N AND no `ingest.normalized` or `ingest.completed` events fire for those files (per UR-003 ACCEPTANCE-CRITERIA scenario "opening a new session does not duplicate corpus state" + SP-008 exit criterion `session_start_idempotency_test_in_execution_journal_shows_sqlite_corpus_table_row_count_unchanged_after_second_session_start_with_identical_inbox`).
3. **Given** an agent session attempts to invoke `corpus.find` BEFORE the corpus has been initialized (the SP-007 install has not been run, or `~/.claude.json` is missing the `mcpServers.corpus` entry, or `Paths.indexDb()` doesn't exist), **When** the session starts, **Then** the agent receives a clear unavailable error (per UR-003 ACCEPTANCE-CRITERIA scenario "agent session started before corpus init returns a clear unavailable error"). The exact surface: if the MCP-client config registers the corpus server but the binary or index is missing, the agent's `tools/list` handshake fails with a structured error citing the missing resource; if the MCP-client config does NOT register the corpus server, the agent has no `corpus.find` to invoke.
4. **Given** a corpus with documents from a prior session AND a new agent session, **When** the operator asks any question, **Then** the agent's `corpus.find` query returns from the existing index without re-classifying or re-embedding any document AND the response surface latency is at the SP-005 NFR-003 budget (not at the SP-007 90-second first-run budget — the corpus is not being initialized at session-start, it is being read).
5. **Given** an install + session sequence, **When** the operator inspects the SP-006 telemetry log over the session boundary, **Then** the per-session-start events are READ-ONLY (`resource.read`, `tools/list`, `daemon.started` if the daemon was stopped) AND ZERO `ingest.normalized` / `classify.completed` / `embed.completed` / `index.completed` / `edges.completed` events fire at session-start for an unchanged inbox.

### User Story 4 — Maya Week-1 Engagement-Proxy Gate (per C-028): instrumentation + `corpus engagement-proxy report` CLI + Track A/B split (Priority: P1)

The user (Shon-as-Maya-during-the-dogfood-window) has a SP-007 install. The operator uses the substrate naturally over a 7-day calendar window (the C-028 mitigation: "≥5 corpus.find queries in first 7 days of dogfooding + ≥1 acceptance event"). During the window: the operator drops documents into the inbox at their natural pace (no synthetic prompt to drop, no fabricated workload); the operator opens Claude Code sessions at their natural pace; the operator asks knowledge-grounded questions at their natural pace; the agent invokes `corpus.find` at the agent's natural rate (the SP-002 FR-009 prompt template biases the agent toward retrieval when the question is knowledge-grounded). Every `corpus.find` invocation emits a SP-005 `search.query` event (existing) AND an SP-008 `engagement.corpus_find_invoked` event (new, with engagement-grade fields). When the operator finds a returned SearchHit useful, the operator runs `corpus accept <request-id>` (manual, deliberate, friction-bearing — this is the operator-attested D2 acceptance-event definition per the SP-008 Clarifications block below); the command appends an `engagement.acceptance_event` event keyed by `request_id`. At the end of the 7-day window (or at any point during it for an interim check), the operator runs `corpus engagement-proxy report --since=<7d-ago>`. The CLI subcommand scans `Paths.telemetry()` for `engagement.corpus_find_invoked` and `engagement.acceptance_event` events in the named window, computes the counts, computes whether the C-028 threshold is met (`queries_in_window ≥ 5` AND `acceptance_events_in_window ≥ 1`), and prints a PASS or FAIL verdict. On FAIL with `queries_in_window < 3`, the CLI surfaces the KILL signal per SPRINT-PLAN.yaml line 253 (`week_1_engagement_proxy_below_3_queries_in_7d_after_dogfood_period`) and recommends Stage 4 recycle per C-028. On PASS, the SP-008 exit criterion `maya_persona_week_1_engagement_proxy_at_least_5_corpus_find_queries_in_7d_with_at_least_1_acceptance_event_per_c_028` is satisfied AND the sprint's PM-Review gate may proceed.

**Why this priority**: P1 because (a) C-028 is the SOLE Stage-5-gateable early-signal proxy for A-002; without it, A-002 is a 30-day post-ship black box and the v1 ship decision is uninformed; (b) the engagement-proxy is a sprint exit criterion verbatim from SPRINT-PLAN.yaml; (c) the rollback criterion (`week_1_engagement_proxy_below_3_queries_in_7d_after_dogfood_period`) is a v1 KILL signal, not a sprint retry — the gate's outcome decides whether v1 ships at all. ### Edge Cases

- **Telemetry log rotation mid-window**: The SP-003 telemetry layer rotates the NDJSON log at a size threshold. If a rotation occurs within the engagement-proxy report's `--since/--until` window, the report's CLI scans BOTH the active log and any rotated logs whose mtime falls within the window. The Track A reader is rotation-aware (uses the existing SP-003 rotation file-naming convention). If the operator has post-pruned old rotated logs (manual `rm`), the report emits a warning ("rotated logs were missing from window <X>; reported count may be a lower bound") but still emits the report — does not exit non-zero.
- **`corpus accept` for a query that returned `result_count == 0`**: The semantic intent is to attest the result was useful. A query that returned no results cannot be "useful". The CLI exits non-zero with a clear message ("cannot accept a query with zero results") and zero acceptance event is appended.
- **`corpus accept` for a query older than the current window**: The CLI does NOT enforce a window on `corpus accept`; the operator may accept any past `request_id` from the telemetry log. The window enforcement is done by `corpus engagement-proxy report`'s `--since/--until` filter, not by the accept CLI.
- **Operator drops 100 documents into the inbox mid-window**: The SP-006 watcher + idempotency + failure-lane handles natural workload variance; the engagement-proxy report counts `corpus.find` invocations, not ingest invocations. Heavy ingest does NOT inflate the engagement metric.
- **The agent invokes `corpus.find` programmatically without explicit operator intent (e.g., a stale background hook fires)**: The metric counts every `corpus.find` invocation; the SP-008 spec acknowledges this is a feature, not a bug — the agent-as-principal framing means the agent's natural invocations are real signal. The `engagement.corpus_find_invoked` event records the invocation; the operator only attests `corpus accept` for results that were actually useful. The acceptance event is the operator's quality filter.
- **`engagement.corpus_find_invoked` event size budget**: Per Constitution IX (≤ 4 KB append atomicity), the event payload is bounded. The `query` field is truncated to 1024 chars (matching the SP-003 `BoundedMessage` cap); longer queries are stored as a SHA-256 hash with a `query_truncated: true` flag and a `query_hash` field. The report aggregator can group by `query_hash` if the operator wants to see duplicate-query patterns; the primary metric is invocation count, not query uniqueness.
- **Multiple acceptance attestations for the same query**: An operator could conceptually accept a query multiple times (e.g., later notices a SearchHit they didn't initially appreciate). The CLI's duplicate-detection (US4 Acceptance Scenario 6) treats subsequent invocations as idempotent no-ops; only the first attestation per `request_id` counts toward the metric.
- **Telemetry log is corrupt or partially missing**: The report's reader uses defensive parsing (the SP-006 `failures.sidecar_parse_failed` pattern adapted): malformed lines are skipped with a `engagement.report_telemetry_parse_failed` event AND counted in the report's `parse_errors_count` field. The report still emits even if parse errors occurred; the operator decides whether to trust the result given the error count.
- **Operator forgets to run `corpus accept` during the window**: This is the FAIL-acceptance-events scenario (Acceptance Scenario 3 above). The verdict is FAIL. The KILL signal does NOT fire (queries ≥ 5 clears the ≥3-query floor). The operator has a documented remediation path: continue dogfooding, accept future queries, re-run the report — OR accept past queries by `request_id` retroactively (the accept CLI does not enforce a temporal window on the accepted query's age).
- **Operator gaming the metric**: An operator could trivially spam 5 `corpus.find` queries and 1 `corpus accept` to clear the gate. The SP-008 metric is a Stage-5 early-signal proxy, NOT a fraud-resistant adoption measurement. The trust model is "the operator is the v1 user and has no incentive to game the metric on themselves" (single-user, single-machine per Principle IV). The metric's value is signal, not enforcement. This is documented honestly in the spec per Constitution XVI.
- **Daemon was not running during part of the window**: A daemon-down period would produce zero `corpus.find` invocations during that period. The report does NOT compensate; if the window spans 7 days and the daemon was up for only 2 of them, the report reflects the actual 2-day query count. The operator's responsibility is to keep the daemon up during the dogfood window (the SP-007 `--enable-autostart` option helps here).
- **Window crosses a clock change (DST or operator manual clock adjustment)**: The telemetry events are stored with ISO-8601 timestamps in UTC offset. The window filter operates on the UTC-normalized timestamp; DST and clock changes do not affect the count.
- **`corpus engagement-proxy report` invoked while the daemon is mid-ingest**: The report scans the telemetry log (read-only file IO); it does NOT acquire `Paths.drainLock()`. Concurrent daemon writes to the log are tolerated by the SP-003 atomic-append discipline. The report may miss in-flight events that were appended after the scan began; this is acceptable for a metric-aggregation read (the operator re-runs the report later for a fresh view).
- **Report runs against a future `--until`**: If the operator passes `--until=<future-ts>`, the CLI accepts it (the window is `[since, until]` whether or not `until` is in the past). The report counts events up to "now"; future events have not occurred yet.
- **The operator wants to see per-query engagement detail, not just aggregates**: Out of scope for SP-008; the report's `--format=json` output gives the aggregate counts. Per-query telemetry detail is available via `tail -f Paths.telemetry()` or via a future `corpus engagement-proxy queries` subcommand (future-horizon).

## Clarifications

This block records the SP-008-specific decisions made during this spec-kit specify pass. Each decision binds a class of [NEEDS CLARIFICATION] markers that would otherwise live in this spec.

### Decision 1 — Acceptance-event definition: D2 (operator-attested) chosen

**Context**: The C-028 mitigation requires "≥1 acceptance event" but does not define what an acceptance event IS. The SP-008 dispatch prompt enumerated three viable definitions (D1 auto-detected, D2 operator-attested, D3 proxy/result-count).

**Decision**: SP-008 v1 uses **D2 — operator-attested** via the `corpus accept <request-id>` CLI subcommand. The operator explicitly marks `corpus.find` results that were useful; no implicit detection.

**Rationale**: D1 (auto-detected) requires cross-process visibility into the MCP client's post-response behavior — the MCP protocol does not surface client-side message-handling, and per AG-004 / OOS-011 cross-agent surfaces are out of scope for v1. D3 (proxy/result-count) conflates "got results" with "results were useful" and would inflate the metric on irrelevant returns. D2 is friction-bearing but trustworthy — the operator is the single authority on usefulness, per Principle IV (single-user, single-machine). D2 may evolve in v1.5+ as cross-agent telemetry surfaces ship; for v1 the explicit operator attestation is correct.

**This decision IS LOAD-BEARING for the spec; do not re-litigate without a /speckit-clarify pass.**

### Decision 2 — Track A / Track B split: Pallas ships code; operator produces metric

**Context**: The C-028 engagement-proxy is a 7-day behavioral metric, not a code test. SP-008 must be honest about what code can produce vs what only operator action can produce.

**Decision**: SP-008 has two tracks. **Track A (code)** ships: telemetry instrumentation (FR-ENGAGEMENT-001..004), `corpus accept` CLI (FR-ENGAGEMENT-002), `corpus engagement-proxy report` CLI (FR-ENGAGEMENT-003), UR-001/UR-002/UR-003 Gherkin + integration tests (FR-ENGAGEMENT-007..009), adversary tests (FR-ENGAGEMENT-010..011), C-046 e2e smoke (FR-ENGAGEMENT-006). **Track B (operator action)**: the 7-day dogfood window during which the operator uses the installed substrate naturally; at end-of-window the operator runs `corpus engagement-proxy report` and the verdict is captured in `RETROSPECTIVE.md` as SC-008-035 evidence.

**Rationale**: Per Constitution XVI (Validation Honesty), code shipping does NOT equal user-acceptance complete. The C-028 mitigation REQUIRES real usage. Conflating Track A and Track B would be a Goodhart's-law violation. Track A's PR can merge on Track A's tests passing; SP-008's sprint-completion gate requires Track B's verdict.

**This decision IS LOAD-BEARING.**

### Decision 3 — Engagement instrumentation surface: ≥ 3 (effectively 4) new telemetry event classes, additive to TelemetryEvent

**Context**: Where do the new engagement-grade fields live? Options: (a) extend the existing SP-005 `search.query` event with new fields; (b) add new, separate `engagement.*` event classes.

**Decision**: New, separate `engagement.*` event classes (FR-ENGAGEMENT-004). The four classes: `EngagementCorpusFindInvokedEvent`, `EngagementAcceptanceEvent`, `EngagementReportGeneratedEvent`, and the defensive `EngagementReportTelemetryParseFailedEvent`. Added as additive variants to `TelemetryEvent` discriminated union, matching the SP-007 12-class addition pattern.

**Rationale**: Extending `search.query` would couple SP-005 observability to SP-008 acceptance metrics; separation lets the engagement layer evolve independently. The SP-007 pattern (additive variants, never modify prior schemas) preserves event-class stability across sprints.

**This decision IS LOAD-BEARING.**
