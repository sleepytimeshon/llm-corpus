# Feature Specification: Hybrid Retrieval — BM25 + Dense + Graph + Confidence Fusion, Cancellable Search, Atomic Index Updates

**Feature Branch**: `005-retrieval`
**Created**: 2026-05-13
**Status**: Draft
**Input**: SP-005 from `.product/SPRINT-PLAN.yaml` / `.product/ROADMAP.yaml` RM-006 ("Hybrid retrieval: BM25 + dense + graph + confidence-weighted fusion"). Source FR text: `.product/REQUIREMENTS.yaml` FR-002 (corpus.find accepts structured query, returns ranked SearchHit list), FR-003 (corpus.find ranks results via hybrid retrieval — BM25 + dense + graph + confidence; "the four signals MUST all be inputs"), FR-004 (deterministic structured error envelope returned as a normal MCP tool response, not a transport error), FR-015 (vector-similarity ranking). Source constitutional principles: I (Local-First, No Egress — Ollama embedding adapter at `localhost:11434`; sqlite-vec runs in-process), III (Substrate, Not Surface — `corpus.find` MCP tool remains read-only; no new MCP mutation surface), V (Schema-Enforced Structured Output — SearchHit array + error envelope validated via Zod), VII (Cancellable, Bounded IO — every embedding HTTP call + every index query takes `AbortSignal` with per-call timeout), VIII (Atomic Writes & Transactional Index Updates — "Index writes (FTS5 row + docs row + sqlite-vec row for the same document) MUST commit together in a single transaction or not at all; partial index state is a forbidden permitted state"), IX (Concurrency-Safe Shared State — drain process + reindex process serialize via `Paths.drainLock()`), X (Idempotent Pipeline Transitions — embedding + indexing on already-indexed row is a no-op), XI (Library/CLI Boundary — no `process.exit` in libraries), XII (Subprocess Hygiene — embedding adapter is HTTP only, no subprocess), XIII (Telemetry-or-Die — every catch emits structured event; every search query emits `search.query` event with tier-used and result-count fields), XIV (XDG Paths via Single Resolver), XVI (Validation Honesty — ARCHITECTURE-FINAL §10.6 latency numbers are TARGETS, not guarantees; no formal retrieval-eval harness as v1 success criterion).

This is feature 005: the **retrieval owner** for the four-signal hybrid ranking surface. SP-001 shipped a placeholder `corpus.find` MCP tool that returns `{hits: [], query}` (per the `packages/transport/src/corpus-find-tool.ts` SP-001 comment "Real implementation lands in SP-005") and an `export {};` `packages/index/src/index.ts` stub. SP-002 added the read-only resource layer. SP-003 produced sentinel `documents` rows. SP-004 populated the classifier columns + body-file frontmatter mirror + proposed-state taxonomy rows. SP-005 closes the loop: a local Ollama embedding adapter (default `nomic-embed-text`, 768-dim) produces per-document dense vectors immediately AFTER classify-stage commits; an FTS5 virtual table indexes the BM25 fields (summary, tags, facet_topic, title, body_excerpt) with field-weighted scoring (ARCHITECTURE-FINAL §10.1); a sqlite-vec virtual table holds the dense vectors with cosine distance; an `edges` table materializes tag-overlap + summary-similarity + explicit-related edges (ARCHITECTURE-FINAL §10.4); a fusion module combines the four signals via Reciprocal Rank Fusion (RRF, k=60 per §10.3) with confidence weights (§10.5) applied as a multiplicative post-fusion adjustment. The `corpus.find` MCP tool now returns a ranked SearchHit list. The atomic index-update contract (Constitution VIII) holds: FTS5 row + sqlite-vec row + edges-from-this-doc commit together with the classifier UPDATE in a single SQLite transaction, OR none commit. SP-005 ships Tier 0 (hybrid) only — Tier 1/2/3 fallthrough optimization (BM25-only / grep-CATALOG / fs-grep) is explicitly deferred to SP-006. A new `corpus reindex [--dry-run]` CLI subcommand backfills existing classified documents after schema migrations or model swaps; it reuses the SP-003 drain-lock. Confidence weights are configurable in `config.toml` per §10.5 defaults. SP-005 introduces zero new MCP mutation surfaces (Constitution III): the daemon's classify-stage hook chain extends to embed + index + edges-build sub-stages, all inside the same drain-lock window.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A natural-language query against the corpus returns a ranked list of relevant documents, autonomously indexed in the same drain that ingested and classified them (Priority: P1)

The user (Maya/Shon persona) drops a file into `Paths.inbox()`. SP-003 ingests it (sentinel row). SP-004's post-persist hook fires: classifier produces the metadata; the classify-persister commits the UPDATE + body-file frontmatter rewrite. SP-005 EXTENDS the same hook chain — immediately after the classify commit (same drain-lock holder, same process), an embed-stage POSTs the document's `(title + summary + facet_topic + tags + body_excerpt)` to `http://localhost:11434/api/embeddings` with the configured embedding model (default `nomic-embed-text`), receives a 768-dim float vector, and an index-stage commits THREE rows in a single SQLite transaction: (a) the FTS5 virtual-table row for BM25 (field-weighted: summary=5, tags=3, facet_topic=2, title=2, body_excerpt=1), (b) the sqlite-vec virtual-table row for dense cosine similarity, (c) the `edges` rows newly produced by the edges-build sub-stage (tag_overlap if Jaccard ≥ 0.3 vs. any existing classified doc; summary_similarity if cosine ≥ 0.7; explicit_related verbatim from the document's frontmatter `related` array). All four sub-stages (classify → embed → index → edges-build) run inside the same `Paths.drainLock()` window — no second writer can race with this pipeline. The user then invokes the `corpus.find` MCP tool from any MCP-aware client with `{query: "vector database retrieval"}`; the tool runs four retrievers in parallel (BM25 over FTS5 top-K; dense cosine over sqlite-vec top-K; graph traversal over `edges` for query-matching seed docs; confidence weights from each result's facet_type + recency), fuses via RRF (k=60), applies post-fusion confidence multipliers, and returns the top-K ranked SearchHit list to the client. The four signals are ALL inputs — no signal is silently disabled when partially unavailable (FR-003 acceptance scenario explicitly forbids this).

**Why this priority**: This is the load-bearing stakeholder requirement for SP-005 and the entire retrieval sprint. Without it, the corpus has classified rows but no agent-facing query surface — `corpus.find` continues to return an empty list, the WHITEPAPER promise of "ask the corpus for what's relevant" is unfulfilled, and the four-month investment in SP-001 through SP-004 doesn't reach the agent's working memory. P1 because every other retrieval behavior (backfill via reindex, filter-narrowed search, partial-signal degradation handling) is a refinement on or guard around this primary autonomous path.

**Independent Test**: Initialize a clean corpus on the user's primary machine with the SP-003 daemon running (post-SP-005 — daemon now drives the four-sub-stage chain: classify → embed → index → edges-build), Ollama 0.21.0 loaded with `qwen3.5:9b` AND `nomic-embed-text` available, and a seed taxonomy. Drop a small batch of mixed-MIME documents (e.g., 5 papers + 3 markdown notes + 2 HTML articles). Without further user action, observe that each row transitions through sentinel → classified → embedded → indexed within the per-doc budget (Plan-resolved 30s interactive / 120s batch). After the batch settles, invoke `corpus.find` via an MCP client with a natural-language query (e.g., `"agent memory architectures"`). Verify: (a) the response is a structured JSON object with a `hits` array and an `query` field; (b) `hits` contains ≥ 1 SearchHit if the seed batch has relevant documents; (c) each SearchHit carries `uri` (resolvable via `corpus://docs/{id}`), `score` (numeric), `title`, `facet_domain`, `facet_type`, `tags`, `snippet` (BM25-derived); (d) the ordering reflects all four signals fused, not just one (verifiable by toggling the dense-signal off via fixture and observing measurable change in top-K).

**Acceptance Scenarios**:

1. **Given** a clean corpus with the SP-003 daemon running (post-SP-005 extension) and `nomic-embed-text` loaded in Ollama, **When** the user drops one file of any allowlisted MIME type into the inbox, **Then** within the per-document four-sub-stage budget (Plan-resolved 30s interactive / 120s batch) the row's SQL columns transition from sentinel through classified through embedded through indexed AND a new FTS5 row exists AND a new sqlite-vec row exists AND zero-or-more `edges` rows exist with this doc as `src_id` AND the body file's YAML frontmatter is unchanged from SP-004 (SP-005 writes index tables; it does NOT touch body-file frontmatter).
2. **Given** a freshly-indexed row, **When** the user invokes `corpus.find` with `{query: <natural-language>}`, **Then** the response is a structured JSON object `{hits: SearchHit[], query: string}` AND `hits` is ordered by descending fused score AND each hit carries `uri`, `score`, `title`, `facet_domain`, `facet_type`, `tags`, `snippet` AND the `uri` field is `corpus://docs/<doc-id>` (resolvable via the SP-002 read path).
3. **Given** a query that matches strongly on BM25 but weakly on dense-vector similarity (e.g., a literal rare-term lookup), **When** the four retrievers run, **Then** the BM25 retriever surfaces the strong matches AND the dense retriever surfaces semantically-close near-matches AND RRF fusion (k=60) combines them AND the top-K result list contains both kinds of matches in a sensible order.
4. **Given** sqlite-vec is loaded but the embedding model `nomic-embed-text` is temporarily unavailable in Ollama, **When** a query runs, **Then** the dense retriever fails with `EmbeddingUnavailableError` AND the BM25 + graph + confidence retrievers still return their results AND fusion combines the available signals AND the response carries a `search.degraded` telemetry event (NOT a transport-level error) AND the SearchHit response includes a `degraded_signals: ['dense']` annotation. The dense-vector failure does NOT silently disable ranking (FR-003 explicit acceptance scenario).
5. **Given** the search succeeds, **When** telemetry is inspected, **Then** the `search.query` event class appears with `query_hash` (NEVER the raw query text — Constitution I body-content prohibition generalizes), `tier_used='hybrid'`, `result_count`, `duration_ms`, `signals_used: ['bm25','dense','graph','confidence']` fields.

### User Story 2 — The user backfills the index for existing classified documents via `corpus reindex` after a schema migration or embedding-model swap (Priority: P1)

A pre-SP-005 corpus has classified rows but no FTS5 / sqlite-vec / edges entries. After upgrading to SP-005 (or after changing `[embedding].model` in `config.toml`, or after a future schema-version bump invalidates existing embeddings), the user runs `corpus reindex` from the CLI. The command acquires `Paths.drainLock()` (single serialization point per Principle IX — concurrent SP-003 ingest, SP-004 reenrich, and SP-005 reindex serialize via the same lock), selects every successfully-classified row (`WHERE facet_type != 'unclassified' AND status='success'`), and re-runs the embed + index + edges-build sub-stages for each. The classify-stage is NOT re-run (already-classified rows keep their existing classifier output — Constitution X idempotency). Progress is emitted to stderr under interactivePolicy. On completion, the CLI exits 0 with a structured summary line: `indexed=N, failed=M, skipped=K`.

**Why this priority**: P1 because the backfill scenario is real (any user landing on SP-005 from SP-004 has it) and because the manual-trigger path is the only way to re-embed after a model change until a future automated migration-detector lands. The CLI command is also the canonical surface for users who deliberately re-index a subset (`corpus reindex --since <date>` is out of scope for v1 — the v1 surface is whole-corpus only). The `--dry-run` mode lets the user inspect the set of rows that WOULD be re-indexed without making Ollama calls or SQL writes.

**Independent Test**: Start with a corpus containing 30 classified rows (post-SP-004 state) where the `documents_fts`, `documents_vec`, and `edges` tables are EMPTY (simulated by deleting the rows post-classify — represents a pre-SP-005 install). Run `corpus reindex`. Observe: stderr shows progress under interactivePolicy (one line per doc, e.g., `[1/30] doc-ab12cd34 ... embedded + indexed in 2.4s`); the command acquires `Paths.drainLock()` (a concurrent `corpus drain` or `corpus reenrich` invocation emits `pipeline.lock_contention` and exits 0); on completion, the exit code is 0 and the summary line names exact counts. Run `SELECT COUNT(*) FROM documents_fts`, `SELECT COUNT(*) FROM documents_vec`, `SELECT COUNT(*) FROM edges` — each is ≥ 30 (edges count depends on tag-overlap / summary-similarity matches). Run `corpus.find` with a natural-language query and observe ranked results.

**Acceptance Scenarios**:

1. **Given** a corpus with N ≥ 1 classified-but-unindexed rows (rows where `facet_type != 'unclassified'` and `documents_fts` / `documents_vec` are missing the corresponding entries), **When** the user runs `corpus reindex`, **Then** the command acquires `Paths.drainLock()` with `flock(LOCK_EX | LOCK_NB)` AND iterates the target rows AND invokes the embed + index + edges-build sub-stages (the classify-stage is NOT re-invoked) AND exits 0 on completion with a summary line `indexed=N, failed=M, skipped=K` to stdout.
2. **Given** a concurrent `corpus drain` (SP-003 batch ingest), `corpus reenrich` (SP-004 backlog drain), or `corpus daemon` (SP-003 watcher) is already holding the drain-lock, **When** `corpus reindex` is invoked, **Then** the command observes lock contention AND emits a `pipeline.lock_contention` telemetry event AND exits 0 cleanly (the FR-INGEST-011 / FR-CLASSIFY-015 single-serialization-point contract preserved).
3. **Given** a row whose `documents_fts` AND `documents_vec` AND edges-from-this-doc entries already exist, **When** `corpus reindex` iterates and reaches it, **Then** the row is skipped without invoking Ollama (the per-row idempotency check at SQL level: `WHERE NOT EXISTS (SELECT 1 FROM documents_vec WHERE doc_id = documents.id)`) — Constitution X.
4. **Given** the user runs `corpus reindex --dry-run`, **When** the command executes, **Then** it lists the candidate rows AND issues ZERO Ollama HTTP calls AND issues ZERO SQL writes to the FTS5 / sqlite-vec / edges tables AND exits 0.
5. **Given** a SIGTERM during `corpus reindex`, **When** the signal is delivered, **Then** the in-flight sub-stage is aborted via `controller.abort()` AND the in-flight row's partial index state rolls back (transactional Constitution VIII) AND the command exits within 2 seconds with a non-zero status AND a `pipeline.aborted` telemetry event is emitted with `stage='reindex'`.

### User Story 3 — The user narrows search results via filter facets in the `corpus.find` query (Priority: P2)

The user invokes `corpus.find` with a structured input that includes filter facets: `{query: "retrieval", filters: {facet_domain: "agent-systems", facet_type: ["tutorial", "reference"], since: "2026-01-01"}}`. SP-005's retriever applies the filters BEFORE fusion (per-retriever filter pushdown into the SQL `WHERE` clauses), then runs the four-signal fusion against the filtered candidate set. The response carries only documents matching all filter facets, ranked by fused score. Filters with empty / null values are no-ops (don't constrain the search). Filters MUST be in a closed vocabulary (filter keys are typed via the SearchInput Zod schema; unknown filter keys produce a `validation_error` envelope per FR-004).

**Why this priority**: P2 because filter-narrowed search is genuinely useful (agent workflows often want "tutorials about X published in the last 30 days") but the load-bearing P1 surface — ranked open search — is the primary deliverable. P2 ships in the same sprint because filter pushdown is trivial-to-implement once the retrievers exist; deferring it would require a churning re-touch of every retriever in SP-006.

**Independent Test**: With the SP-005 baseline (US1 path verified end-to-end), invoke `corpus.find` with `{query: "memory", filters: {facet_type: ["tutorial"]}}` against a corpus containing tutorials AND reference docs about memory. Verify: response includes ONLY documents with `facet_type='tutorial'`; reference docs are excluded; ordering within the filtered set reflects the four-signal fusion. Repeat with `{query: "memory", filters: {since: "2026-04-01"}}`; verify response includes only documents with `ingest_timestamp >= '2026-04-01'`. Repeat with an unknown filter key: response is an error envelope `{error_code: 'validation_error', message: ..., hint: ...}` returned as a successful MCP tool response (NOT a transport error per FR-004).

**Acceptance Scenarios**:

1. **Given** the user invokes `corpus.find` with `filters.facet_domain` set, **When** the query runs, **Then** each retriever's underlying SQL adds `AND documents.facet_domain = ?` AND fusion runs on the filtered candidates AND the response contains only matching documents AND the response carries a `filters_applied: {facet_domain: ...}` annotation.
2. **Given** the user invokes `corpus.find` with `filters.facet_type: ['tutorial', 'reference']` (a list), **When** the query runs, **Then** the SQL adds `AND documents.facet_type IN (?, ?)` AND the response contains only `tutorial` and `reference` documents.
3. **Given** the user invokes `corpus.find` with `filters.since: '2026-04-01'`, **When** the query runs, **Then** the SQL adds `AND documents.ingest_timestamp >= ?` AND the response contains only documents ingested on-or-after the cutoff.
4. **Given** the user invokes `corpus.find` with an unknown filter key (e.g., `filters.zzz_unknown: 'foo'`), **When** the Zod input validation runs, **Then** the response is a `{error_code: 'validation_error', message: '...', hint: '...'}` envelope as a successful MCP tool response (NOT a transport error) AND a `search.error` telemetry event is emitted with `error_code='validation_error'`.
5. **Given** the user invokes `corpus.find` with `filters: {}` (empty filter object), **When** the query runs, **Then** the SQL adds NO filter clauses AND the response is the unfiltered ranked list (equivalent to omitting `filters` entirely).

### Edge Cases

- **Empty index — first query before any document is indexed**: `corpus.find` against a corpus with zero indexed documents returns `{hits: [], query: <echo>, signals_used: ['bm25','dense','graph','confidence'], result_count: 0, tier_used: 'hybrid'}` as a successful response. A `search.query` event is emitted with `result_count=0`. NO error envelope; NO transport-level error. The four retrievers each return empty sets and fusion produces an empty ranked list.
- **Single-signal match — query matches ONLY one of the four signals**: A literal-rare-term query may have zero dense matches (no semantic similarity to anything) but strong BM25 matches. The fusion handles this — RRF combines whatever signal-rank lists the retrievers produced; an empty list from one retriever contributes nothing to fusion but does NOT disqualify the response (FR-003 acceptance scenario "ranking handles a query matching only one of the four signals").
- **Embedding model offline at query time**: The dense retriever needs to embed the QUERY (not the documents — documents are pre-embedded). If `nomic-embed-text` is unavailable when a query arrives, the dense retriever fails with `EmbeddingUnavailableError`; BM25 + graph + confidence still produce results; fusion produces a degraded-but-non-empty ranked list; the response includes `degraded_signals: ['dense']` and a `search.degraded` telemetry event fires. The user is informed; the response is honest (Constitution XVI). The query is NOT silently downgraded to BM25-only (which would violate FR-003).
- **FTS5 table corrupted / missing**: A corrupt FTS5 virtual table is rare but possible (manual SQL surgery, disk corruption). The BM25 retriever fails with `IndexUnavailableError`; the other three retrievers still produce results; fusion proceeds with the available signals; the response includes `degraded_signals: ['bm25']` and a `search.degraded` telemetry event. The user is informed.
- **Edges-builder timeout**: The edges-build sub-stage materializes O(N) edges per new document (Jaccard tag-overlap + summary-similarity cosine against existing classified docs). For a corpus of ~10k documents this is sub-second. For pathological corpora (very long tag lists; very dense summary-similarity graph), the per-doc edges-build may exceed a per-stage budget; the AbortController fires, the partial edges set is rolled back via Constitution VIII transaction, and a `pipeline.aborted` event is emitted with `stage='edges-build'`. The document's documents_fts + documents_vec entries are also rolled back (single transaction). The row goes to the failure lane with `error_code='edges_build_timeout'` for the next reindex / drain.
- **Concurrent reindex + classify**: The drain-lock at `Paths.drainLock()` serializes them. A second writer observes `pipeline.lock_contention` and exits 0 (matching the SP-003 FR-INGEST-011 + SP-004 FR-CLASSIFY-015 contract).
- **Schema migration on populated DB**: SP-005 introduces THREE new tables (FTS5 virtual `documents_fts`, vec0 virtual `documents_vec`, plain `edges`). The schema migration uses `CREATE VIRTUAL TABLE IF NOT EXISTS` (for FTS5 and vec0) and `CREATE TABLE IF NOT EXISTS` (for edges). Existing data in `documents` and `taxonomy_terms` is preserved verbatim. On a populated DB upgrading from SP-004, the migration is idempotent: re-running creates nothing new. After migration, the existing classified rows have NO FTS5/vec/edges entries until `corpus reindex` runs (or the daemon's classify-stage hook chain processes a new ingest).
- **Very long query**: A query string of 10k+ chars (pathological input) is rejected by Zod validation with `validation_error` envelope (Plan-resolved max query length: 2048 chars). The dense retriever's embedding call would otherwise consume disproportionate latency.
- **Query with no matches at all**: All four retrievers return empty sets. Response: `{hits: [], query: <echo>, result_count: 0, signals_used: [...]}`. A `search.query` event with `result_count=0` fires. NO error envelope.
- **Query that matches only via filter**: e.g., `{query: "", filters: {facet_domain: "agent-systems"}}`. The retrievers degrade: BM25 against empty query has no terms; dense against empty query produces a degenerate embedding; the filter+confidence pathway still produces results. SP-005's contract: empty query is permitted (Zod allows `query: ''`); when query is empty, fusion falls back to filter-and-recency-only ordering (the confidence retriever's confidence × recency multiplier becomes the sole signal). Response is honest with `signals_used: ['confidence']`.
- **Two retrievers both fail simultaneously**: e.g., dense + graph both unavailable. The two surviving retrievers (BM25 + confidence) still produce a fused list. Response carries `degraded_signals: ['dense', 'graph']`. A `search.degraded` event fires. Only when ALL FOUR retrievers fail does the response become `search.error` with `error_code='all_signals_failed'` — extremely rare structural event (would mean the SQLite file itself is unreadable).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-RETRIEVAL-001 — `corpus.find` MCP tool surfaces a structured query interface and returns a ranked SearchHit list.** The `corpus.find` MCP tool registered by SP-001 (currently returning empty hits per the `packages/transport/src/corpus-find-tool.ts` placeholder) is EXPANDED in SP-005. The tool surface remains FR-001 (read-only MCP tool; no mutation surface added). The input schema accepts `{query: string, filters?: SearchFilters, limit?: number}` where `SearchFilters` is a closed-vocabulary Zod object (`facet_domain?: string`, `facet_type?: string | string[]`, `tags?: string[]`, `since?: ISO8601`, `until?: ISO8601`, `source_type?: string`). The output schema is `{hits: SearchHit[], query: string, result_count: number, tier_used: 'hybrid', signals_used: string[], degraded_signals?: string[], filters_applied?: SearchFilters}`. The tool's input/output are Zod-validated; invalid input returns the FR-RETRIEVAL-004 error envelope. *(Sources: FR-001, FR-002; Principles III, V.)*

- **FR-RETRIEVAL-002 — `corpus.find` ranks results via four-signal hybrid retrieval — BM25 + dense + graph + confidence — all four signals MUST be inputs to ranking.** The hybrid retriever runs FOUR parallel retrievers — (a) BM25 over the FTS5 virtual table with field weights (summary=5, tags=3, facet_topic=2, title=2, body_excerpt=1) per ARCHITECTURE-FINAL §10.1, (b) dense cosine similarity over the sqlite-vec virtual table using the query's nomic-embed-text 768-dim embedding (§10.2), (c) graph traversal over the `edges` table for documents reachable from seed candidates via tag_overlap / summary_similarity / explicit_related edges (§10.4), (d) confidence weights from each candidate's facet_type + recency (§10.5 defaults: research-paper=1.20, manual=1.10, form=1.10, reference=1.10, article=1.00, notes=0.95, transcript=0.90, podcast=0.90, video=0.90, book=1.05; recency +0.05 if ingested last 90 days; -0.10 if older than current schema-version-bump). Fusion is Reciprocal Rank Fusion with k=60 per §10.3 — `score(doc) = sum over retrievers r of 1/(k + rank_r(doc))` — then multiplied by confidence weight (§10.5 applied AFTER RRF fusion). The four signals MUST ALL be inputs — no signal is silently disabled when partially unavailable. *(Sources: FR-003 — "the four signals MUST all be inputs"; ARCHITECTURE-FINAL §10.)*

- **FR-RETRIEVAL-003 — Disabling any single ranking signal measurably changes top-K results.** Per FR-003 acceptance scenario "disabling any single ranking signal measurably changes top-K results", the implementation MUST be testable by toggling any one of the four retrievers off and observing the top-K result list change in a measurable way. This is verified by the SC-RETRIEVAL-005 test surface — a fixture-driven test that runs the same query four times, each time disabling one signal, and asserts the top-K result list is materially different across the runs. *(Sources: FR-003 acceptance scenarios.)*

- **FR-RETRIEVAL-004 — `corpus.find` returns deterministic structured error envelopes as successful MCP tool responses on every failure path.** Every non-success outcome returns `{error_code: string, message: string, hint: string}` as a SUCCESSFUL MCP tool response — NOT as a transport-level error. The `error_code` is a closed enum: `validation_error` (invalid input shape or values), `embedding_unavailable` (Ollama embedding endpoint unreachable AND no graceful degradation possible — only fires when EVERY retriever has failed), `index_unavailable` (SQLite index file unreadable), `query_aborted` (caller's AbortSignal fired before the response was ready), `all_signals_failed` (every retriever errored), `internal_error` (uncaught exception). The envelope shape is Zod-validated. Successful responses do NOT carry the envelope; partial-degradation responses (≥ 1 signal succeeded) carry `degraded_signals` annotation but remain success-shaped (SearchHit list). *(Sources: FR-004; Principle V.)*

- **FR-RETRIEVAL-005 — Vector-similarity ranking via cosine distance against per-document embeddings.** Per-document embeddings are produced over the concatenated text `(title + summary + facet_topic + tags-joined + body_excerpt)` (ARCHITECTURE-FINAL §10.2). Embedding dimension is determined by the configured model (default `nomic-embed-text` = 768-dim). At query time, the query string is embedded via the same model; cosine distance via `vec_distance_cosine` is the SQL primitive. Top-K per-retriever K is 64 (Plan-resolved; configurable). Document granularity = single embedding per document (no chunking in v1.0.0; chunked embedding is a future ADR). *(Sources: FR-015; ARCHITECTURE-FINAL §10.2.)*

- **FR-RETRIEVAL-006 — Embedding generation runs in-process as a sub-stage AFTER classify-stage commits, inside the same `Paths.drainLock()` window.** The SP-003 daemon's hook chain extends from `ingest → classify → embed → index → edges-build`. All four post-ingest sub-stages run inside the daemon's existing drain-lock acquisition (the lock is acquired ONCE per drain; the four sub-stages each execute serially while the lock is held). The embed-stage POSTs `(title + summary + facet_topic + tags-joined + body_excerpt)` to `http://localhost:11434/api/embeddings` via `undici` with the AbortSignal end-to-end. The response is a `{embedding: number[]}` shape (Ollama's legacy single-prompt embedding endpoint). The vector is validated (length matches the configured dimension; all entries are finite numbers) before being passed to the index-stage. *(Sources: Principles I, VII, IX; ARCHITECTURE-FINAL §10.2; Decision D in research.md.)*

- **FR-RETRIEVAL-007 — Index-stage commits FTS5 row + sqlite-vec row + edges-from-this-doc in a single SQLite transaction (Constitution VIII atomic index contract).** Per Constitution VIII: "Index writes (FTS5 row + docs row + sqlite-vec row for the same document) MUST commit together in a single transaction or not at all; partial index state is a forbidden permitted state." SP-005 implements this contract for the three SP-005-owned index writes. **Transaction-scope (R6, plan.md):** SP-004's classify-persister is a self-contained transaction merged in PR #13 with COMMIT-then-rename ordering. SP-005 does NOT reopen the classify transaction; the retrieval-orchestrator opens a SEPARATE SQLite transaction AFTER classify-persister returns: `BEGIN IMMEDIATE → INSERT documents_fts (BM25) → INSERT documents_vec (vec) → INSERT edges (zero-or-more, edges-build) → COMMIT`. The Constitution VIII atomicity unit is the three index writes; the classify UPDATE is NOT in this transaction (it's in SP-004's already-committed transaction). If embed/index/edges fail post-classify, the document row stays classified but un-indexed; recovery via `corpus reindex` re-runs the embed → index → edges chain (idempotent: same body + same model = same vector). On any failure between SP-005's BEGIN and COMMIT: ROLLBACK; row returns to classified-but-unindexed state; an `<doc-id>.error.json` sidecar with appropriate `error_code` is written at `Paths.failed()`. *(Sources: Constitution VIII verbatim; plan.md R6; Decision F in research.md.)*

- **FR-RETRIEVAL-008 — Edges materialization triggers AFTER classify and embed sub-stages, computes Jaccard-tag-overlap + summary-cosine-similarity + frontmatter-explicit-related edges, and persists into the `edges` table inside the same transaction.** The edges-build sub-stage runs AFTER embed (it needs the new doc's embedding to compute summary-similarity vs. existing docs). For the new document `D_new`: (a) for every existing classified doc `D_i`, compute `tag_jaccard(D_new.tags, D_i.tags)`; if ≥ 0.3 (configurable threshold), insert edge `(D_new.id, D_i.id, 'tag_overlap', tag_jaccard)`; (b) for every existing classified doc `D_i`, compute `cosine_similarity(D_new.embedding, D_i.embedding)` via `vec_distance_cosine`; if ≥ 0.7 (configurable threshold), insert edge `(D_new.id, D_i.id, 'summary_similarity', cosine)`; (c) for every entry in `D_new.frontmatter.related` array (if present), insert edge `(D_new.id, entry, 'explicit_related', 1.0)` unconditionally. Edges are directional (`src_id`, `dst_id`) but for `tag_overlap` and `summary_similarity` the relationship is symmetric — SP-005 inserts ONLY `(D_new, D_i)` edges, NOT the reverse; queries on `edges` JOIN both directions. The edges-build is O(N) per new doc — O(N²) cumulative — acceptable for N ≤ 10k (Plan-resolved bound; future HNSW/LSH sprint if needed). *(Sources: ARCHITECTURE-FINAL §10.4; Decisions E in research.md.)*

- **FR-RETRIEVAL-009 — Confidence weights are configurable in `config.toml` with the §10.5 defaults, applied as a multiplicative post-fusion adjustment.** The default confidence-weight map (per ARCHITECTURE-FINAL §10.5 verbatim): `research-paper=1.20, manual=1.10, form=1.10, reference=1.10, article=1.00, notes=0.95, transcript=0.90, podcast=0.90, video=0.90, book=1.05`. Plus a recency adjustment: `+0.05` for documents whose `ingest_timestamp` is within the last 90 days; `-0.10` for documents older than the current schema-version-bump cutoff. The weights are applied as `final_score = rrf_score × confidence_weight × recency_adjustment`. The defaults are overridable via `config.toml [confidence_weights]` with each facet_type as a key. Unknown facet_type values get a default weight of 1.0. *(Sources: ARCHITECTURE-FINAL §10.5; Decision G in research.md.)*

- **FR-RETRIEVAL-010 — Tier-fallthrough optimization (Tier 1/2/3) is EXPLICITLY OUT OF SCOPE for SP-005 (deferred to SP-006).** Per ARCHITECTURE-FINAL §10.6: Tier 0 (hybrid) is SP-005 scope; Tier 1 (BM25-only when dense fails AND sub-20ms target), Tier 2 (grep-CATALOG when SQLite fails), Tier 3 (fs-grep when everything fails) are SP-006 scope. SP-005's `tier_used` field is hardcoded to `'hybrid'`. Partial-signal degradation within Tier 0 (FR-RETRIEVAL-004's `degraded_signals` annotation) is NOT tier-fallthrough — it's transparent reporting that some Tier 0 retrievers failed, with Tier 0 fusion still running on the available signals. - **FR-RETRIEVAL-011 — Cancellable, bounded IO: every embedding HTTP call + every retriever's SQL query takes `AbortSignal` and respects per-call timeouts.** Per Constitution VII: every IO operation accepts `AbortSignal` and propagates it end-to-end. Per-call timeouts via `AbortController + setTimeout(() => controller.abort('per_call_timeout'), perCallTimeoutMs)` with `clearTimeout` on success (NEVER `Promise.race(setTimeout)`; Constitution VII forbidden pattern). Defaults: embedding HTTP call 10s; each retriever's SQL query 5s; whole `corpus.find` budget 30s p99. Caller-supplied AbortSignal MUST be respected — if the MCP transport's `signal` fires (caller cancelled), all four retrievers abort within 2s and the response is `error_code='query_aborted'`. *(Sources: Principle VII; Decision I in research.md.)*

- **FR-RETRIEVAL-012 — Idempotency: re-running embed + index + edges-build on an already-indexed row is a no-op.** Per Constitution X: idempotent transitions. The index-stage queries `WHERE NOT EXISTS (SELECT 1 FROM documents_vec WHERE doc_id = ?)` before invoking the embedding adapter; if the entry exists, the sub-stage exits early as a no-op. The `corpus reindex` CLI batches with the same idempotency at SQL level. `corpus reindex` on a fully-indexed corpus produces `indexed=0, failed=0, skipped=N` and issues ZERO Ollama HTTP calls. Re-running after a model swap is NOT a no-op — the model-name fingerprint changes; future ADR may extend with model-fingerprint-stored embeddings to detect this; for v1, a manual `DELETE FROM documents_vec` + reindex is the operator's path. *(Sources: Principle X; Decision J in research.md.)*

### Key Entities *(include if feature involves data)*

- **SearchInput**: the Zod-typed input shape for `corpus.find`. Fields: `query: z.string().max(2048)`, `filters: SearchFilters.optional()`, `limit: z.number().int().min(1).max(100).default(20)`. SearchFilters is a closed-vocabulary record: `{facet_domain?: string, facet_type?: string | string[], tags?: string[], since?: ISO8601, until?: ISO8601, source_type?: string}`. Strict mode — unknown keys produce `validation_error`.
- **SearchHit**: the Zod-typed shape of each entry in the response's `hits` array. Fields: `uri: string` (always `corpus://docs/<doc-id>`), `score: number` (the fused + confidence-adjusted score), `title: string`, `facet_domain: string`, `facet_type: enum`, `tags: string[]`, `snippet: string` (BM25-derived; first 200 chars of the body_excerpt with highlight markers). Validated via the `SearchHitZodSchema` exported from `packages/contracts/src/search-schemas.ts`.
- **RankingSignal**: an internal-only record per retriever output. Fields: `kind: 'bm25' | 'dense' | 'graph' | 'confidence'`, `results: {doc_id: string, rank: number, score: number}[]`, `succeeded: boolean`, `error?: string`. The fusion module consumes RankingSignal[] and produces SearchHit[].
- **EdgeRecord**: a row in the `edges` table. Fields: `src_id: string` (NEW doc), `dst_id: string` (existing doc), `kind: 'tag_overlap' | 'summary_similarity' | 'explicit_related'`, `weight: number` (Jaccard for tag_overlap; cosine for summary_similarity; 1.0 for explicit_related). PRIMARY KEY (src_id, dst_id, kind) — duplicate inserts are conflict-no-ops.
- **ConfidenceWeights**: the configurable record loaded from `config.toml [confidence_weights]` at module-load time. Defaults per ARCHITECTURE-FINAL §10.5. Type: `Record<facet_type, number>` plus `recency_boost_days: number = 90`, `recency_boost_weight: number = 0.05`, `schema_bump_penalty_weight: number = -0.10`.
- **SearchTelemetryEvent**: one of the FR-RETRIEVAL-013 event classes. Validated against the existing `TelemetryEvent` Zod discriminated union (additive variants).
- **SearchErrorEnvelope**: the structured error response from FR-RETRIEVAL-004. Fields: `error_code: enum`, `message: string (max 1024)`, `hint: string (max 1024)`. Wrapped in the SearchHit response shape's metadata; returned as a successful MCP tool response.
- **EmbeddingVector**: the in-memory representation of a per-document or per-query embedding. Type: `Float32Array` of length matching the configured embedding dimension (768 for nomic-embed-text). Validated for finite-numbers and correct length before sqlite-vec INSERT or vec_distance_cosine query.

- **SC-RETRIEVAL-022 — Schema migration idempotency.** Running the SP-005 schema migration twice on the same database produces zero side effects on the second run — every `CREATE VIRTUAL TABLE IF NOT EXISTS` and `CREATE TABLE IF NOT EXISTS` is a no-op when the target table exists with the same shape. (Constitution X.)

The configured model's dimension MUST match the `documents_vec` virtual table's `float[N]` schema; v1 hardcodes N=768; future ADR may allow dimension change with migration.
- **`Paths.*` resolver keys**: SP-005 reuses existing getters only — `Paths.indexDb()`, `Paths.failed()`, `Paths.telemetry()`, `Paths.drainLock()`, `Paths.cache()`. NO new XDG base. (Confirmed against `packages/contracts/src/paths.ts`.)

SP-005 consumes `parseMarkdownWithFrontmatter` at index-stage to extract `summary` and `facet_topic` fields; SP-005 does NOT write to body files (only the classifier does in SP-004).
- **`undici` HTTP client**: Existing SP-001 transport-layer dependency; used by the SP-001 egress hook surface AND the SP-004 OllamaAdapter. SP-005's embedding HTTP POST goes through `undici` for AbortSignal compatibility and Principle I egress-hook integration.
- **`sqlite-vec`**: Existing dependency in v1 native-addon allowlist. Loaded once per database connection via `sqliteVec.load(db)` at connection-open time. Provides `vec0` virtual table and `vec_distance_cosine` SQL function.

- **No `process.exit` in any SP-005 library code**: Per Principle XI, all SP-005 source under `packages/index/`, `packages/inference/`, `packages/pipeline/`, `packages/storage/`, `packages/transport/` MUST NOT exit the process. Only `packages/cli/` and `packages/daemon/` may.

## Out of Scope (deferred to other features or explicitly excluded)

- **Tier 1 / Tier 2 / Tier 3 fallthrough optimization** — explicitly deferred to SP-006 per ARCHITECTURE-FINAL §10.6. SP-005 ships Tier 0 (hybrid) only. The `tier_used` field in the SearchHit response is hardcoded to `'hybrid'` in SP-005; SP-006 adds the state-machine that may set it to `'bm25-only'`, `'grep-catalog'`, or `'fs-grep'` on tier-fallthrough.
- **Kill-9 (SIGKILL) cross-stage survival with named recovery checkpoints** — SP-006. SP-005 ships single-transaction atomicity for the embed → index → edges-build chain (Constitution VIII), but cross-stage recovery (resume an aborted edges-build from the partial-edges checkpoint) is SP-006 territory.
- **`corpus://failures` MCP resource** — SP-006. SP-005 writes `<doc-id>.error.json` sidecars consistent with SP-003 / SP-004; the MCP resource exposing them is SP-006.
- **Formal retrieval evaluation harness (NDCG, MRR, recall@K)** — explicitly out of scope per Constitution XVI / NFR-009. v1.0.0 ships no labeled-query benchmark; quality is operator-judged. Future-horizon sprint (v1.5+) introduces the eval harness.
- **Chunked-document embeddings (sub-document granularity)** — out of scope. SP-005 ships per-document embeddings (single vector per doc) per ARCHITECTURE-FINAL §10.2. A future ADR introduces chunked embeddings if quality data demands.
- **Re-ranking with a cross-encoder model** — out of scope. SP-005 ships RRF fusion only. Future-horizon if quality data demands.
- **HNSW or LSH for approximate nearest-neighbor edge materialization** — out of scope. SP-005 ships O(N²) cumulative edge materialization (O(N) per new doc), acceptable for N ≤ 10k per Plan-resolved bound. Future sprint introduces ANN if corpus grows.
- **Worker-pool parallelism for embedding** — out of scope. SP-005 ships single-threaded per-doc embed + index + edges-build, mirroring SP-004 FR-CLASSIFY-019.
- **Auto-detection of embedding-model change** — out of scope. v1 stores embeddings without a model-name fingerprint; switching models requires manual `DELETE FROM documents_vec` + `corpus reindex`. Future ADR may add fingerprint-stored embeddings.
- **Multi-model embedding ensembling** — out of scope. SP-005 invokes one embedding model per call (config-driven default; switchable via config).
- **MCP mutation surfaces of any kind** — FORBIDDEN by Principle III. SP-005 expands the EXISTING `corpus.find` tool's handler; introduces no new agent-facing surfaces.
- **`corpus reindex --since <date>` subset reindexing** — out of scope. v1 surface is whole-corpus reindex only. Operator manually filters via SQL UPDATE → `corpus reindex` if needed.
- **Cross-agent compatibility claims** — FR-022 / FR-023 / NFR-007 / NFR-008; should-priority next-horizon; non-goal for v1 per Principle XVI.
- **User-facing review UI for low-confidence rankings** — out of scope. Confidence weights are config-driven; a future user-review sprint may surface low-score rankings for re-review.
- **Embedding cache** — out of scope. SP-005 stores embeddings in `documents_vec` only; no separate cache. Embedding is computed once per document at index time and reused; re-embedding requires explicit `corpus reindex`.
- **Cross-corpus / federated search** — out of scope per Constitution IV (single-user, single-machine).

This feature delivers the retrieval owner for SP-004 classified rows: a local-Ollama-driven embedding adapter, a four-signal hybrid retriever (BM25 + dense + graph + confidence) with RRF fusion and confidence-weighted post-adjustment, an atomic index-update contract literally implementing Constitution VIII, and the `corpus.find` MCP tool's first real ranking implementation. SP-005 is honest about what it produces (three new tables, four-signal hybrid retrieval, Tier 0 only) and what it defers (Tier 1/2/3 fallthrough to SP-006, kill-9 cross-stage recovery to SP-006, retrieval-eval harness to v1.5+, chunked embeddings + ANN + cross-encoder re-ranking to future horizons). The agent, as principal user of the corpus, finally has a real query surface — `corpus.find "what do I know about X"` returns a ranked, signal-fused, confidence-weighted answer.
