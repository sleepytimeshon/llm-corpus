## Constitution Compliance (cross-cutting, non-template)

- [x] **Principle I (Local-First, No Egress)** — SC-007-026 + FR-INSTALL-003: the install introduces ZERO new outbound non-loopback endpoints. The only outbound call during the entire install flow is the FR-INSTALL-003 preflight Ollama-reachability check against `http://127.0.0.1:11434/api/tags` (loopback-only). The `npx` package download happens at the npm-registry layer BEFORE `corpus init` ever runs and is therefore outside NFR-002's scope per ADR-001 §"Origin of these alternatives". Runtime telemetry never crosses non-loopback.
- [x] **Principle II (User Curates, LLM Classifies Metadata)** — SP-007 introduces ZERO new LLM body-generation. The curated taxonomy seed (FR-INSTALL-008) is human-curated by Shon at SP-007 plan-stage; the install bulk-inserts it. The `corpus taxonomy promote` CLI is operator-curated (operator chooses which proposed terms to elevate). No LLM-generated body content is written.
- [x] **Principle III (Substrate, Not Surface)** — SC-007-027 + FR-INSTALL-023 commit to ZERO new MCP mutation surfaces. The new CLI subcommands (`corpus init`, `corpus uninstall`, `corpus taxonomy promote`) are CLI-only, NOT MCP-exposed. The existing SP-002 four resources + SP-006 `corpus://failures` are read-only and unchanged. SP-007 changes `corpus.find`'s discoverability surface only (adds the registration entry to the MCP-client config; does NOT change the tool itself).
- [x] **Principle IV (Knowledge, Not Memory; Single-User, Single-Machine)** — Assumptions explicit: one corpus per user per machine. The install captures `os.userInfo().uid` and provisions a UID-scoped firewall rule. No multi-user, no cross-machine sync.
- [x] **Principle V (Schema-Enforced Structured Output)** — FR-INSTALL-020 + SC-007-028 bind every install / uninstall / taxonomy-promote input and output to Zod schemas: `InstallReceiptZodSchema`, `TaxonomySeedZodSchema`, `MCPClientConfigEntryZodSchema`, `TaxonomyPromoteArgsZodSchema`, `ConfigTomlZodSchema` (existing SP-001+). Defense-in-depth validation at write time AND at read time (install-receipt is validated at uninstall preflight).
- [x] **Principle VI (One Pipeline, Two Policies)** — The `--smoke` end-to-end harness (FR-INSTALL-013) invokes the SAME `ingestStage` / `classifyStage` / `embedStage` / `indexStage` / `edgesBuildStage` pipeline that production uses (no per-test stub pipeline; the daemon and the CLI both invoke the same library functions). One pipeline, one policy at smoke time (daemon-policy).
- [x] **Principle VII (Cancellable, Bounded IO)** — FR-INSTALL-017 + SC-007-029 + SC-007-034 commit to AbortSignal propagation through every install / uninstall / taxonomy-promote IO operation. The install's 90-second budget is enforced via `setTimeout(() => controller.abort('install_budget_exceeded'), 90000)` + `clearTimeout` on cascade completion (NEVER `Promise.race(setTimeout)` — Constitution VII forbidden pattern); the smoke's 30-second sub-budget is enforced the same way; SIGINT aborts in-flight steps and unwinds via the receipt.
- [x] **Principle VIII (Atomic Writes & Transactional Index Updates)** — The install-receipt is written atomically via `withTempDir` (FR-INSTALL-012); the MCP-client config mutation is written back atomically via `withTempDir` (FR-INSTALL-009). The single SQLite index is committed transactionally per the SP-005 index-persister contract (preserved). The curated taxonomy seed is inserted in a single transaction (FR-INSTALL-008).
- [x] **Principle IX (Concurrency-Safe Shared State)** — FR-INSTALL-014 + SC-007-022 bind `Paths.drainLock()` acquisition by `corpus taxonomy promote` BEFORE mutating `taxonomy_terms`; concurrent CLI invocations during a held lock emit `pipeline.lock_contention` and exit non-zero (the SP-003 / SP-004 / SP-005 / SP-006 contract preserved across the new taxonomy-mutation surface). The install acquires the drain-lock for its single seed-insert transaction (preserving the same contract).
- [x] **Principle X (Idempotent Pipeline Transitions)** — FR-INSTALL-004 (install) + FR-INSTALL-014 (taxonomy promote: already-established term is a no-op) + FR-INSTALL-017 (uninstall: already-reversed side-effects are skipped per receipt-driven incremental reverse): SP-007 is idempotent across re-runs. The curated seed uses `INSERT OR IGNORE` (idempotent across re-installs). The OS firewall provisioning is tag-keyed (anchor `corpus` on macOS; comment `llm-corpus` on Linux) and skip-if-present.
- [x] **Principle XI (Library/CLI Boundary)** — FR-INSTALL-019 + SC-007-032 commit to `process.exit` only in `packages/cli/src/install-command.ts` and `packages/cli/src/uninstall-command.ts`. Library helpers return `Result<T, E>` or throw typed errors. The SP-001 `no-process-exit-in-libs` ESLint rule is scoped over SP-007 source.
- [x] **Principle XII (Subprocess Hygiene)** — FR-INSTALL-018 + SC-007-030 commit to every external command (`pfctl`, `iptables`, `systemctl`, `launchctl`, `sudo`, `corpus daemon start`, `corpus mcp`) invoked via `runTool(cmd, args[], opts)` with an arg array; ZERO string-formed shell commands. The SP-001 `no-shell-string-exec` lint scopes over SP-007 source.
- [x] **Principle XIII (Telemetry-or-Die)** — FR-INSTALL-021 + SC-007-033 commit to ≥ 12 new SP-007 event classes across install + uninstall + taxonomy-promote; every catch block emits a telemetry event before returning or re-throwing (existing AST-level lint covers SP-007). Telemetry NEVER includes secrets (the sudo password prompt is handled by `runTool()`'s stdin inheritance, never logged).
- [x] **Principle XIV (XDG Paths via Single Resolver)** — FR-INSTALL-005 + SC-007-006 + SC-007-031 commit to every install / uninstall path routing through `Paths.*`; ZERO new `Paths.*` getters in SP-007. The MCP-client config path (`~/.claude.json`) is the only path the install touches outside `Paths.*` and is recorded explicitly in the install-receipt. The `paths-from-resolver-only` lint scopes over SP-007 source.
- [x] **Principle XV (Dynamic Taxonomy with User-Reviewed Promotion)** — The `corpus taxonomy promote` CLI subcommand IS the user-reviewed-promotion surface for v1. Per Principle XV, taxonomy growth requires explicit user review; SP-007's promote CLI is the canonical surface. The curated seed (FR-INSTALL-008) is a one-time human-curated bootstrap, not auto-promoted. Promote operations are Zod-validated (per Principle V) and atomic (per Principle VIII).
- [x] **Principle XVI (Validation Honesty)** — SC-007-034 commits to the 90-second budget as an enforced ceiling, NOT a marketing target; the install fails non-zero if exceeded.
