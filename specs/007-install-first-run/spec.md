Source constitutional principles: I (Local-First, No Egress — install pulls only the npm package via the user's existing npm registry path; runtime introduces ZERO new outbound endpoints; OS firewall rule provisioned at install time per ADR-001 path (b)), III (Substrate, Not Surface — ZERO new MCP mutation surfaces; `corpus init` / `corpus uninstall` / `corpus taxonomy promote` are CLI subcommands, NOT MCP tools), V (Schema-Enforced Output — install-state manifest, uninstall-receipt, taxonomy-seed payload, and `corpus taxonomy promote` arguments are all Zod-validated at their boundaries), VII (Cancellable, Bounded IO — every IO operation during install + first-run accepts an AbortSignal and respects a per-step timeout; the 90-second budget is enforced via AbortController, not a wall-clock poll), XII (Subprocess Hygiene — the firewall provisioning step invokes `pfctl` (macOS) or `iptables` (Linux) via `runTool()` with an arg array, never via a string-formed shell command), XIV (XDG Paths via Single Resolver — every path the install creates and every path the uninstall removes routes through `Paths.*`; the install introduces ZERO new `Paths.*` getters), XI (Library/CLI Boundary — only `packages/cli/src/install-command.ts` and `packages/cli/src/uninstall-command.ts` may `process.exit`; library helpers return `Result<T, E>`).

This is feature 007 — the install owner for the SP-001..SP-006 substrate. SP-006 closed the production-hardening loop (kill-9 cross-stage recovery, `corpus://failures` resource, Tier 1/2/3 fallthrough); after SP-006 merged at squash `5237916` the substrate is feature-complete on `main`, but the path from "user has Node 18+ and an empty home directory" to "first successful `corpus.find` over a seeded corpus" is informal: the only working install on pai-node01 today is the Pallas-side advance deliverable (D-027) — a bash shim at `~/.local/bin/corpus`, an XDG subtree at `~/.local/share/llm-corpus/`, a systemd user unit at `~/.config/systemd/user/corpus.service` (enabled, active), and an MCP server registered in `~/.claude.json` (user scope). That install works, but it is not the spec target. TR-001 / TR-002 require `npx <pkg> init` as the canonical install path — a publishable npm package, a discoverable `corpus init` subcommand that the `npx` flow invokes, and a `corpus uninstall` subcommand that reverses every install side-effect. SP-007 produces (a) a publishable `@llm-corpus/cli` npm package that exposes a single `corpus` binary; (b) a `corpus init` subcommand that creates the XDG subtree, opens an empty single-file SQLite index, generates a default `config.toml`, registers the MCP server in the user's MCP-client config, provisions the ADR-001 OS firewall rule, optionally installs a systemd / launchd auto-start unit, and prints next-step instructions — all within 90 seconds end-to-end on a clean Linux or macOS machine with Node 18+ and Ollama already running; (c) a `corpus uninstall` subcommand that consults a persistent install-receipt and reverses every recorded side-effect (XDG subtree removed only with `--purge`; MCP-client registration unconditionally removed; OS firewall rule unconditionally reversed; auto-start unit unconditionally removed) — verifiable via filesystem diff + post-uninstall firewall query; (d) a `corpus taxonomy promote` CLI subcommand that closes the SP-006 C-045 cold-start vocabulary UX gap by elevating proposed terms to established — plus a curated ≤ 50-term taxonomy seed delivered at first-run so the operator can classify documents on the first try without raw SQL; (e) an end-to-end smoke harness (per C-046 from SP-006 retro) that spawns the production CLI binary, runs `corpus init` against a clean fixture HOME, drops a seed document, invokes `corpus.find` via the real MCP server, and asserts the result — closing the SP-006 "library-handler test passes therefore done" gap permanently. SP-007 ships ZERO new MCP mutation surfaces (Principle III); ZERO new SQL tables (the substrate schema is frozen post-SP-006); ZERO new `Paths.*` getters (Principle XIV); and ZERO new outbound endpoints at runtime (Principle I — the only outbound call across the entire install flow is the user's local npm registry fetch, which happens at `npx` time before the corpus runtime starts and is therefore outside NFR-002's scope per ADR-001 §"Origin of these alternatives"). SP-007 reconciles the three open SP-006 retrospective concerns with explicit include/defer decisions: **C-045 (HIGH) — INCLUDED** (the `corpus taxonomy promote` CLI + a curated ≤ 50-term seed delivered at first-run; this is on the 90-second critical path because cold-start documents cannot classify without seeded vocabulary, blocking the NFR-014 exit criterion); **C-043 (low) — DEFERRED** to a post-SP-007 polish PR (the `signals_used: []` bug in tier-orchestrator is a contract-cosmetic issue in `packages/index/src/tier-orchestrator.ts`, NOT first-run-UX-blocking; deferral with a referenced TODO in the SP-007 spec is sufficient); **C-044 (low) — DEFERRED** to the same post-SP-007 polish PR (the `regenerateCatalogFromDb` references a non-existent `summary` column failure is non-fatal and only fires on `corpus reindex`, not on the first-run path). After SP-007 merges, the substrate is install-complete AND first-run-functional — SP-008 (the final must-have sprint, user-level acceptance per UR-001/UR-002/UR-003) can run a Maya-persona Week-1 engagement-proxy gate from a clean install on a clean machine, end-to-end.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A new user on a clean machine runs `npx @llm-corpus/cli init` and has a working corpus within 90 seconds (Priority: P1)

The user (Maya / new operator / Shon-on-a-fresh-laptop) starts with a clean Linux or macOS machine: Node 18+ is installed (the `npx` requirement), Ollama is installed and running locally with `qwen3:8b` (classifier) and `nomic-embed-text` (embedder) models pulled, and the user has a normal POSIX home directory but no existing `~/.local/share/llm-corpus/`, no `corpus` binary on PATH, and no MCP-client registration. The user runs:

```
npx @llm-corpus/cli init
```

The `npx` flow downloads the `@llm-corpus/cli` npm package (single package; no Docker; no native compile step — `better-sqlite3` and `sqlite-vec` ship prebuilt binaries for Linux and macOS via the npm prebuild path), invokes the package's `bin` entry as `corpus init`, and the `corpus init` subcommand performs the following sequence within an enforced 90-second AbortController budget: (1) preflight checks — Node ≥ 18.0.0 with a clear-remediation error if not; Ollama reachable on `http://127.0.0.1:11434/api/tags` with a clear-remediation error if not; XDG directories writable with a clear-remediation error if not (per ACCEPTANCE-CRITERIA TR-001 scenario "install on machine without write permission"); (2) idempotency check — if `Paths.indexDb()` already exists (re-running on existing install per TR-001 scenario "re-running `corpus init` on existing install does not destroy user data"), the command prints "already initialized at `<path>`" and exits 0 without destroying anything; (3) XDG subtree creation — `Paths.config()`, `Paths.data()`, `Paths.state()`, `Paths.cache()` and the eight subdirectories `Paths.docs()`, `Paths.inbox()`, `Paths.pending()`, `Paths.processed()`, `Paths.failed()`, `Paths.trash()`, `Paths.docsStore()`, `Paths.pilotTelemetry()` — created via `fs.mkdir({recursive: true})`; (4) empty SQLite index — open `Paths.indexDb()` via the substrate's index-open path, run the migration set, leave the DB in single-file form per NFR-010 (the single-file commitment is enforced by SC-007-009 below: no `*.db-shm` or `*.db-wal` sidecars survive the install — WAL is enabled but checkpointed before the install exits so the on-disk surface is one file); (5) default `config.toml` — write a minimal config at `Paths.configFile()` with the SP-004 classifier model (`qwen3:8b`), the SP-005 embedder model (`nomic-embed-text`), the SP-005 retrieval budget defaults, and the SP-006 tier-fallthrough defaults — every value Zod-validated against the existing config schema; (6) curated vocabulary seed — bulk-insert the SP-007 curated seed (≤ 50 terms spanning domain / type / tag / source_type axes, drawn from the SP-006 USER-GUIDE.md workaround list as the source-of-truth seed: 5 domains, 6 types, 9 tags, 5 source_types) into `taxonomy_terms` with `state='established'` via `INSERT OR IGNORE` (idempotent across re-runs); (7) MCP-client registration — append a corpus-server entry to the user's MCP-client config (`~/.claude.json` by default; configurable via `--mcp-client-config <path>`) keyed by client; the entry's `command` field is the absolute path to the installed `corpus` binary (resolved via `process.execPath` + `npm bin -g` lookup or the `npx`-cached path), `args` is `["mcp"]`, no environment variables; (8) OS firewall rule — provision the ADR-001 path (b) UID-scoped firewall rule via `runTool('pfctl', [...])` on macOS or `runTool('iptables', [...])` on Linux (rule blocks outbound non-loopback for the corpus runtime UID; the install captures `os.userInfo().uid` and writes a UID-scoped rule); record the rule's reverse-command in the install-receipt for `corpus uninstall` to consume; (9) optional auto-start unit — if `--enable-autostart` is provided (default OFF for the 90-second-budget critical path; the operator explicitly opts in), write a systemd user unit (Linux) or launchd plist (macOS); the unit's `ExecStart` is the absolute corpus-binary path with `daemon start`; (10) install-receipt — write a single JSON receipt at `Paths.state() + '/install-receipt.json'` listing every side-effect with enough information for uninstall to reverse it (paths created, files written, MCP-client config entries added, firewall rules added, auto-start units installed); the receipt's Zod-validated shape lets uninstall verify the install was complete before reversing; (11) next-step output — print to stdout the next-step instructions per TR-001 ACCEPTANCE-CRITERIA: "drop documents into `<inbox-path>`; ask your Claude Code session 'check my corpus for X' to query"; (12) end-to-end smoke (per C-046) — if `--smoke` is provided, spawn `corpus daemon start`, drop a single seed document (the SP-007 ships a 1-paragraph fixture at `packages/cli/fixtures/first-run-seed.md`) into `Paths.inbox()`, wait for the four-sub-stage pipeline to complete (validation → classify → embed → index → edges-build) within the per-doc budget, invoke `corpus.find` via a child process speaking real MCP-stdio against the daemon's MCP server (NOT a library-level handler test), assert the result has ≥ 1 hit, and exit 0; without `--smoke`, the command exits at step (11) and the user runs the smoke manually. The entire flow (steps 1-11, without the optional smoke at step 12) completes in ≤ 90 seconds on a clean Linux or macOS machine with Node 18+ and Ollama already running. The smoke (step 12) is exempt from the 90-second budget because it depends on Ollama wall-clock per the SP-004 classifier budget (~10 s) + the SP-005 embed budget (~2 s) + the SP-005 index budget (~1 s); it has its own 30-second budget.

**Why this priority**: P1 because (a) the 90-second first-run UX is the public-facing differentiator vs Letta / Mem0 per RM-008 success metric and per Stage 2 Research (Mem0 self-hosted ~2-5 min with Docker pulls; Letta Code sub-minute with Node 18+ present; corpus's commitment is `<` 90 s without Docker); (b) cannot ship v1 without this; the SP-008 user-acceptance sprint depends on a clean-install path that doesn't require Pallas to hand-build it; (c) closes TR-001 + NFR-014 atomically — the single install command and the 90-second budget are inseparable. P1 also because the current state (Pallas-side bash-shim install per D-027) is operator-hostile for anyone but Shon — every install requires the developer to be the operator, which is a Goodhart's-law violation of "single command" semantics.

**Independent Test**: On a clean Linux or macOS machine (e.g., a fresh VM or a docker-machine without prior corpus state — verified by `ls ~/.local/share/llm-corpus/ ~/.config/llm-corpus/ ~/.local/state/llm-corpus/ 2>&1 | grep -c "No such file"` returns ≥ 3), with Node 18+ installed and Ollama running with `qwen3:8b` + `nomic-embed-text` pulled, the operator runs `npx @llm-corpus/cli init` and `time` the invocation. Observe: stdout shows the preflight banner, the XDG-paths-created list, the vocabulary-seed count, the MCP-client-registration confirmation, the firewall-rule-installed line, and the next-step block; the `time` output reports wall-clock < 90 seconds; `ls ~/.local/share/llm-corpus/` shows the inbox/processed/failed/store subtree; `sqlite3 ~/.local/share/llm-corpus/index.db ".schema documents"` shows the SP-003 documents table; `sqlite3 ~/.local/share/llm-corpus/index.db "SELECT count(*) FROM taxonomy_terms WHERE state='established'"` returns ≥ 25 (the curated seed); `~/.local/state/llm-corpus/install-receipt.json` exists and is Zod-valid; `jq '.mcpServers.corpus' ~/.claude.json` shows the corpus-server entry; on macOS `sudo pfctl -sr | grep corpus-uid` shows the UID-scoped block rule; on Linux `sudo iptables -L OUTPUT -n | grep <uid>` shows the UID-scoped REJECT rule. With `--smoke`, the operator additionally drops the seed document, waits for the daemon to process it, invokes `corpus.find` via the production MCP server (spawned as a child process speaking stdio), and asserts the response has ≥ 1 SearchHit pointing at the seed-document body.

**Acceptance Scenarios**:

1. **Given** a clean Linux or macOS machine with Node 18+ and Ollama running with `qwen3:8b` and `nomic-embed-text` pulled AND no prior corpus install, **When** the user runs `npx @llm-corpus/cli init`, **Then** the command completes in ≤ 90 seconds wall-clock AND the XDG-compliant subtree is created at `Paths.data()` / `Paths.config()` / `Paths.state()` / `Paths.cache()` AND an empty single-file SQLite index exists at `Paths.indexDb()` with the SP-006 schema migrated AND a curated ≥ 25-term vocabulary seed is loaded into `taxonomy_terms` with `state='established'` AND `~/.claude.json` contains a `mcpServers.corpus` entry AND the OS firewall rule blocks outbound non-loopback for the corpus runtime UID AND an install-receipt at `Paths.state() + '/install-receipt.json'` records every side-effect AND stdout shows next-step instructions AND the command exits 0.

### User Story 2 — The user runs `corpus uninstall` and every install side-effect is reversed, verifiable via filesystem diff and firewall query (Priority: P1)

The user has a working corpus install (the SP-007 US1 flow has run successfully — XDG subtree, single-file SQLite index, MCP-client registration, OS firewall rule, install-receipt, possibly optional auto-start unit). The user runs:

```
corpus uninstall
```

The `corpus uninstall` subcommand consults `Paths.state() + '/install-receipt.json'` to determine what to reverse. If the receipt is missing or malformed, the command exits non-zero with a clear-remediation message (the operator can manually rm the XDG subtree if the receipt is unrecoverable). With a valid receipt, the command performs the following sequence: (1) preflight — verify the receipt's Zod shape; verify the OS firewall reverse-commands are still valid for the current OS (the receipt records the OS at install time); abort with a clear error if anything is unrecoverable; (2) MCP-client config — remove the corpus-server entry from each MCP-client config file recorded in the receipt; preserve every other entry; the JSON is parsed, mutated, written back atomically via `withTempDir` (Constitution VIII); (3) OS firewall rule — invoke `runTool('pfctl', [...])` (macOS) or `runTool('iptables', [...])` (Linux) with the reverse-command recorded in the receipt; on success, the post-uninstall firewall query (`pfctl -sr | grep corpus-uid` returns empty; `iptables -L OUTPUT -n | grep <uid>` returns empty) confirms reversal; (4) optional auto-start unit — if the install recorded a systemd or launchd unit, disable + remove it via the OS-appropriate mechanism (`systemctl --user stop corpus.service && systemctl --user disable corpus.service && rm <unit-path>` on Linux; `launchctl unload <plist>` + rm on macOS); (5) XDG subtree — IF AND ONLY IF `--purge` is provided, delete the four XDG directories (`Paths.data()`, `Paths.config()`, `Paths.state()`, `Paths.cache()` scoped to `llm-corpus`); WITHOUT `--purge`, the user data (index.db, inbox documents, processed documents, frontmatter files, taxonomy state) is preserved by default per TR-002 (the operator can re-run `corpus init` later and pick up where they left off); (6) install-receipt — IF `--purge`, delete the receipt; ELSE, mark the receipt as `uninstalled` (preserving the side-effect history so a future re-install can audit); (7) post-uninstall verification — print a filesystem-diff summary (XDG paths still present / absent), a firewall-query summary (rule still present / absent), and a MCP-client-config summary (corpus entry still present / absent); the user can paste this into a bug report; (8) exit 0.

**Why this priority**: P1 because (a) TR-002 + the SP-007 exit criterion `uninstall_reverses_all_install_side_effects_verifiable_via_filesystem_diff` and `uninstall_reverses_os_firewall_rule_verified_via_post_uninstall_firewall_query` together require a deterministic reversal path; without it, the install is operator-hostile (any install error leaves debris the operator can't easily clean up); (b) firewall-rule cleanup is load-bearing — a stale block rule on the corpus UID after `corpus uninstall` is a security-stance violation (the user trusts the system to clean up after itself per ADR-001's "Trigger to revisit" provision); (c) `--purge` semantics protect against catastrophic data loss — uninstall without `--purge` MUST NOT remove user data (per ACCEPTANCE-CRITERIA TR-002 scenarios). P1 also because uninstall is the most common operator path during dogfooding (operator iterates: install → smoke → uninstall → reinstall against a fixed version) — broken uninstall blocks the entire SP-008 user-acceptance gate.

**Independent Test**: On a machine with a successful `corpus init` install (US1 verified), the operator captures the pre-uninstall filesystem state: `find ~/.local/share/llm-corpus ~/.local/state/llm-corpus ~/.config/llm-corpus -type f | sort > /tmp/pre-install-tree.txt`, `jq '.mcpServers.corpus' ~/.claude.json > /tmp/pre-install-mcp.json`, `sudo pfctl -sr 2>/dev/null | grep corpus-uid > /tmp/pre-install-firewall.txt` (macOS) or `sudo iptables -L OUTPUT -n | grep <uid> > /tmp/pre-install-firewall.txt` (Linux). The operator runs `corpus uninstall` (NO `--purge`). Observe: stdout prints the side-effect reversal summary; `jq '.mcpServers.corpus' ~/.claude.json` returns `null` (the entry is gone); the firewall query returns empty (the rule is gone); the XDG subtree at `~/.local/share/llm-corpus/` is INTACT (user data preserved). The operator then runs `corpus uninstall --purge`; the XDG subtree is gone; the install-receipt is gone. Final filesystem diff (`diff /tmp/pre-install-tree.txt <(find ~/.local/share/llm-corpus ~/.local/state/llm-corpus ~/.config/llm-corpus -type f 2>/dev/null | sort)`) shows every install-created path is removed.

**Acceptance Scenarios**:

1. **Given** a successful prior `corpus init` install with 100 ingested documents and an inbox of 10 pending files, **When** the user runs `corpus uninstall` (no flag), **Then** the MCP-client registration is removed from `~/.claude.json` AND the OS firewall rule for the corpus UID is reversed (post-uninstall firewall query returns empty) AND any auto-start unit is disabled and removed AND the SQLite index file is preserved AND the inbox directory and its contents are preserved AND the document frontmatter files are preserved (per ACCEPTANCE-CRITERIA TR-002 scenario "removes server registration but preserves user data by default").
2. **Given** a successful prior install with ingested documents, **When** the user runs `corpus uninstall` without any flag, **Then** no document, index, or frontmatter file is deleted (per ACCEPTANCE-CRITERIA TR-002 scenario "does NOT remove user data without explicit destructive flag").
3. **Given** a successful prior install, **When** the user runs `corpus uninstall --purge`, **Then** the MCP-server registration is removed AND the OS firewall rule is reversed AND the index file is deleted AND ingested documents and frontmatter files are deleted AND the install-receipt is deleted AND the XDG subtree at `Paths.data() / Paths.config() / Paths.state() / Paths.cache()` is removed (per ACCEPTANCE-CRITERIA TR-002 scenario "removes user data only with explicit destructive flag").
4. **Given** the install-receipt at `Paths.state() + '/install-receipt.json'` is missing or malformed, **When** the user runs `corpus uninstall`, **Then** the command exits non-zero AND stderr includes a clear-remediation message naming the receipt path AND no XDG paths are deleted AND no firewall rule is reversed (the operator must manually clean up).
5. **Given** a successful prior install where the OS firewall rule was provisioned via `pfctl` (macOS) at install time, **When** the user runs `corpus uninstall`, **Then** the install-receipt's recorded reverse-command (`pfctl -a corpus -F all`) is invoked via `runTool()` AND the post-uninstall firewall query (`pfctl -sr | grep corpus-uid`) returns empty AND a confirmation line is printed.
6. **Given** a successful prior install where an optional auto-start unit was installed (`--enable-autostart` was passed at init), **When** the user runs `corpus uninstall`, **Then** the auto-start unit is disabled AND the unit file is removed AND a confirmation line is printed.
7. **Given** the user runs `corpus uninstall` mid-flow (Ctrl+C / SIGINT), **When** the AbortSignal fires, **Then** in-flight IO is cancelled AND the install-receipt is updated to reflect partial-uninstall state AND a clear-remediation message names the step that was interrupted AND the user can re-run `corpus uninstall` to complete the reversal (the install-receipt's reverse-list is consulted incrementally — already-reversed side-effects are skipped per Constitution X idempotent transitions).

### User Story 3 — The user runs `corpus taxonomy promote` to elevate proposed terms to established, closing the cold-start vocabulary UX gap (Priority: P1)

The user has a working corpus install (US1 verified), has dropped some documents into the inbox, and the SP-004 classifier has produced output where some documents have `facet_type='unclassified'` because their proposed domain / type / tag / source_type values are not yet in the established vocabulary (the SP-007 curated seed covers the most common terms but cannot anticipate every operator's corpus). The user inspects the proposed-term table via `corpus taxonomy list --state=proposed` (a list-only subcommand; no mutation) and decides which terms to elevate. The user runs:

```
corpus taxonomy promote --axis=domain --term=climbing
corpus taxonomy promote --axis=tag --term=mountaineering --term=alpine
corpus taxonomy promote --from-proposed-with-count-ge=3
```

The `corpus taxonomy promote` subcommand validates its arguments via Zod (closed `--axis` enum: `domain`/`type`/`tag`/`source_type`; `--term` is a string per term; `--from-proposed-with-count-ge=N` elevates every proposed term whose `proposed_count >= N`; mutually exclusive with `--axis/--term`), opens the substrate's SQLite database via the existing index-open path, updates the `taxonomy_terms` rows from `state='proposed'` to `state='established'` for the named terms, sets `established_at=datetime('now')`, prints a confirmation line per term, and exits 0. The subcommand acquires `Paths.drainLock()` BEFORE mutating taxonomy state (Constitution IX — taxonomy_terms is shared state across the daemon + the classifier hook); on contention the command emits `pipeline.lock_contention` and exits non-zero with a clear-remediation message (the operator stops the daemon and retries). After promotion, the user runs `corpus reenrich` (existing CLI subcommand from SP-004) to re-classify the previously-stuck documents against the now-richer vocabulary. The previously-stuck documents become classified and indexed.

### Edge Cases

- **`npx` cache poisoning / partial download**: If `npx` fails to download the package (network blip, disk full), the command exits non-zero before `corpus init` ever runs. No XDG state is created. Re-run resumes from a clean slate.
- **Concurrent `corpus init` invocations**: If two `corpus init` processes race (e.g., two terminals start install simultaneously), the second invocation observes contention via the existing SP-003 `Paths.drainLock()` pattern, emits `pipeline.lock_contention`, and exits non-zero. The first completes; the second can be re-run after observing the existing install.
- **Re-run `corpus init` after partial failure**: If the install failed after creating some XDG paths but before writing the install-receipt, the next `corpus init` sees an inconsistent state. Decision: the install-receipt is the source-of-truth for "install completed"; a partial install (XDG paths exist but no receipt) is detected by the preflight step which prints "partial install detected at `<paths>`; run `corpus uninstall --purge` to clean up, then re-run `corpus init`" and exits non-zero. The operator manually resolves.
- **`~/.claude.json` already has a non-corpus MCP-server entry**: The install appends the corpus-server entry without disturbing other entries; the JSON is parsed, mutated, written back atomically. If the parse fails (the user's config is malformed), the install exits non-zero before any other side-effect; the operator fixes the config and re-runs.
- **`~/.claude.json` does not exist at install time**: The install creates it with `{"mcpServers": {"corpus": {...}}}` shape; ownership is preserved (the operator owns the file).
- **OS firewall provisioning requires sudo**: `pfctl` and `iptables` require root. The install detects this at the preflight step; if not running as root and `sudo` is available, the install invokes `sudo pfctl ...` via `runTool()` and prompts the user once for the password; if `sudo` is unavailable, the install exits non-zero with a clear-remediation message naming the manual command to run.
- **OS firewall rule already exists on a re-install**: The provisioning step is idempotent per Constitution X; the rule is either matched by tag (`pfctl -a corpus`) and replaced, or detected as already-present and skipped. The install-receipt records the rule's hash for idempotent reverse on uninstall.
- **Auto-start unit conflict**: If a unit file already exists at the install's target path (the operator hand-wrote one for the Pallas-side D-027 install), the install detects the conflict and exits non-zero with a clear-remediation message offering `--force-autostart` to overwrite; the operator can also opt out of auto-start via `--no-autostart`.
- **`corpus uninstall` on a machine where the install-receipt was lost**: The command exits non-zero with a clear-remediation message naming the four XDG paths to manually `rm -rf` and the firewall command to manually reverse. (The receipt-driven uninstall is the supported path; the manual fallback is documented.)
- **`corpus uninstall --purge` on a corpus with active daemon**: The uninstall detects the active daemon (PID file in `Paths.state()`), invokes `corpus daemon stop` first, waits for shutdown, then proceeds. If the daemon doesn't stop within the Constitution VII 2-second budget, the uninstall exits non-zero with a clear-remediation message instructing the operator to kill the daemon manually.
- **`corpus taxonomy promote` race with the classifier**: The classifier reads `taxonomy_terms` on every classify call; if a promote runs mid-classify, the next classify call sees the updated vocabulary. The `Paths.drainLock()` serialization (FR-INSTALL-014 below) prevents intra-call corruption (the classifier holds a read lock while the promote tries to mutate; the promote acquires the lock when the read releases). The promote is fast (single SQL UPDATE); operator-perceived latency is negligible.
- **`corpus taxonomy promote --from-proposed-with-count-ge=N` with N=0**: Promotes every proposed term in the table, including singletons. Documented as the most-permissive option; the operator chooses N based on noise tolerance (the SP-006 USER-GUIDE.md guidance suggests N ≥ 3 for typical corpora; the SP-007 default is no default — the operator must specify either `--term <t>` or `--from-proposed-with-count-ge <N>`).
- **First-run on a machine without Ollama**: The preflight check fires before any XDG paths are created; the user gets a clear-remediation message naming Ollama and the two required models. No partial install state. Re-run after the operator installs Ollama proceeds normally.
- **`--smoke` step exceeds its 30-second sub-budget**: The smoke is exempt from the 90-second non-smoke budget but bounded by 30 seconds of its own. If Ollama is unusually slow (cold-load on first call), the smoke fails the budget; the operator sees a clear-remediation message; the non-smoke install is already complete (steps 1-11 succeeded). The operator can retry the smoke manually by running `corpus daemon start && cp <seed> ~/.local/share/llm-corpus/docs/inbox/ && corpus mcp ...`.
- **NFR-010 single-file enforcement vs WAL mode**: SQLite WAL produces `.db-shm` + `.db-wal` sidecars during normal operation. SP-007 commits to single-file at install-exit (WAL is checkpointed before exit so the on-disk surface is one file). During daemon runtime, the sidecars exist; they are NOT a violation of NFR-010 because the "single SQLite file" commitment is measured at the file-level backup/copy/move boundary (per NFR-010 description verbatim: "Backup, copy, move operations are file-level requiring no service coordination") — operationally, the operator can stop the daemon, checkpoint WAL, and the on-disk index is one file.
- **The curated taxonomy seed conflicts with an operator's prior `taxonomy_terms` entries**: The seed uses `INSERT OR IGNORE`; existing entries with `state='established'` or `state='proposed'` are NOT modified. The seed is purely additive.

## Assumptions

- **Primary user**: shonrs on Fedora workstation (pai-node01) with Claude Code as the primary MCP-aware client; same scope as SP-001..SP-006. Secondary user: Maya-persona on a clean Linux or macOS machine.
- **Supported platforms for v1**: Linux (Fedora baseline; tested on pai-node01) and macOS (Apple Silicon + Intel). Windows out of scope for v1 (the `pfctl` / `iptables` / `systemd` provisioning paths don't translate; Windows would require a separate firewall-and-service provisioning path).
- **Prerequisite — SP-001..SP-006 merged**: The substrate is feature-complete on `main` at SP-006 squash `5237916` (2026-05-14). SP-007 consumes the SP-006 schema (taxonomy_terms with `state` column; documents table; failure-lane sidecar shape; `corpus://failures` resource; tier-fallthrough orchestrator). The Pallas-side advance deliverable D-027 (the bash-shim install on pai-node01) is operationally working but is NOT the SP-007 spec target; SP-007's `npx` flow is what the canonical install becomes.
- **Node 18+ is the floor**: Per NFR-014 verbatim. Node 16 is unsupported (the preflight check rejects it with a clear-remediation message).
- **Ollama is a prerequisite installed by the operator before `corpus init`**: SP-007 does NOT bundle or install Ollama. The preflight check (FR-INSTALL-003) verifies Ollama reachability; absent Ollama, the install exits non-zero with a clear-remediation message naming Ollama and the two required models (`qwen3:8b` classifier + `nomic-embed-text` embedder).
- **`pfctl` (macOS) and `iptables` (Linux) are available on PATH**: SP-007's FR-INSTALL-010 firewall provisioning depends on these binaries being present. Both are part of the default OS install (macOS Big Sur+ and Fedora baseline). If absent (e.g., a minimal container), the install exits non-zero with a clear-remediation message.
- **`sudo` is available for firewall provisioning**: The install requires root to mutate the OS firewall. If the install is not run as root and `sudo` is available, the install invokes `sudo runTool(...)` and prompts the user once. If `sudo` is unavailable (e.g., a non-admin user), the install exits non-zero with a clear-remediation message naming the manual command to run.
- **The MCP-client config path defaults to `~/.claude.json`**: SP-007's MCP-client registration assumes Claude Code as the canonical client (per the user's profile). Other MCP-aware clients (Gemini CLI, Codex CLI) are post-v1 per AG-004 / OOS-011; SP-007 does not register the corpus server in their configs.
- **`better-sqlite3` and `sqlite-vec` prebuild binaries cover Linux x64 + macOS x64 + macOS arm64**: SP-007's FR-INSTALL-001 commits to no native compile step. If a prebuild binary is unavailable for a target platform (vanishingly rare on the v1 supported platforms), the `npm install` step fails before `corpus init` ever runs; the operator sees an npm-level error.
- **`Paths.*` resolver getters are stable across SP-006 → SP-007**: SP-007 reuses every existing `Paths.*` getter and introduces ZERO new getters per the dispatch prompt's discipline. Any change to `Paths.*` is an out-of-scope concern (`packages/contracts/src/paths.ts` is owned by the substrate sprints, not by SP-007).
- **`taxonomy_terms` schema is stable post-SP-004**: SP-007's seed insert and the `corpus taxonomy promote` mutation both depend on the SP-004 `taxonomy_terms` schema (`axis`, `term`, `state`, `established_at`, `proposed_count`). The schema is frozen for v1; any change is an out-of-scope concern.
- **The curated seed is sourced from the SP-006 USER-GUIDE.md workaround list as the seed-source-of-truth**: The 25-term floor comprises 5 domains + 6 types + 9 tags + 5 source_types from the documented workaround. The seed may expand up to 50 terms during SP-007 plan-stage research (Pallas-side curation against the pai-node01 corpus content), but never exceeds 50 (per dispatch prompt C-045 "≤ 50-term seed" cap).
- **The `--smoke` step uses a deterministic seed document**: SP-007 ships a 1-paragraph English-language fixture at `packages/cli/fixtures/first-run-seed.md` whose content matches a deterministic `corpus.find` query. The smoke's pass condition is deterministic across Ollama's tolerance for embedding non-determinism.
- **The 90-second budget excludes Ollama-dependent work**: The non-smoke install (steps 1-11 of FR-INSTALL-002) is Ollama-reachability-only; no Ollama generation calls fire. The 30-second smoke sub-budget (step 12) is Ollama-dependent and exempt from the 90-second non-smoke ceiling.
- **The install-receipt is the ONLY supported uninstall path**: Manual uninstall (operator-driven `rm -rf` + `pfctl -F` + `jq` edits) is documented as a fallback but is not warranted. The SP-007 commitment is the receipt-driven path.
- **Auto-start is OFF by default**: `--enable-autostart` is opt-in. The operator can manually start the daemon via `corpus daemon start` and stop via `corpus daemon stop` post-install. This protects the 90-second budget and avoids surprising the operator with an always-on background service.
- **C-046 smoke harness is gated on Ollama availability in CI**: The CI runner is `ubuntu-latest`; Ollama may or may not be present (Ollama-in-CI is a separate infra concern out of SP-007 scope). The smoke test conditionally skips when Ollama is absent; locally on every dev machine with Ollama, it runs unconditionally.
- **The install-receipt's `corpus_binary_path` is captured at install time and stays stable**: If the operator moves the `corpus` binary post-install (rare for `npx` flows, more common for global installs), the recorded path becomes stale; `corpus uninstall` continues to reverse the recorded side-effects but the MCP-client config's `command` entry points at the stale path. The operator re-runs `corpus init` to refresh the path.
- **The SP-007 build is a single-engineer-agent dispatch**: Per `feedback-build-tier-sizing-rule`, SP-007's total surface is estimated at ~2000-2500 LOC (install command + uninstall command + taxonomy-promote command + install-receipt schema + curated seed + firewall provisioner + MCP-client config mutator + auto-start unit installer + end-to-end smoke harness + tests + fixtures). Pre-planned single-agent build with `/simplify` + `feature-dev:code-reviewer` review before merge; if the surface expands above the 2000-LOC / 15-file threshold during plan-stage research, the build is pre-split into N≥2 Engineer agent invocations per the build-tier sizing rule.
