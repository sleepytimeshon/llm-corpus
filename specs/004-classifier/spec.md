# Feature Specification: Local LLM Classifier — Grammar-Constrained Metadata, Dynamic Vocabulary, Proposed-Term Routing

**Feature Branch**: `004-classifier`
**Created**: 2026-05-13
**Status**: Draft
**Input**: SP-004 from `.product/SPRINT-PLAN.yaml` / `.product/ROADMAP.yaml` RM-005 ("Local-LLM classifier with grammar-constrained metadata"). Source FR text: `.product/REQUIREMENTS.yaml` FR-012 (local-LLM grammar-constrained classification), FR-013 (defense-in-depth schema validation before frontmatter writeback), FR-014 (dynamic-vocabulary validation with separate proposed-term field). Source constitutional principles: I (Local-First, No Egress — Ollama at `localhost:11434`), II (User Curates, LLM Classifies Metadata — frontmatter-only output; confidence excluded from frontmatter persistence), V (Schema-Enforced Structured Output — Ollama `format` parameter rendered from canonical Zod-derived JSON Schema), VII (Cancellable, Bounded IO — every HTTP / SQLite call takes `AbortSignal`), VIII (Atomic Writes & Transactional Index Updates — paired SQL UPDATE + body-file frontmatter rewrite in a single transaction), IX (Concurrency-Safe Shared State — drain-lock reuse, append-only JSONL ≤ 4 KB), X (Idempotent Pipeline Transitions — already-classified row re-run is a no-op), XIII (Telemetry-or-Die — ≥ 6 classify-stage event classes), XV (Dynamic Taxonomy with User-Reviewed Promotion — vocabulary live from `taxonomy_terms WHERE state='established'`, proposed terms route to `state='proposed'` without auto-promotion).

This is feature 004: the **deferred semantic-classification owner** for SP-003. SP-003 ships rows with sentinel classifier-owned columns (`facet_domain=''`, `tags_json='[]'`, `facet_type='unclassified'`). SP-004 reads those sentinel rows, invokes the local Ollama LLM with grammar-constrained generation against a canonical Zod-derived JSON Schema, validates the response defense-in-depth, mirrors the result into the three SQL columns AND the body-file YAML frontmatter inside a single atomic transaction, and routes never-before-seen domain / tag proposals into the `taxonomy_terms` table at `state='proposed'`. SP-004 introduces zero MCP mutation surfaces (Principle III): the daemon's post-persist hook auto-classifies fresh rows, and a new `corpus reenrich` CLI subcommand drains backlog sentinels manually. SP-005 (embedding + ranking) consumes the SP-004-classified rows; until SP-005 lands, `corpus.find` continues to return an empty SearchHit list. SP-006 (kill-9 survival + `corpus://failures` MCP resource) strengthens cross-stage recovery; SP-004 ships single-stage atomicity and on-disk failure-lane sidecars (`*.error.json`) consistent with SP-003's contract.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A freshly ingested document is autonomously classified within the per-document budget without user intervention (Priority: P1)

The user (Maya/Shon persona) drops a file into `Paths.inbox()`. SP-003's watcher detects it, validates it, hashes it, normalizes it to Markdown + YAML frontmatter, and inserts a `documents` row with sentinel classifier columns (`facet_domain=''`, `tags_json='[]'`, `facet_type='unclassified'`). Without any further user action, the SP-003 daemon's post-persist hook invokes the SP-004 classify-stage on the new row. SP-004 loads the live established vocabulary from `taxonomy_terms WHERE state='established'`, constructs a prompt containing (a) the vocabulary block, (b) classification rules, (c) document title + source + first 2000 characters of body, sends a structured-output POST to `http://localhost:11434/api/chat` with the canonical JSON Schema as the `format` parameter, validates the response with Zod, and writes the classifier output to BOTH the SQL row (UPDATE of `facet_domain`, `tags_json`, `facet_type`) AND the body-file YAML frontmatter (mirroring those three fields + `summary` if produced) inside a single SQLite transaction. The row is no longer sentinel; the body file's frontmatter now carries semantic facets. Confidence scores produced by the classifier remain in-memory for retry-decision logic and telemetry only — they are NEVER written to disk (Principle II).

**Why this priority**: This is the load-bearing stakeholder requirement for SP-004 and the entire classifier sprint. Without it, every SP-003 row stays in sentinel state forever and `corpus://taxonomy` continues to return empty established lists indefinitely. P1 because every other classifier behavior (manual reenrichment, proposed-term routing, failure-lane integration) is a refinement on or guard around this primary autonomous path. The agent (principal user) cannot consult the corpus by domain or tag until at least one row is classified.

**Independent Test**: Initialize a clean corpus on the user's primary machine with the SP-003 daemon running and a small taxonomy seed (e.g., 3 established domains, 5 established tags pre-inserted into `taxonomy_terms`). Drop one PDF, one Markdown note, one plain-text file, and one HTML article into the inbox. Without issuing any further command, observe that within a per-document classifier budget (60 s default for qwen3.5:9b on CPU; see Edge Case "classifier wall-clock per document") each row's SQL columns transition from sentinel to populated AND each body file's YAML frontmatter contains `facet_domain` (non-empty), `facet_type` (a value from the SCHEMA.md 7-value enum), and `tags` (a non-empty array of 3-10 strings). Verified end-to-end on the SP-001 + SP-002 + SP-003 baseline.

**Acceptance Scenarios**:

1. **Given** a clean corpus with the SP-003 daemon running and a seed taxonomy of ≥ 1 established domain and ≥ 3 established tags, **When** the user drops one file of any allowlisted MIME type into the inbox, **Then** within the per-document classifier budget the row's SQL columns transition from sentinel (`facet_domain=''`, `tags_json='[]'`, `facet_type='unclassified'`) to populated (`facet_domain != ''`, `tags_json` parses to a JSON array of 3-10 strings, `facet_type` ∈ the SCHEMA.md 7-value enum) AND the body file's YAML frontmatter contains the same three fields with the same values AND `source_type` remains `'inbox-filesystem'` (SP-003-owned, untouched by SP-004).
2. **Given** a freshly ingested row, **When** classify-stage completes successfully, **Then** the `documents` row's `body_path` file is byte-prefix-identical to the SP-003-written body up to the frontmatter delimiter close (`\n---\n`) AND the frontmatter is rewritten via the SP-002 single YAML codec helper (`stringifyMarkdownWithFrontmatter`) AND no `confidence`, `provenance_*`, `origin`, `captured_at`, or `corpus capture` field appears anywhere in the persisted frontmatter (Principle II).
3. **Given** a successful classification, **When** the persistence step commits, **Then** the SQL UPDATE and the body-file frontmatter rewrite commit in a single SQLite transaction AND the body-file write uses `withTempDir` + `tmp + fsync + rename + dirsync` atomicity (Principle VIII) AND on any mid-transaction failure the row stays sentinel AND no orphan tmp body file remains on disk.
4. **Given** classify-stage completes successfully, **When** telemetry is inspected, **Then** the `classify.completed` event class appears with `doc_id`, `model_name`, `duration_ms`, and `retry_count` fields (and `confidence` numeric scores recorded in telemetry, NOT frontmatter — Principle II permits telemetry-only retention).
5. **Given** an MCP client reads `corpus://docs/{id}` for the freshly-classified row, **Then** the response carries the same `facet_domain` / `facet_type` / `tags` values that the SQL row carries (no SQL ↔ frontmatter divergence) AND `corpus://taxonomy` returns the established vocabulary the classifier was given as input (no new established terms appear from a single classify call).

### User Story 2 — The user manually drains the sentinel-row backlog with `corpus reenrich` (Priority: P1)

After upgrading from a pre-SP-004 install (or after running the SP-003 daemon with `qwen3.5:9b` unloaded so classify-stage was a no-op), the user has a backlog of sentinel rows. The user runs `corpus reenrich` from the CLI. The command acquires `Paths.drainLock()` (single serialization point per Principle IX — concurrent SP-003 ingest is blocked while reenrich runs, and vice versa). It selects every row where `facet_type='unclassified'`, batches them, and invokes the same classify-stage code path the daemon uses (Constitution VI — one pipeline, two policies). Progress is emitted to stderr under the interactive policy (Decision H pattern from SP-003). On completion, the CLI exits 0 with a structured summary line: `classified=N, failed=M, skipped=K`.

**Why this priority**: P1 because the backlog scenario is real (any user landing on SP-004 from SP-003 has it) and because the manual-trigger path is the only way to re-attempt failed classifications until SP-006 ships `corpus drain --retry-failed`. The CLI command is also the canonical surface for users who prefer not to run the daemon continuously — they can run `corpus reenrich` after each ingest session. FR-019 (`corpus reclassify <doc-id>`) is explicitly out of scope (next-horizon SP-010); SP-004 ships only the batch-backlog command, not the per-doc-re-classify command.

**Independent Test**: Start with a corpus containing 20 sentinel rows produced by SP-003 (daemon ran with classifier disabled). Run `corpus reenrich`. Observe: stderr shows progress under interactivePolicy (one line per doc, e.g., `[1/20] doc-ab12cd34 ... classified in 4.2s (domain=ai-systems, type=tutorial)`); the command acquires `Paths.drainLock()` (a concurrent `corpus drain` invocation emits `pipeline.lock_contention` and exits 0); on completion, the exit code is 0 and the summary line names exact counts. Run `SELECT COUNT(*) FROM documents WHERE facet_type='unclassified'` — count is 0 (or = M for the failures-lane count, with M rows in `Paths.failed()` carrying `.error.json` sidecars). The same drain-lock contract used by SP-003 (FR-INGEST-011 / SC-INGEST-015) holds here.

**Acceptance Scenarios**:

1. **Given** a corpus with N ≥ 1 sentinel rows (rows where `facet_type='unclassified'`), **When** the user runs `corpus reenrich`, **Then** the command acquires `Paths.drainLock()` with `flock(LOCK_EX | LOCK_NB)` AND iterates the sentinel rows AND invokes the same classify-stage function the daemon uses (one pipeline, two policies — Constitution VI) AND exits 0 on completion with a summary line of the form `classified=N, failed=M, skipped=K` to stdout.
2. **Given** a concurrent `corpus drain` (SP-003 batch ingest) or `corpus daemon` (SP-003 watcher) is already holding the drain-lock, **When** `corpus reenrich` is invoked, **Then** the command observes lock contention AND emits a `pipeline.lock_contention` telemetry event AND exits 0 cleanly (matching SP-003's FR-INGEST-011 contract — the lock is the single serialization point across both ingest and reenrich).
3. **Given** a row whose SQL columns are already populated (`facet_type != 'unclassified'`), **When** `corpus reenrich` iterates the sentinel set, **Then** the populated row is skipped without invoking the classifier (the sentinel query `WHERE facet_type='unclassified'` filters at SQL level — no idempotency burden on the classifier itself; defense-in-depth at FR-CLASSIFY-012).
4. **Given** the user runs `corpus reenrich --dry-run`, **When** the command executes, **Then** it lists the sentinel rows it WOULD classify AND issues NO Ollama HTTP calls AND issues NO SQL UPDATEs AND exits 0.
5. **Given** a SIGTERM during `corpus reenrich`, **When** the signal is delivered, **Then** the in-flight classify-stage is aborted via `controller.abort()` AND the in-flight row stays sentinel (no partial UPDATE) AND the command exits within 2 seconds with a non-zero status AND an `ingest.aborted` telemetry event is emitted with `stage='classify'`.

### User Story 3 — The classifier proposes a never-before-seen domain or tag, and the proposal lands in `taxonomy_terms` at `state='proposed'` without polluting the established vocabulary (Priority: P1)

A document arrives whose subject matter doesn't fit any existing established domain. The classifier emits the closest-fitting established value into `facet_domain` AND emits the proposed new domain into the OPTIONAL `facet_domain_proposed` field of its structured output. Similarly for tags: closest-fitting established tags go into `tags`; never-seen tag candidates go into `facet_tags_proposed`. SP-004 reads the proposed-* fields and INSERTs them into the `taxonomy_terms` table with `axis='domain'` or `axis='tag'` and `state='proposed'` via `INSERT INTO taxonomy_terms (axis, term, state, established_at) VALUES (?, ?, 'proposed', NULL) ON CONFLICT(axis, term) DO NOTHING`. Established-state rows are NEVER inserted by SP-004 — promotion is a future user-review workflow (Principle XV gate, not auto-trigger). The classifier MUST NOT silently introduce a never-seen term into the established `facet_domain` SQL column; FR-014 explicitly forbids this. If the classifier output's `facet_domain` value isn't in the established vocabulary snapshot at prompt-construction time, defense-in-depth validation rejects the row and routes it to the failure lane with `error_code='vocabulary_violation'`.

**Why this priority**: P1 because Principle XV is NON-NEGOTIABLE and FR-014 explicitly forbids silent established-vocabulary expansion. Without the proposed-term routing, either (a) the classifier hallucinates established domains (SQL column has values that aren't in `taxonomy_terms WHERE state='established'`, breaking the SP-002 `corpus://taxonomy` contract), or (b) every document with a novel subject gets routed to the failure lane (effectively unusable for users with diverse corpora). The dual-field design — established values in `facet_domain` / `tags`, proposals in `facet_domain_proposed` / `facet_tags_proposed` — is the constitutional shape Principle XV mandates.

**Independent Test**: Seed `taxonomy_terms` with two established domains (e.g., `state='established'`: `agent-systems`, `distributed-systems`) and zero established tags. Drop a document about a wholly novel domain (e.g., `quantum-cryptography`). Run classify-stage. Observe: the row's `facet_domain` SQL column carries one of the two seeded established values (the closest fit, classifier's call); the row's `tags_json` carries 3-10 strings (all defense-in-depth-validated against the established tags table or against the row's own proposed-tags set). The `taxonomy_terms` table now contains a NEW row: `axis='domain', term='quantum-cryptography', state='proposed', established_at=NULL`. The `corpus://taxonomy` MCP resource (SP-002 read path) continues to return ONLY the two original established domains — the proposed row is invisible to that resource by Principle XV. Telemetry includes a `classify.term_proposed` event class with `axis`, `term`, `doc_id` fields.

**Acceptance Scenarios**:

1. **Given** an established vocabulary snapshot loaded at batch start (live `SELECT term FROM taxonomy_terms WHERE axis='domain' AND state='established'` and `WHERE axis='tag' AND state='established'`), **When** the classifier emits a `facet_domain` value present in the snapshot, **Then** the value is accepted AND written to the SQL `facet_domain` column AND mirrored to the body-file frontmatter.
2. **Given** the same snapshot, **When** the classifier emits a `facet_domain_proposed` value NOT present in the established domain snapshot, **Then** SP-004 INSERTs `(axis='domain', term=<proposed>, state='proposed', established_at=NULL)` into `taxonomy_terms` via `ON CONFLICT(axis, term) DO NOTHING` AND emits a `classify.term_proposed` telemetry event AND the value does NOT appear in the SQL `facet_domain` column for this document (the proposed-term field is metadata about the proposal, not the document's official classification).
3. **Given** a classifier response whose `facet_domain` value is NOT in the established snapshot AND whose `facet_domain_proposed` field is empty / missing, **When** defense-in-depth validation runs, **Then** the row routes to the failure lane (`Paths.failed() + '/<doc-id>.error.json'`) with `error_code='vocabulary_violation', retriable=true` AND a `classify.failed` telemetry event is emitted AND the SQL row stays sentinel AND no taxonomy_terms INSERT happens.
4. **Given** a classifier response whose `tags` array contains a tag NOT in the established tag snapshot AND NOT in the same response's `facet_tags_proposed` array, **When** defense-in-depth validation runs, **Then** the row routes to the failure lane with `error_code='vocabulary_violation'` (same contract as for domain).
5. **Given** the user-review workflow does NOT exist yet in SP-004, **When** proposed-state rows accumulate in `taxonomy_terms`, **Then** SP-004 introduces ZERO auto-promotion code AND ZERO frequency-threshold trigger AND ZERO CLI subcommand for promotion (Principle XV: N≥3 in 30 days is a GATE, not an auto-trigger; the user-review surface is a future-horizon sprint).
6. **Given** the SP-002 `corpus://taxonomy` resource is read after proposed terms accumulate, **When** the response is inspected, **Then** the response surfaces ONLY rows with `state='established'` AND zero proposed terms appear in the `established_domains` / `established_tags` arrays (SP-002's existing reader contract is preserved by SP-004's write semantics).

### Edge Cases

- **Ollama service unavailable at classify-stage start**: The HTTP POST to `http://localhost:11434/api/chat` fails with `ECONNREFUSED` or equivalent. The classifier MUST emit a `classify.ollama_unavailable` telemetry event (additional class beyond the FR-CLASSIFY-010 floor) AND route the document to the failure lane with `error_code='ollama_unavailable', retriable=true` (FR-012 acceptance scenario "Ollama service unavailable routes documents to failure lane with retriable flag"). The SP-003 row stays sentinel until the next reenrich attempt. The daemon does NOT crash; the next document in the batch is still attempted (one Ollama failure doesn't poison the queue). After three consecutive `ollama_unavailable` events in a single batch run, the batch is halted and a `classify.batch_halted` event emitted — this circuit-breaker prevents the daemon from spamming a dead Ollama process. The halt threshold (3) is a Plan-stage resolution.
- **Malformed Ollama response — JSON-parse fails despite `format` parameter**: Ollama's structured-output mode MAY return a non-JSON string in degenerate cases (model not loaded, GPU OOM, prompt budget overflow). Defense-in-depth Zod validation rejects this; the row routes to failure lane with `error_code='malformed_response', retriable=true` (1 retry per FR-CLASSIFY-005 retry policy, then failure lane).
- **Oversized body — body excerpt boundary safety**: The prompt template includes the first 2000 characters of the body. The truncation MUST occur at a UTF-8-safe codepoint boundary, NOT mid-multibyte-sequence (which produces malformed UTF-8 in the prompt and breaks Ollama's tokenizer). The implementation uses `Buffer.from(body, 'utf-8').slice(0, 2000).toString('utf-8', undefined, validUtf8Boundary)` or equivalent codepoint-safe truncation.
- **Classifier returns an established-but-unseen term in the SQL column**: The classifier's grammar-constrained output COULD theoretically emit any string in the `facet_domain` slot (Zod schema enforces "is a string", not "is in established vocab"). FR-014 mandates rejection of this. Defense-in-depth validation (FR-CLASSIFY-005, post-Ollama) cross-checks the response's `facet_domain` against the same established-vocab snapshot the prompt was rendered with. Mismatch → vocabulary violation, failure-lane routing (acceptance scenario 3.3).
- **Concurrent reenrich invocations**: The drain-lock at `Paths.drainLock()` is the single serialization point across SP-003 ingest, SP-003 daemon watcher, and SP-004 reenrich (Principle IX + FR-INGEST-011). A second `corpus reenrich` while the first holds the lock emits `pipeline.lock_contention` and exits 0 (FR-CLASSIFY-015).
- **Kill-9 (SIGKILL) mid-classify**: SP-004 ships single-stage atomicity — the SQL UPDATE + body-file frontmatter rewrite commit in one SQLite transaction or both fail. A SIGKILL between the body-file tmp-write and the rename, OR between the rename and the SQL COMMIT, leaves the row sentinel (rollback semantics) and a stale tmp file under `Paths.cache()` (swept by the SP-003-era janitor under one hour). The daemon's next post-persist hook OR the next `corpus reenrich` re-attempts the document. Full cross-stage SIGKILL-recovery (e.g., classify-in-progress checkpoint that resumes after restart) is SP-006 territory.
- **Per-document classifier wall-clock budget**: qwen3.5:9b on CPU is dramatically slower than on GPU; the per-document budget is 60 s default (interactive) / 300 s (batch — for re-attempts of degenerate inputs). The exact p95 is empirically measured at implementation time on the user's hardware (Constitution XVI honesty). If qwen3.5:9b proves too slow for batch comfort, the fallback model is `gemma3:4b` (Plan-stage Decision A); switching models is a single config change, not a code change.
- **Body without title or source**: SP-003 always writes `title`, `source_path`, `ingest_timestamp`, `mime_type`, `hash` (FR-008 minimum frontmatter). The prompt template references all three. If for any reason a row is missing one of those fields (e.g., manual SQL surgery), the classifier substitutes a deterministic placeholder (`<missing-title>`, `<missing-source>`) and emits a `classify.frontmatter_incomplete` warn-level telemetry event. The classifier does NOT fail-stop on missing fields.
- **Vocabulary snapshot stale during long batch**: A `corpus reenrich` batch of 500 docs MAY take minutes. If the user runs a separate session that adds proposed terms mid-batch (proposed-state INSERTs from concurrent classify-stage), those terms are NOT visible to the current batch's vocabulary snapshot (taken at batch start per Decision E in `research.md`). This is correct: established vocabulary changes only via user-review promotion (Principle XV — no auto-promotion); proposed terms accumulating mid-batch don't change the established set. A future user-review promotion happens at user-driven cadence, not at classify-stage cadence.
- **Document body in summary leak risk**: The classifier MAY produce a 15-25-word summary (per SCHEMA.md `summary` field). SP-004 writes the summary into the body-file frontmatter (which is local-disk-only — Principle I enforced by SP-001's egress hook intercepts any accidental network transmission). The summary is NEVER part of telemetry payloads (Principle I + SP-003 SC-INGEST-014 invariant generalized to SP-004 events).
- **Three-folder routing for classify failures**: Classify-stage failures route the FAILED CLASSIFICATION into `Paths.failed()` with a `<doc-id>.error.json` sidecar. The underlying SP-003 `processed/` body file and SQL row remain (the row stays sentinel; the body file unchanged). This differs slightly from SP-003's failure-lane semantics (where the failed file is moved INTO `failed/`); for classify failures, the failure-lane sidecar is a doc-id-keyed marker indicating "this row failed classification at attempt N." SP-006's `corpus drain --retry-failed` will replay these. The `pending/` invariant (Constitution X) still holds — classify-stage doesn't touch `pending/`.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-CLASSIFY-001 — Classify-stage triggers automatically post-persist via the SP-003 daemon's hook.** The SP-003 daemon (`packages/daemon/src/index.ts`) MUST invoke SP-004's classify-stage immediately after each successful SP-003 persist, passing the new row's `doc_id` and the AbortSignal from the daemon's master controller. The hook MUST be in-process (no subprocess, no IPC) — same address space, same drain-lock holder. Classify-stage is single-document per invocation in SP-004 (no batch parallelism within the hook — defer to SP-005 if benchmark requires); the daemon serializes calls. *(Sources: FR-012; Principle VI — one pipeline, two policies; Principle IX — single serialization point.)*

- **FR-CLASSIFY-002 — Classify-stage queries sentinel rows via `WHERE facet_type='unclassified'` and processes them one at a time.** The reenrich CLI command and the daemon's post-persist hook share the same query primitive: `SELECT id, title, body_path, source_path, mime_type FROM documents WHERE facet_type='unclassified' ORDER BY ingest_timestamp ASC` (FIFO; oldest sentinel first). The reenrich command iterates the result set; the daemon's hook passes a single `doc_id` and the query reduces to `WHERE id = ?`. *(Sources: FR-012; spec data-model.md sentinel rationale R6 from SP-003.)*

- **FR-CLASSIFY-003 — Ollama HTTP call uses the `format` parameter rendered from the canonical Zod-derived JSON Schema.** The classifier invokes `POST http://localhost:11434/api/chat` (Ollama 0.5+ structured-outputs endpoint) with request body containing (a) `model: 'qwen3.5:9b'` (primary, Decision A in research.md), (b) `messages: [{role: 'system', content: <rules>}, {role: 'user', content: <vocab + doc>}]`, (c) `format: <canonical Zod-derived JSON Schema>`, (d) `stream: false`, (e) `options: { temperature: 0.1 }` for determinism. The JSON Schema is generated at module-load time via `zodToJsonSchema(ClassifierOutputZodSchema)` (Decision J in research.md) and frozen as a module constant. The HTTP call uses `undici` (existing SP-001 transport dependency) with `AbortSignal` propagated end-to-end (Principle VII). *(Sources: FR-012; Principles I, V, VII.)*

- **FR-CLASSIFY-004 — Classifier output is structurally constrained at the token-generation level — invalid metadata is impossible by construction.** Ollama's `format` parameter binds the model's token sampler to the JSON Schema, so the response body parses as a JSON document whose top-level shape matches the schema. The classifier MUST NOT post-hoc string-parse free-form LLM output; the structured-output contract is the primary guarantee (Principle V). FR-CLASSIFY-005's defense-in-depth Zod validation is the safety net for degenerate cases (Ollama bugs, schema mismatches across Ollama versions), not the primary validation surface. The classifier acceptance scenario "classifier without grammar constraint configuration refuses to start" is enforced: the module asserts the `format` parameter is non-empty in the request constructor; missing schema throws `ClassifierConfigurationError` at boot. *(Sources: FR-012; Principle V; FR-012 acceptance scenarios.)*

- **FR-CLASSIFY-005 — Defense-in-depth Zod validation runs on every Ollama response before frontmatter writeback or SQL UPDATE.** After Ollama returns, the classifier parses the response body's `message.content` as JSON and validates it against the canonical `ClassifierOutputZodSchema` via `z.parse()`. Validation failure routes the document to the failure lane with `error_code='schema_invalid', retriable=true` AND emits a `classify.schema_invalid` telemetry event. Validation MUST NOT silently coerce missing required fields (FR-013 acceptance scenario "validator does not coerce missing required fields") — strict mode, no defaults. The validated output is the input to FR-CLASSIFY-006's vocabulary check and FR-CLASSIFY-008's persistence. *(Sources: FR-013; Principles V, XVI.)*

- **FR-CLASSIFY-006 — Established-vocabulary lookup happens at batch start and feeds the prompt + the defense-in-depth check.** At the start of any classify-stage invocation (daemon post-persist hook OR `corpus reenrich` batch), the classifier executes `SELECT term FROM taxonomy_terms WHERE axis IN ('domain','tag','type') AND state='established'` and snapshots the result into an in-memory `EstablishedVocabulary` record. The snapshot is valid for the lifetime of the invocation; subsequent classifies within the same batch reuse it (Decision E in research.md — per-batch refresh cadence). The vocabulary block in the prompt is rendered from this snapshot. The post-Ollama defense-in-depth check cross-references the response's `facet_domain` value against `snapshot.domains` and each `tags[i]` against `snapshot.tags`. Mismatch routes to failure lane unless the response separately lists the new term in `facet_domain_proposed` / `facet_tags_proposed` (FR-CLASSIFY-007). *(Sources: FR-014; Principle XV.)*

- **FR-CLASSIFY-007 — Proposed terms route to `taxonomy_terms` with `state='proposed'` via ON CONFLICT DO NOTHING.** When the classifier response carries a `facet_domain_proposed` string OR a `facet_tags_proposed` array of strings, SP-004 INSERTs each proposed term: `INSERT INTO taxonomy_terms (axis, term, state, established_at) VALUES (?, ?, 'proposed', NULL) ON CONFLICT(axis, term) DO NOTHING`. The INSERT is part of the same SQLite transaction as the documents-row UPDATE (FR-CLASSIFY-008) — atomic. A `classify.term_proposed` telemetry event is emitted per inserted-or-conflicted term. Established-state INSERTs are FORBIDDEN in SP-004; the only way to land an `state='established'` row in `taxonomy_terms` is via a future user-review promotion workflow (Principle XV gate). *(Sources: FR-014; Principle XV.)*

- **FR-CLASSIFY-008 — Paired SQL UPDATE + body-file frontmatter rewrite commit in a single SQLite transaction (atomicity contract).** On successful classify, the persister executes a single `BEGIN TRANSACTION ... COMMIT` block containing: (a) `UPDATE documents SET facet_domain=?, tags_json=?, facet_type=? WHERE id=? AND facet_type='unclassified'` (the `AND facet_type='unclassified'` clause provides defense-in-depth idempotency — if the row was concurrently classified, the UPDATE affects 0 rows and the transaction rolls back), (b) ZERO-or-more `INSERT INTO taxonomy_terms ... ON CONFLICT DO NOTHING` rows for proposed terms (FR-CLASSIFY-007), (c) the atomic rename of the tmp body-file (written via `withTempDir` outside the transaction) into the canonical path `Paths.docs() + '/' + row.body_path` (matching SP-003's persistence layout). On ANY failure: SQL transaction rollback + delete tmp body-file → row stays sentinel + no orphan tmp file. *(Sources: FR-013; Principles V, VIII.)*

- **FR-CLASSIFY-009 — AbortSignal propagation through HTTP + per-doc timeout via AbortController.** Every IO call in the classify-stage MUST accept `AbortSignal` and propagate it: the `undici` POST takes `signal` directly; SQLite prepared-statement steps check `signal.throwIfAborted()` between rows; the body-file rewrite via `withTempDir` accepts signal. SIGTERM → master controller `abort()` → in-flight classify aborts → row stays sentinel → in-flight tmp file cleaned up → `ingest.aborted` event emitted with `stage='classify'`. The per-document timeout is enforced via `AbortController` + `setTimeout(() => controller.abort('per_doc_timeout'), perDocTimeoutMs)` with `clearTimeout` on success (NEVER `Promise.race(setTimeout)`; Principle VII). Default `perDocTimeoutMs`: 60_000 (interactive) / 300_000 (batch) per Decision D. *(Sources: FR-012 acceptance scenarios; Principle VII.)*

- **FR-CLASSIFY-010 — Classify-stage emits ≥ 6 named telemetry event classes.** The SP-004 classify-stage MUST emit structured Zod-validated telemetry events for every state transition (Constitution XIII). Minimum SP-004 event classes: `classify.started` (doc_id, model_name, vocabulary_snapshot_id), `classify.ollama_request` (doc_id, model_name, prompt_token_estimate), `classify.ollama_response` (doc_id, response_token_count, duration_ms), `classify.schema_invalid` (doc_id, validation_errors), `classify.vocabulary_violation` (doc_id, offending_field, offending_value), `classify.term_proposed` (axis, term, doc_id), `classify.completed` (doc_id, facet_domain, facet_type, confidence_summary, retry_count, duration_ms), `classify.failed` (doc_id, error_code, message), `classify.ollama_unavailable` (errno, message), `classify.batch_halted` (consecutive_failures, threshold). These exceed the 6-class floor; some events may be emitted per-batch, others per-document. Every event validates against the existing `TelemetryEvent` Zod discriminated union (additive variants; PREREQ-003 pattern from SP-003 extends here). *(Sources: NFR-016; Principles XIII, V.)*

- **FR-CLASSIFY-011 — Failure-lane integration via `<doc-id>.error.json` sidecars (SP-003 failure-lane pattern).** Any classify-stage failure (Ollama unavailable, schema invalid, vocabulary violation, aborted, persist failed) MUST write a sidecar at `Paths.failed() + '/<doc-id>.error.json'` with fields `error_code` (closed enum extending the SP-003 sidecar enum with `ollama_unavailable`, `schema_invalid`, `vocabulary_violation`, `classify_aborted`), `message`, `retriable`, `stage='classify'`, `doc_id`, `timestamp`. The sidecar write is atomic per Principle VIII (`withTempDir` + rename). The underlying SP-003 `processed/` body file and SQL row remain (the row stays sentinel; the body file unchanged) — see Edge Case "Three-folder routing for classify failures". The sidecar is keyed by `<doc-id>.error.json` (NOT the original filename — for classify failures the doc-id is the stable forensic key, unlike SP-003 sidecars which preserve the original filename for replay). *(Sources: FR-013; Principles VIII, XVI; SP-003 FR-INGEST-007 pattern.)*

- **FR-CLASSIFY-012 — Idempotency — re-running classify-stage on an already-classified row is a no-op.** The SQL query `WHERE facet_type='unclassified'` filters sentinel rows at the storage layer; a classified row is never selected. Additionally, the per-row UPDATE statement carries `AND facet_type='unclassified'` (FR-CLASSIFY-008) — if concurrent classification races collide, the second UPDATE affects 0 rows and the second transaction rolls back. Re-running `corpus reenrich` on a fully-classified corpus is a 0-document batch (`classified=0, failed=0, skipped=0`). *(Sources: FR-012 acceptance scenarios; Principle X.)*

- **FR-CLASSIFY-013 — Confidence is excluded from persisted frontmatter (Principle II forbidden-fields contract).** The classifier output schema includes a `confidence` object (domain / type / tags sub-scores in [0, 1]) per FR-012. SP-004 MUST NOT write any `confidence` value to the body-file YAML frontmatter, nor to any SQL column, nor to any user-visible artifact on disk. Confidence is retained in-memory for retry-decision logic (FR-CLASSIFY-005's retry-once policy) and in the `classify.completed` telemetry event payload (`confidence_summary` field — telemetry is permitted by Principle II). Persisted frontmatter MUST also NOT contain `origin`, `provenance_*`, `captured_at`, or `corpus capture` fields. *(Sources: FR-012; Principle II.)*

- **FR-CLASSIFY-014 — No hardcoded `enum FacetDomain` — vocabulary is live from `taxonomy_terms`.** SP-004 source files MUST NOT contain `enum FacetDomain { ... }` or any equivalent fixed list of domain values. The classifier prompt template renders the vocabulary block by joining `EstablishedVocabulary.domains.join(', ')` at call time. The defense-in-depth check cross-references against the same snapshot. The `facet_type` enum IS hardcoded (the SCHEMA.md 7-value enum: `entity`, `concept`, `tutorial`, `analysis`, `reference`, `synthesis`, `cheat-sheet`) because facet_type is a structural taxonomy axis with constitutional stability — Principle XV's dynamic-vocabulary mandate applies to OPEN vocabularies (domain, tag), not to structural enums. The SCHEMA.md enum is the source of truth; SP-004 imports it from a single constant declared in `packages/contracts/src/classifier-schema.ts`. *(Sources: FR-014; Principle XV.)*

- **FR-CLASSIFY-015 — Drain-lock reuse — classify-stage serializes through `Paths.drainLock()` (single point of writer serialization).** Both the SP-003 daemon's post-persist hook and the `corpus reenrich` CLI command MUST acquire `Paths.drainLock()` with `flock(LOCK_EX | LOCK_NB)` before the first SQL UPDATE. Since the daemon already holds the lock during SP-003's drain loop, the post-persist hook reuses the held lock (no nested acquire). The `corpus reenrich` CLI acquires the lock independently; if SP-003's daemon is running, the CLI observes contention and emits `pipeline.lock_contention`, exiting 0 (FR-INGEST-011 contract preserved). The lock is the SINGLE write-serialization point across SP-003 ingest, SP-004 classify, and (future) SP-006 retry — no separate classify-lock. *(Sources: Principle IX; FR-INGEST-011.)*

- **FR-CLASSIFY-016 — Read-only-by-design MCP surface; no SP-004 mutation surface on the MCP server.** SP-004 introduces ZERO new MCP tools, resources, prompts, or mutation surfaces. The user-facing trigger surfaces are (a) automatic via the SP-003 daemon's post-persist hook and (b) manual via the `corpus reenrich` CLI command. The MCP server remains read-only per Constitution Principle III. `corpus://taxonomy` continues to surface ONLY `state='established'` rows — SP-004's proposed-state INSERTs are invisible to that resource by design. *(Sources: Principle III; SP-002 read-only resource contract.)*

- **FR-CLASSIFY-017 — Library/CLI boundary; no `process.exit` in `packages/inference/` or related SP-004 source.** All SP-004 source files under `packages/inference/`, `packages/pipeline/`, `packages/storage/` MUST NOT contain `process.exit`. Library functions return `Result<T, E>` or throw typed errors (`ClassifierError`, `OllamaUnavailableError`, `SchemaInvalidError`, `VocabularyViolationError`, `ClassifyPersistError`). Only the CLI wrapper `packages/cli/src/reenrich-command.ts` and the daemon entry point may exit the process. SP-001's existing `no-process-exit-in-libs` ESLint rule covers SP-004 source. *(Sources: Principle XI; SP-001 lint surface.)*

- **FR-CLASSIFY-018 — XDG-compliant paths only; no writes outside `Paths.*`.** Every file SP-004 reads or writes routes through `Paths.*`: `Paths.indexDb()` for the SQLite store; `Paths.docs()` for body files (mirroring SP-003's `body_path` resolution); `Paths.failed()` for `<doc-id>.error.json` sidecars; `Paths.telemetry()` for JSONL appends; `Paths.drainLock()` for the flock; `Paths.cache()` for `withTempDir` tmp. NO new XDG base; SP-004 uses existing getters only. *(Sources: Principle XIV.)*

- **FR-CLASSIFY-019 — Single-document, single-threaded classify-stage (no worker-pool parallelism in SP-004).** SP-004 classifies one document at a time within any single drain invocation. No worker pool, no Promise.all over multiple Ollama calls, no batching at the Ollama API level (Ollama can be configured for `OLLAMA_NUM_PARALLEL > 1` but SP-004 doesn't exploit it). The single-thread design (a) keeps the drain-lock window per-document instead of per-batch, (b) avoids GPU contention with the user's other tools (the user may be running Claude Code, Open WebUI, etc. against the same Ollama process), (c) bounds memory: only one prompt context resident at a time. SP-005 (embedding) is the natural place to revisit parallelism if benchmarks demand it. *(Sources: Principles VI, VII; Decision F in research.md — atomicity strategy depends on single-doc transactions.)*

- **FR-CLASSIFY-020 — Body excerpt truncation at UTF-8-safe codepoint boundary.** The prompt template's document-body block uses the first 2000 characters (NOT bytes — codepoints) of the normalized Markdown body. The truncation MUST occur at a valid UTF-8 codepoint boundary; mid-multibyte-sequence truncation is FORBIDDEN. Implementation: read the body file via `fs.readFile(path, 'utf-8')`, slice the resulting string at index 2000 (JavaScript string indexing on UTF-16 code units; the practical effect is safe for the BMP and benign for codepoint-pairs at the boundary). The 2000-codepoint cap is a Plan-time-resolved budget covering Decision H in research.md. *(Sources: Principle V — structured input handling; Decision H.)*

### Key Entities *(include if feature involves data)*

- **ClassifierOutput**: the Zod-typed shape returned by Ollama's structured-output endpoint. Required fields: `facet_domain: string`, `facet_type: enum (SCHEMA.md 7-value)`, `tags: string[3..10]`, `summary: string`, `confidence: { domain: number, type: number, tags: number }` (all in [0, 1]). Optional fields: `facet_domain_proposed: string | undefined`, `facet_tags_proposed: string[] | undefined`. Persisted to SQL (domain/type/tags) and to body-file frontmatter (domain/type/tags + summary if produced). NEVER persisted: `confidence`, `*_proposed` (proposed fields produce `taxonomy_terms` rows, not document fields).
- **EstablishedVocabulary**: an in-memory snapshot of `taxonomy_terms WHERE state='established'`, grouped by axis. Fields: `domains: Set<string>`, `tags: Set<string>`, `types: Set<string>` (types snapshot is informational — facet_type's enum is constitutional per FR-CLASSIFY-014). Loaded once per classify-stage invocation; stable for the invocation's lifetime; refreshed at the start of each subsequent invocation.
- **ProposedTerm**: a row in `taxonomy_terms` with `state='proposed'`, `axis IN ('domain', 'tag')`, `term: string` (the proposed value), `established_at: NULL`. Inserted via `ON CONFLICT(axis, term) DO NOTHING` so duplicate proposals collapse to a single row. Invisible to `corpus://taxonomy` MCP resource (that resource filters on `state='established'`).
- **ClassifyTelemetryEvent**: one of the FR-CLASSIFY-010 event classes. Validated against the existing `TelemetryEvent` Zod discriminated union (additive variants). Includes `event`, `timestamp`, `severity`, `outcome`, and class-specific fields (doc_id, model_name, duration_ms, confidence_summary, etc.). Confidence MAY appear in telemetry; body content MAY NOT (Principle I + SP-003 SC-INGEST-014 generalized).
- **ClassifyErrorSidecar**: `<doc-id>.error.json` at `Paths.failed()`, mirroring SP-003's error-sidecar shape. Stage is `'classify'`; error_code is one of the SP-004 extension enum values; retriable flag indicates whether `corpus drain --retry-failed` (SP-006) should re-attempt.

## Success Criteria *(mandatory)*

### Verification Strategy

For SP-004 verification, several criteria depend on the SP-001 + SP-002 + SP-003 baseline; several others depend on a running local Ollama process with at least one structured-output-capable model loaded. The honest partitioning:

- **End-to-end classification of real documents (US1, SC-CLASSIFY-001 through SC-CLASSIFY-005)**: Verified directly against the SP-003 baseline on the user's primary machine with Ollama + qwen3.5:9b loaded. Canonical SP-004 verification surface.
- **Sentinel-backlog reenrichment (US2, SC-CLASSIFY-006, SC-CLASSIFY-007)**: Verified by seeding a corpus with N sentinel rows (SP-003 with classifier-disabled config flag), then running `corpus reenrich`. Reproducible on the user's machine.
- **Proposed-term routing (US3, SC-CLASSIFY-008 through SC-CLASSIFY-010)**: Verified by seeding `taxonomy_terms` with a constrained established set, ingesting a document outside the established domain coverage, and asserting the proposed-state INSERT happened AND the established-state set is unchanged AND `corpus://taxonomy` still surfaces only the original established set.
- **Telemetry coverage (SC-CLASSIFY-011)**: Verified by a fixture-driven mixed-workload run covering happy path, schema-invalid response, vocabulary violation, Ollama unavailable, abort. The fixture harness mocks Ollama for the failure-injection cases; the happy-path case uses the real local Ollama.
- **Atomicity contract (SC-CLASSIFY-012, SC-CLASSIFY-013)**: Verified by integration tests that simulate mid-transaction failures and assert the sentinel-row + no-orphan-tmp-file invariant. Live-verifiable on user machine via SIGTERM during a long classification.
- **Drain-lock concurrency (SC-CLASSIFY-014)**: Verified by launching `corpus reenrich` and `corpus drain` concurrently and asserting the SP-003 FR-INGEST-011 contract holds.
- **Per-document classifier wall-clock budget**: Empirically measured at implementation time on the user's hardware (Constitution XVI honesty). Plan commits to "documented, measured, within the per-doc budget envelope" — specific p95 is reported in plan.md after the integration test runs.
